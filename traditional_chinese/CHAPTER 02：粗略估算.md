# 第02章：粗略估算



在系統設計面試中，有時會要求你使用粗略估算來估計系統的容量或性能需求。根據谷歌高級研究員傑夫·迪恩（Jeff Dean）的說法，「粗略估算是使用一系列思維實驗和常見性能數據的組合進行估算，以便對哪種設計能夠滿足你的要求有一個良好的了解」\[1]。

要有效地進行粗略估算，你需要對可擴展性基礎知識有很好的了解。以下概念應該被深入理解：二的冪\[2]、每個程式設計師都應該知道的延遲數字和可用性數字。

### 注意事項

ns = 納秒, μs = 微秒, ms = 毫秒

$$1 \space ns = 10^{-9} \space 秒$$

$$1 \space \mu s= 10^{-6} \space 秒 = 1,000 \space ns$$

$$1 \space ms = 10^{-3} \space 秒 = 1,000 \space \mu s = 1,000,000 \space ns$$

通過分析圖2-1中的數字，我們得出以下結論：

* 記憶體速度快，但磁碟速度慢。
* 如果可能的話，應避免磁碟搜尋。
* 簡單的壓縮算法速度快。
* 在發送資料到互聯網之前，盡可能對資料進行壓縮。
* 資料中心通常位於不同的區域，發送資料之間需要一定的時間。

### 可用性數據

高可用性是系統持續運行的能力，期望能夠長時間保持操作。 高可用性通常以百分比表示，100%意味著服務沒有任何停機時間，大多數服務的可用性介於99%到100%之間。

服務水平協議（SLA）是服務提供商常用的術語。這是你（服務提供商）與你的客戶之間的協議，該協議正式定義了你的服務將提供的運行時間水平。 雲服務提供商Amazon、Google和Microsoft將它們的SLA設置在99.9%或更高。系統的運行時間傳統上以數字的形式進行測量。 數字越多，表示系統的運行時間越長。 如表2-3所示，數字數量與預期系統停機時間相關。 ![img.png](../images/chapter2/table2-3.png)

### 示例：估算Twitter的查詢量和存儲需求

請注意，以下數字僅用於本練習，不是 Twitter 的真實數據。

假設：

* 每月活躍用戶為3億。
* 50%的用戶每天使用 Twitter。
* 用戶平均每天發布2條推文。
* 10%的推文包含媒體。
* 資料存儲時間為5年。

估算： 查詢每秒次數（QPS）估計：

* 每日活躍用戶（DAU）= 3億 \* 50% = 1.5億
* 推文 QPS = 1.5億 \* 2推文 / 24小時 / 3600秒 = 約 3500
* 峰值QPS = 2 \* QPS = 約 7000

(譯者注：這裡更準確地說應該估算的 TPS，而不是QPS。僅個人觀點，原文翻譯還是保持原作者意思為 QPS。)

我們這裡只會估算媒體存儲。

* 平均推文大小：
  * 推文ID 64 bytes（位元組）
  * 文字 140 bytes（位元組）
  * 媒體 1MB
* 媒體存儲：1.5億 \* 2 \* 10% \* 1MB = 每天30TB
* 5年媒體存儲：30TB \* 365 \* 5 = 約 55PB

### 小貼士

粗略估計更注重過程而非結果。 解決問題比得到準確結果更為重要。 面試官可能會測試你的解決問題的能力。 以下是一些建議：

* 四捨五入和近似值。在面試中進行複雜的數學運算是困難的。例如，「99987 / 9.1」的結果是多少？沒有必要花費寶貴的時間來解決複雜的數學問題。精確度並不是必需的。使用整數和近似值來簡化問題。例如，「100,000 / 10」。
* 記下你的假設。寫下你的假設是個好主意，以便日後參考。
* 標記你的單位。當你寫下「5」時，它是指5 KB還是5 MB？這可能會讓你感到困惑。寫下單位，因為「5 MB」有助於消除歧義。
* 常見的粗略估計問題包括：QPS、峰值QPS、存儲、快取、伺服器數量等。在準備面試時，你可以練習這些計算。孰能生巧。
