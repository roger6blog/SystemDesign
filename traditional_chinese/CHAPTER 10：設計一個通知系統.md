# 第10章：設計一個通知系統


近年來，通知系統已經成為許多應用程式非常流行的功能。 通知提醒使用者重要資訊，如突發新聞、產品更新、活動、優惠等。它已成為我們日常生活中不可或缺的一部分。 在本章中，您需要設計一個通知系統。

通知不僅是行動推送通知，三種類型的通知格式：行動推送通知、簡訊和電子郵件。 圖 10-1 顯示了每個通知的範例。

![](../images/chapter10/figure10-1.jpg)

### 第1步：了解問題並確定設計範圍

構建一個每天發送數百萬條通知的可擴展系統並非易事。 它需要對通知生態系統有深入的了解。 面試問題特意設計為開放式和模棱兩可的，你有責任提出問題以明確要求。

候選人：系統支援哪些類型的通知？ 面試官：推送通知、簡訊、郵件。

候選人：它是一個即時的系統嗎？

採訪者。讓我們說這是一個軟即時系統。我們希望使用者能盡快收到通知。但是，如果系統處於高負荷工作狀態，稍有延遲也是可以接受的。

候選人：支援的裝置有哪些？

面試官：iOS裝置、安卓裝置和筆記型電腦/桌上型電腦。

候選人：什麼觸發通知？

面試官：通知可以由用戶端應用程式觸發。 他們也可以由伺服端排程。

候選人：使用者是否能夠選擇退出？

面試官：是的，選擇退出的使用者將不再收到通知。

候選人：每天發出多少份通知？

面試官：1000萬條行動推送通知，100萬條簡訊，500萬封電子郵件。

### 第2步：提出高層次的設計方案並獲得認同

本節展示了支援各種通知類型的高層設計：iOS推送通知、Android推送通知、簡訊和電子郵件。它的結構如下：

* 不同的通知類型
* 聯絡人資訊收集流程
* 通知發送/接收流程

#### 不同的通知類型

我們先看一下每種通知類型在高層次上是如何工作的。

**iOS推送通知**

![](../images/chapter10/figure10-2.jpg)

我們主要需要三個元件來發送 iOS 推送通知：

1. Provider（提供商）：提供商構建通知請求並將其發送到 Apple 推送通知服務 (APNS)。 要構建推送通知，提供者提供以下資料：
   1. 裝置 token：這是用於發送推送通知的唯一識別碼。
   2.  有效載荷：是一個包含通知有效載荷的 JSON 字典。 範例：

       ```json
       {
           "aps": {
               "alert": {
                   "title": "Game Request",
                   "body": "Bob wants to play chess",
                   "action-loc-key": "PLAY"
               },
               "badge": 5
           }
       }
       ```
2. 蘋果推送通知服務（APNS）： 這是 Apple 提供的遠端服務，用於將推送通知傳播到 iOS 裝置。
3. iOS 裝置：它是接收推送通知的終端用戶端。

**Android推送通知**

Android 採用了類似的通知流程。 Firebase Cloud Messaging (FCM) 通常用於向 Android 裝置發送推送通知，而不是使用 APN。

![](../images/chapter10/figure10-3.jpg)

**簡訊**

對於SMS訊息，通常使用第三方SMS服務，如Twilio\[1]、Nexmo\[2]和其他許多服務。它們中的大多數是商業服務。

![](../images/chapter10/figure10-4.jpg)

**郵件**

雖然公司可以設定自己的電子郵件伺服器，但其中許多公司選擇商業電子郵件服務。 Sendgrid \[3] 和 Mailchimp \[4] 是最受歡迎的電子郵件服務之一，它們提供更好的交付率和資料分析。

#### 通知發送/接收流程

我們將首先介紹初始設計；然後，提出了一些優化方案。

**高層設計**

圖 10-9 顯示了設計，下面解釋了每個系統元件。

![](../images/chapter10/figure10-9.jpg)

**服務1到N**：一個服務可以是一個微服務、一個cron job，或者一個觸發通知發送事件的分散式系統。例如，一個計費服務發送電子郵件提醒客戶到期付款，或者一個購物網站通過簡訊告訴客戶他們的包裹明天會被送到。

**通知系統**：通知系統是發送/接收通知的中心環節。從簡單的東西開始，只使用一個通知伺服器。它為服務1到N提供API，並為第三方服務建立通知有效載荷。

**第三方服務**：第三方服務負責向使用者發送通知。在與第三方服務整合時，我們需要特別注意可擴展性。良好的可擴展性意味著一個靈活的系統可以很容易地插入或拔出第三方服務。另一個重要的考慮因素是，第三方服務可能在新的市場或在未來無法使用。例如，FCM在中國是不可用的。因此，在那裡使用替代的第三方服務，如Jpush、PushY等。

**iOS, Android, SMS, Email**：使用者在其裝置上收到通知。

在這個設計中，發現了三個問題：

1. 單點故障（SPOF）：單一通知伺服器意味著SPOF。
2. 難以擴展：通知系統在一台伺服器上處理所有與推送通知有關的事情。要獨立擴展資料庫、快取和不同的通知處理元件是很有挑戰性的。
3. 效能瓶頸：處理和發送通知可能是資源密集型的。例如，構建HTML頁面和等待第三方服務的回應可能需要時間。在一個系統中處理所有事情可能會導致系統過載，尤其是在高峰期。

**高層設計（改進後的）**

在列舉了初始設計中的挑戰後，我們對設計進行了如下改進：

* 將資料庫和快取從通知伺服器中移出
* 新增更多的通知伺服器，並設定自動水平縮放功能
* 引入訊息佇列，使系統元件解耦

圖10-10顯示了改進後的高層設計。

![](../images/chapter10/figure10-10.jpg)

瀏覽上圖的最佳方式是從左到右。

* **服務1到N**：它們代表不同的服務，通過通知伺服器提供的API發送通知。
* **通知伺服器**: 它們提供以下功能：
  * 為服務提供發送通知的API。這些API只能由內部或經過驗證的客戶存取，以防止垃圾郵件。
  * 進行基本驗證，以核實電子郵件、電話號碼等。
  * 查詢資料庫或快取以取得渲染通知所需的資料。
  *   將通知資料放到訊息佇列中進行並行處理。

      下面是一個發送電子郵件的API的例子：

      POST https://api.example.com/v/sms/send

      Request body：

      ![](../images/chapter10/figure10-hello.jpg)
* **快取**：使用者資訊、裝置資訊、通知模板都被快取了。
* **資料庫**：它儲存了關於使用者、通知、設定等方面的資料。
* **訊息佇列**：它們消除了元件之間的依賴性。當大量的通知被發送出去時，訊息佇列可以作為緩衝區。每種通知類型都被分配了一個不同的訊息佇列，所以一個第三方服務的中斷不會影響其他通知類型。
* **Workers**：Workers 是一組伺服器，它們從訊息佇列中拉取通知事件並將它們發送到相應的第三方服務。
* **第三方服務**：在最初的設計中已經解釋過。
* **iOS, Android, SMS, Email**：在最初的設計中已經說明。

接下來，讓我們來看看每個元件是如何一起工作來發送通知的。

1. 一個服務呼叫通知伺服器提供的API來發送通知
2. 通知伺服器從快取或資料庫中取得中繼資料，如使用者資訊、裝置令牌和通知設定。
3. 一個通知事件被發送到相應的佇列進行處理。例如，一個iOS推送通知事件被發送到iOS PN佇列中。
4. 工作者從訊息佇列中取得通知事件。
5. 工作者向第三方服務發送通知。
6. 第三方服務向使用者裝置發送通知。

#### 可靠性

在分散式環境中設計通知系統時，我們必須回答幾個重要的可靠性問題。

**如何防止資料遺失？**

通知系統最重要的要求之一是它不能遺失資料。 通知通常可以延遲或重新排序，但絕不會遺失。 為了滿足這個需求，通知系統**將通知資料持久化到資料庫**中，並實現了重試機制。 包含通知日誌資料庫以實現資料持久化，如圖 10-11 所示。

![](../images/chapter10/figure10-11.jpg)

**接收者只會收到一次通知嗎？**

最簡潔的答案是不。 儘管大多數時候通知只發送一次，但分散式特性可能會導致重複通知。 為了減少重複的發生，我們引入了重複資料刪除機制並仔細處理每個失敗案例。 這是一個簡單的重複資料刪除邏輯：

當一個通知事件第一次到達時，我們通過檢查事件的ID來檢查它是否在之前出現過。如果在之前出現過，它將被丟棄。否則，我們將發送通知。有興趣的讀者可以研究一下為什麼我們不能僅一次發送，請參考參考資料\[5]。

#### 其他元件和考慮因素

我們已經討論了如何收集使用者的聯絡資訊，發送和接收通知。一個通知系統遠不止這些。在這裡，我們討論了額外的元件，包括模板重複使用、通知設定、事件追蹤、系統監控、速率限制等。

*   通知模板

    一個大型的通知系統每天會發出數百萬條通知，其中許多通知都遵循類似的格式。通知模板的引入是為了避免從頭開始建立每一個通知。通知模板是一個預先格式化的通知，通過自訂參數、樣式、追蹤連結等來建立你獨特的通知。下面是一個推送通知的例子模板。

    ```
    BODY:
    You dreamed of it. We dared it. [ITEM NAME] is back — only until [DATE].
    CTA:
    Order Now. Or, Save My [ITEM NAME]
    The benefits of using notification templates include maintaining a consistent format, reducing
    the margin error, and saving time.
    ```
*   通知設定

    使用者一般每天都會收到太多的通知，他們很容易感到不堪重負。因此，許多網站和應用程式讓使用者對通知設定進行細化控制。這些資訊儲存在通知設定表中，有以下欄位：

    ```
    user_id   bigInt
    channel  varchar    # push notification, email or SMS
    opt_in   boolean    # opt-in to receive notification
    ```

    在向使用者發送任何通知之前，我們首先檢查使用者是否選擇接收這種類型的通知。
*   速率限制

    為了避免使用者被過多的通知所淹沒，我們可以限制一個使用者可以收到的通知數量。這一點很重要，因為如果我們發送得太頻繁，接收者可能會完全關閉通知。
*   重試機制

    當第三方服務無法發送通知時，該通知將被新增到訊息佇列中進行重試。如果問題持續存在，將向開發者發出警報。
*   推送通知的安全問題

    對於iOS或Android應用程式，appKey和appSecret被用來保護推送通知API\[6]。只有經過認證或驗證的用戶端才允許使用我們的API發送推送通知。有興趣的使用者應該參考參考資料\[6]。
*   監視排隊通知

    要監控的一個關鍵指標是排隊通知的總數。如果這個數字很大，說明 workers 處理通知事件的速度不夠快。為了避免通知交付的延遲，需要更多的工作者。圖10-12（歸功於\[7])顯示了一個待處理的排隊訊息的例子。

    ![](../images/chapter10/figure10-12.jpg)

    打開率、點擊率和參與度等通知指標對於了解客戶行為非常重要。 分析服務實現事件追蹤。 通常需要通知系統和分析服務之間的整合。 圖 10-13 顯示了可能出於分析目的而被追蹤的事件範例。

### 第4步：總結

通知是不可缺少的，因為它們讓我們及時了解重要資訊。它可能是關於你在Netflix上最喜歡的電影的推送通知，關於新產品折扣的電子郵件，或關於你線上購物付款確認的訊息。

在這一章中，我們描述了一個可擴展的通知系統的設計，它支援多種通知格式。推送通知、SMS訊息和電子郵件。我們採用了訊息佇列來解耦系統元件。

除了高層次的設計，我們還深入挖掘了更多的元件和優化。

* 可靠性：我們提出了一個強大的重試機制，以盡量減少失敗率。
* 安全性：AppKey/appSecret對用於確保只有經過驗證的客戶才能發送通知。
* 追蹤和監測：這些都是在通知流程的任何階段實施的，以捕捉重要的統計資料。
* 尊重使用者設定：使用者可以選擇不接收通知。 我們的系統在發送通知之前首先檢查使用者設定。
* 速率限制：使用者會喜歡對他們收到的通知數量設定頻率上限。
