# 第08章：短網址設計


在這一章中，我們將解決一個有趣而經典的系統設計面試問題：設計一個像tinyurl一樣的URL縮短服務。

### 第1步：了解問題並確定設計範圍

系統設計的面試問題是故意留有餘地的。為了設計一個精心設計的系統，關鍵是要問清楚問題。

候選人：你能舉個例子說明URL縮短器的工作原理嗎？

面試官：假設 URL `https://www.systeminterview.com/q=chatsystem&c=loggedin&v=v3&l=long` 是原始 URL。你的服務建立了一個長度較短的別名：`https://tinyurl.com/y7keocwj`。如果你按一下較短的別名URL，它會將你重新導向到原始 URL。

候選人：流量是多少？

面試官：每天產生1億個URL。

候選人：縮短後的URL有多長？

面試官：越短越好。

候選人：縮短的網址中允許使用哪些字元？

面試官：短網址可以是數字（0-9）和字元（a-z，A-Z）的組合。

候選人：縮短的URL可以刪除或更新嗎？

面試官：為了簡單起見，我們假設縮短的URL不能被刪除或更新。

以下是基本用例：

1. URL縮短：給定一個長的URL => 返回一個短得多的URL
2. URL重新導向：給定一個短的URL => 重新導向到原來的URL
3. 高可用性、可擴展性和容錯考慮

#### 粗略估計系統量級

* 寫操作：每天產生1億個URL。
* 每秒寫操作：億/24/3600 = 1160
* 讀操作：假設讀操作和寫操作的比例為10:1，讀每秒操作：1160 \* 10 = 11,600
* 假設 URL 縮短服務將運行 10 年，這意味著我們必須支援 1 億 \* 365 \* 10 = 3650 億條記錄。
* 假設平均 URL 長度為 100。
* 10 年的儲存需求：3650 億 \* 100 位元組 \* 10 年 = 365 TB

重要的是，你要與面試官一起探討假設和計算，以便你們兩個人都在同一個起跑線上。

### 第2步：提出高層次的設計方案並獲得認同

在本節中，我們將討論 API 端點、URL 重新導向和 URL 縮短。

#### API 端點

API端點促進了用戶端和伺服器之間的通訊。我們將設計REST風格的API。如果你不熟悉restful API，你可以查閱外部資料，比如參考資料中的資料\[1]。一個URL縮短器主要需要兩個API端點。

1.  網址縮短。 要建立一個新的短 URL，用戶端發送一個 POST 請求，其中包含一個參數：原始長 URL。 API 如下所示：

    **POST api/v1/data/shorten**

    * 請求參數：{longUrl: longURLString}。
    * 返回 shortURL
2.  URL重新導向。為了將一個短的URL重新導向到相應的長的URL，一個用戶端發送一個GET請求。該API看起來像這樣：

    **GET api/v1/shortUrl**

    * 返回用於HTTP重新導向的 longURL

#### URL 重新導向

圖8-1顯示了當你在瀏覽器上輸入一個tinyurl時會發生什麼。一旦伺服器收到tinyurl請求，它就會用301重新導向將短網址改為長網址。

![](../images/chapter8/figure8-1.jpg)

用戶端和伺服器之間的詳細通訊情況如圖8-2所示。

![](../images/chapter8/figure8-2.jpg)

值得在這裡討論的一件事是 301 重新導向與 302 重新導向。

* **301重新導向**。301重新導向表明，請求的URL被「永久」地移到了長URL上。由於是永久重新導向，瀏覽器會快取回應，對同一URL的後續請求將不會被發送到URL縮短服務上。相反，請求將直接被重新導向到長網址伺服器。
* **302重新導向**。302重新導向意味著URL被「暫時」移到長URL上，這意味著對同一URL的後續請求將首先被發送到URL縮短服務上。然後，它們會被重新導向到長網址伺服器。

每種重新導向方法都有其優點和缺點。如果優先考慮**減少伺服器負載**，使用301重新導向是有意義的，因為只有同一URL的第一個請求被發送到URL縮短伺服器。然而，如果分析是重要的，302重新導向是一個更好的選擇，因為它可以更容易地**追蹤點擊率和點擊的來源。**

實現URL重新導向的最直觀的方法是**使用雜湊表**。假設雜湊表儲存\<shortURL, longURL>對，URL重新導向可以通過以下方式實現。

* 取得longURL: `longURL = hashTable.get(shortURL)`
* 一旦你得到longURL，就執行URL重新導向。

#### 縮短網址

讓我們假設短的URL看起來像這樣：`www.tinyurl.com/{hashValue}`。為了支援縮短URL的用例，我們必須找到一個雜湊函數fx，將長URL映射到hashValue，如圖8-3所示。

![](../images/chapter8/figure8-3.jpg)

雜湊函數必須滿足以下要求。

* 每個longURL必須被雜湊成一個hashValue。
* 每個hashValue都可以被映射回longURL。

雜湊函數的詳細設計將深入討論。

### 第3步：深入設計

到目前為止，我們已經討論了URL縮短和URL重新導向的高層設計。在本節中，我們將深入探討以下內容：資料模型、雜湊函數、URL縮短和URL重新導向。

#### 資料模型

在高層設計中，所有的東西都儲存在一個雜湊表中。這是一個很好的起點；然而，這種方法在現實世界的系統中是不可行的，因為記憶體資源是有限的和昂貴的。一個更好的選擇是將\<shortURL, longURL>映射儲存在一個關聯式資料庫中。圖8-4顯示了一個簡單的資料庫表設計。簡化版的表包含3欄：id、shortURL、longURL。

![](../images/chapter8/figure8-4.jpg)

#### 雜湊函數

雜湊函數用於將一個長的URL雜湊成一個短的URL，也稱為 hashValue。

**雜湊值的長度**

hashValue 由\[0-9, a-z, A-Z]中的字元組成，包含 $$10+26+26=62$$ 個可能的字元。要計算 hashValue 的長度，找出最小的 n，使 $$62^n \ge 365億$$。根據估計，系統必須支援多達 3650 億個 URL。 表 8-1 顯示了 hashValue 的長度和它可以支援的相應最大 URL 數。

![](../images/chapter8/table8-1.jpg)

當 $$n = 7$$ 時， $$62 ^ n \approx 3.5兆$$，3.5 兆足以容納 3650億個URL，所以 hashValue 的長度為 7。

我們將探討兩種類型的URL縮短器的雜湊函數。第一種是「雜湊+碰撞解決」，第二種是「base 62 轉換」。讓我們逐一來看看它們。

**雜湊+碰撞解決**

為了縮短長的URL，我們應該實現一個雜湊函數，將長的URL雜湊成一個7個字元的字串。一個直接的解決方案是使用知名的雜湊函數，如CRC32、MD5或SHA-1。下表比較了在這個URL（[https://en.wikipedia.org/wiki/Systems\_design](https://en.wikipedia.org/wiki/Systems\_design)）上應用不同雜湊函數後的雜湊結果：

![](../images/chapter8/table8-2.jpg)

如表8-2所示，即使是最短的雜湊值（来自CRC32）也太長了（超過7個字元）。我們怎樣才能使它更短呢？

第一種方法是收集雜湊值的前7個字元；然而，這種方法會導致雜湊碰撞。為了解決雜湊碰撞，我們可以遞迴地附加一個新的預定義字串，直到不再發現碰撞。這一過程在圖8-5中得到了解釋。

![](../images/chapter8/figure8-5.jpg)

這種方法可以消除碰撞；但是，查詢資料庫以檢查每個請求是否存在短網址的成本很高。一種叫做**Bloom過濾器**的技術\[2]可以提高效能。布隆過濾器是一種空間效率高的概率技術，用來測試一個元素是否是一個集合的成員。更多細節請參考參考資料\[2]。

**base 62 轉換**

Base 轉換是 URL 縮短器常用的另一種方法。 Base 轉換有助於在不同的數字表示系統之間轉換成相同的數字。 使用 Base 62 轉換，因為 hashValue 有 62 個可能的字元。 讓我們用一個例子來解釋轉換的工作原理：將 $$11157_{10}$$ 轉換為 base 62 表示（ $$11157_{10}$$ 在 base 10 系統中表示 11157）。


* 從名字上看，base 62 是一種使用 62 個字元進行編碼的方式。 映射是： 0-0, ..., 9-9, 10-a, 11-b, ..., 35-z, 36-A, ..., 61-Z，其中「a」代表 10，「Z」 代表 61，等等。
* $$11157_{10} = 2 \times 62^2 + 55 \times 62^1 + 59 \times 62^0 = \left [2, 55, 59 \right ] \rightarrow \left [2, T, X \right]$$ 以 base 62 表示。

    對話過程如圖8-6所示。

    ![](../images/chapter8/figure8-6.jpg)
* 因此，短網址是：https://tinyurl.com/2TX

**兩種方法的比較**

下表顯示了兩種方法的差異。

| 雜湊+碰撞解決                    | base 62 轉換                                    |
| -------------------------- | --------------------------------------------- |
| 固定短URL長度                   | 短URL長度不固定，它隨著 id 變化                           |
| 不需要唯一ID生成器                 | 該選項依賴於唯一ID生成器                                 |
| 可能出現碰撞，必須解決                | 碰撞是不可能的，因為 ID 是唯一的                            |
| 不可能計算出下一個可用的短網址，因為它不依賴於ID。 | 如果新條目的 ID 遞增 1，則很容易找出下一個可用的短 URL。 這可能是一個安全問題。 |

#### URL 縮短的深入研究

作為系統的核心部分之一，我們希望URL縮短的流程在邏輯上是簡單和實用的。在我們的設計中使用了62進制轉換。我們建立了以下圖表（圖8-7）來演示這個流程。

![](../images/chapter8/figure8-7.jpg)

1. longURL 是輸入的
2. 系統檢查 longURL 是否存在於資料庫中
3. 如果是的話，這意味著longURL之前被轉換為shortURL。在這種情況下，從資料庫中取得shortURL並將其返回給用戶端。
4. 如果不是，longURL是新的。一個新的唯一ID（主鍵）由唯一ID生成器生成。
5. 使用base 62轉換將ID轉換為shortURL。
6. 用ID、shortURL和longURL建立一個新的資料庫記錄。

為了使流程更容易理解，讓我們看一個具體的例子。

* 假設輸入的longURL是：https://en.wikipedia.org/wiki/Systems\_design
* 唯一ID生成器返回ID：2009215674938
* 使用62進制轉換將ID轉換為shortURL。ID（2009215674938）被轉換為「zn9edcu」。
*   將ID、shortURL和longURL儲存到資料庫，如表8-4所示。

    ![](../images/chapter8/table8-4.jpg)

值得一提的是分散式唯一 ID 生成器。 它的主要功能是生成全域唯一 ID，用於建立 shortURL。 在高度分散式的環境中，實現唯一 ID 生成器具有挑戰性。 幸運的是，我們已經在「第 7 章：在分散式系統中設計唯一 ID 生成器」中討論了一些解決方案。 你可以回過頭來回顧它來刷新你的記憶。

#### URL重新導向的深入研究

圖 8-8 顯示了 URL 重新導向的詳細設計。 由於讀取多於寫入，`<shortURL, longURL>` 映射儲存在快取中以提高效能。

![](../images/chapter8/figure8-8.jpg)

URL重新導向的流程總結如下：

* 一個使用者點擊了一個短的URL連結：`https://tinyurl.com/zn9edcu`
* 負載平衡器將請求轉發給網路伺服器
* 如果shortURL已經在快取中，直接返回longURL。
* 如果短URL不在快取中，從資料庫中取得長URL。如果它不在資料庫中，很可能是使用者輸入了一個無效的短網址。
* longURL被返回給使用者。

### 第4步：總結

在本章中，我們討論了 API 設計、資料模型、雜湊函數、URL 縮短和 URL 重新導向。

如果在面試結束時有多餘的時間，這裡有幾個額外的談話要點：

* 速率限制器：我們可能面臨的一個潛在安全問題是惡意使用者發送大量的URL縮短請求。速率限制器有助於根據IP位址或其他過濾規則過濾掉請求。如果你想重温有關速率限制的記憶，請參閱「第 4 章：設計速率限制器」。
* Web 伺服器擴展：由於 Web 層是無狀態的，因此很容易通過新增或刪除 Web 伺服器來擴展 Web 層。
* 資料庫的擴展。資料庫複製和分片是常見的技術。
* 分析。資料對商業成功越來越重要。將分析解決方案整合到URL縮短器中，可以幫助回答一些重要的問題，如有多少人點擊一個連結？他們何時點擊連結？等等。
* 可用性、一致性和可靠性。這些概念是任何大型系統成功的核心。我們在第1章中對它們進行了詳細討論，請對這些主題進行複習記憶
