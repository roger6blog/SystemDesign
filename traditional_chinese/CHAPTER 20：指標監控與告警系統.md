## 第 1 步 - 了解問題並確定設計範圍

指標監控與告警系統對不同的公司可能意味著許多不同的事情，因此，首先與面試官明確需求至關重要。例如，如果面試官心裡只想的是基礎設施指標，你肯定不希望設計一個側重於日誌（如 Web 伺服器錯誤或存取日誌）的系統。

在深入細節之前，讓我們首先充分了解問題並建立設計範圍。

**候選人**：我們要為誰建構這個系統？是為像 Facebook 或 Google 這樣的大型公司建構內部系統，還是設計一個像 Datadog [1]、Splunk [2] 等那樣的 SaaS 服務？
**面試官**：這是一個很好的問題。我們僅建構供內部使用。

**候選人**：我們想要收集哪些指標？
**面試官**：我們想要收集運維系統指標。這些可以是作業系統的底層使用資料，例如 CPU 負載、記憶體使用情況和磁碟空間消耗。它們也可以是更高層次的指標，例如服務的每秒請求數或 Web 伺服器池的運行伺服器數量。業務指標不在本次設計的範圍內。

**候選人**：我們用這個系統監控的基礎設施規模有多大？
**面試官**：1 億日活躍使用者，1,000 個伺服器池，每個池 100 台機器。

**候選人**：我們需要保留資料多久？
**面試官**：假設我們想要 1 年的保留期。

**候選人**：我們可以為了長期儲存而降低指標資料的解析度（resolution）嗎？
**面試官**：這是一個很好的問題。我們希望能夠將新接收的資料保留 7 天。7 天後，你可以將其彙總（roll up）為 1 分鐘的解析度並保留 30 天。30 天後，你可以進一步將其彙總為 1 小時的解析度。

**候選人**：支援哪些告警管道？
**面試官**：電子郵件、電話、PagerDuty [3] 或 webhooks（HTTP 端點）。

**候選人**：我們需要收集日誌嗎，比如錯誤日誌或存取日誌？
**面試官**：不需要。

**候選人**：我們需要支援分散式系統追蹤（distributed system tracing）嗎？
**面試官**：不需要。

### 高層需求與假設

現在你已經完成了從面試官那裡收集需求，並有了明確的設計範圍。需求如下：

- 被監控的基礎設施是超大規模的。
  - 1 億日活躍使用者
  - 假設我們有 1,000 個伺服器池，每個池 100 台機器，每台機器 100 個指標 => ~1000 萬個指標
  - 1 年資料保留期
  - 資料保留策略：原始資料保留 7 天，1 分鐘解析度保留 30 天，1 小時解析度保留 1 年
- 可以監控各種指標，包括但不限於：
  - CPU 使用率
  - 請求計數
  - 記憶體使用情況
  - 訊息佇列中的訊息計數

### 非功能性需求

- 可擴展性。系統應該能夠擴展以適應不斷增長的指標和告警量。
- 低延遲。系統需要為儀表板和告警提供低查詢延遲。
- 可靠性。系統應該是高度可靠的，以避免錯過關鍵告警。
- 靈活性。技術在不斷變化，因此流水線（pipeline）應該足夠靈活，以便在未來輕鬆整合新技術。

哪些需求超出了範圍？

- 日誌監控。Elasticsearch, Logstash, Kibana (ELK) 堆疊在收集和監控日誌 [4] 方面非常流行。
- 分散式系統追蹤 [5] [6]。分散式追蹤是指一種追蹤服務請求流經分散式系統時的追蹤解決方案。它在請求從一個服務轉向另一個服務時收集資料。

## 第 2 步 - 提出高層設計並獲得認可

在本節中，我們將討論建構系統的一些基礎知識、資料模型和高層設計。

### 基礎知識

一個指標監控與告警系統通常包含五個元件，如圖 2 所示。

1. 資料收集 (Data collection)
2. 資料傳輸 (Data transmission)
3. 資料儲存 (Data storage)
4. 告警 (Alerting)
5. 可視化 (Visualization)

![Figure5.2.png](../images/v2/chapter05/Figure5.2.png)

圖 2 系統的五個元件

1. 資料收集：從不同來源收集指標資料。
2. 資料傳輸：將資料從來源傳輸到指標監控系統。
3. 資料儲存：整理並儲存接收到的資料。
4. 告警：分析接收到的資料，檢測異常並生成告警。系統必須能夠將告警發送到不同的通訊管道。
5. 可視化：以圖形、圖表等形式展示資料。當資料以可視化方式呈現時，工程師能更好地識別模式、趨勢或問題，因此我們需要可視化功能。

### 資料模型

指標資料通常被記錄為時間序列（time series），它包含一組值及其關聯的時間戳。序列本身可以由其名稱唯一標識，並且可以選擇由一組標籤（labels）來標識。

讓我們看兩個例子。

例子 1：20:00 時生產伺服器執行個體 i631 的 CPU 負載是多少？

![Figure5.3.png](../images/v2/chapter05/Figure5.3.png)

圖 3 CPU 負載

圖 3 中突出顯示的資料點可以用表 1 表示。

| metric_name | cpu.load |
| :--- | :--- |
| labels | host:i631,env:prod |
| timestamp | 1613707265 |
| value | 0.29 |

表 1 用表格表示的資料點

在這個例子中，時間序列由指標名稱、標籤（host:i631,env:prod）以及特定時間點的單個值表示。

例子 2：過去 10 分鐘內 us-west 地區所有 Web 伺服器的平均 CPU 負載是多少？從概念上講，我們會從儲存中提取如下內容，其中指標名稱是「CPU.load」，地區標籤是「us-west」：

CPU.load host=webserver01,region=us-west 1613707265 50
CPU.load host=webserver01,region=us-west 1613707265 62
CPU.load host=webserver02,region=us-west 1613707265 43
CPU.load host=webserver02,region=us-west 1613707265 53
...
CPU.load host=webserver01,region=us-west 1613707265 76
CPU.load host=webserver01,region=us-west 1613707265 83

平均 CPU 負載可以通過對每行末尾的值取平均來計算。上述例子中的行格式被稱為行協議（line protocol）。它是市場上許多監控軟體常用的輸入格式。Prometheus [7] 和 OpenTSDB [8] 就是兩個典型的例子。

每個時間序列由以下內容組成 [9]：

| 名稱 | 類型 |
| :--- | :--- |
| 指標名稱 (A metric name) | String |
| 一組標籤 (A set of tags/labels) | <key:value> 鍵值對列表 |
| 一組值及其時間戳 (An array of values and their timestamps) | <value, timestamp> 對陣列 |

表 2 時間序列

### 資料存取模式

![Figure5.4.png](../images/v2/chapter05/Figure5.4.png)

圖 4 資料存取模式

在圖 4 中，y 軸上的每個標籤代表一個時間序列（由名稱和標籤唯一標識），而 x 軸代表時間。

寫入負載很重。如你所見，在任何時刻都可能有許多時間序列資料點被寫入。正如我們在「高層需求」部分提到的，每天大約有 1000 萬個運維指標被寫入，且許多指標是以高頻率收集的，因此流量無疑是寫入密集型（write-heavy）。

與此同時，讀取負載是突發性的。可視化和告警服務都會向資料庫發送查詢，根據圖表和告警的存取模式，讀取量可能會有峰值。

換句話說，系統處於持續的重度寫入負載之下，而讀取負載則是突發性的。

### 資料儲存系統

資料儲存系統是設計的核心。不建議為此工作建構自己的儲存系統或使用通用儲存系統（如 MySQL）。

理論上，通用資料庫可以支援時間序列資料，但需要專家級的調優才能在我們這種規模下運行。具體來說，關聯式資料庫並未針對時間序列資料通常進行的各種操作進行優化。例如，在滾動時間視窗內計算移動平均值（moving average）需要複雜的 SQL，且難以閱讀（深度探究章節中有一個這方面的例子）。此外，為了支援資料的標記/標籤化，我們需要為每個標籤添加索引。而且，通用關聯式資料庫在持續的重度寫入負載下表現不佳。在我們這種規模下，我們需要投入大量精力來調優資料庫，即便如此，它的表現也可能不盡如人意。

那麼 NoSQL 呢？理論上，市場上的一些 NoSQL 資料庫可以有效地處理時間序列資料。例如，Cassandra 和 Bigtable [11] 都可以用於時間序列資料。然而，這將需要對每個 NoSQL 的內部工作原理有深入的了解，以設計出能夠有效儲存和查詢時間序列資料的可擴展架構。由於現成的工業級時間序列資料庫（TSDB）已經非常成熟，使用通用 NoSQL 資料庫並不是一個理想的選擇。

目前有許多針對時間序列資料進行優化的儲存系統。這種優化使我們能夠使用遠少於通用資料庫的伺服器來處理相同規模的資料。其中許多資料庫還擁有專門為分析時間序列資料而設計的自訂查詢介面，這些介面比 SQL 易用得多。有些甚至提供了管理資料保留和資料彙總的功能。以下是時間序列資料庫的一些範例。

OpenTSDB 是一個分散式時間序列資料庫，但由於它基於 Hadoop 和 HBase，運行 Hadoop/HBase 叢集會增加複雜性。Twitter 使用的是 MetricsDB [12]，亞馬遜（Amazon）提供 Timestream 時間序列資料庫 [13]。根據 DB-engines [14] 的資料，最受歡迎的兩個時間序列資料庫是 InfluxDB [15] 和 Prometheus，它們旨在儲存大量時間序列資料並快速執行即時分析。兩者都主要依賴記憶體快取和磁碟儲存。而且它們在耐久性和效能方面都表現得相當出色。如圖 5 所示，一個擁有 8 核和 32GB RAM 的 InfluxDB 每秒可以處理超過 250,000 次寫入。

![Figure5.5.png](../images/v2/chapter05/Figure5.5.png)

圖 5 InfluxDB 基準測試

由於時間序列資料庫是專用資料庫，除非你在簡歷中明確提到過，否則在面試中你不需要了解其內部細節。就面試而言，重要的是了解指標資料在本質上是時間序列，我們可以選擇像 InfluxDB 這樣的時間序列資料庫來儲存它們。

一個強大的時間序列資料庫的另一個特點是能夠根據標籤（在某些資料庫中也稱為 tags）對大量時間序列資料進行高效的彙總和分析。例如，InfluxDB 為標籤構建索引以方便通過標籤快速查找時間序列 [15]。它提供了關於如何使用標籤的清晰最佳實踐指南，且不會使資料庫過載。關鍵是要確保每個標籤都是低基數（low cardinality）的（即擁有一組較小的可能值）。這一特性對於可視化至關重要，而使用通用資料庫來實現這一點將需要大量的工作。

## 高層設計

高層設計圖如圖 6 所示。

![Figure5.6.png](../images/v2/chapter05/Figure5.6.png)

圖 6 高層設計

- **指標源**（Metrics source）。這可以是應用伺服器、SQL 資料庫、訊息佇列等。
- **指標收集器**（Metrics collector）。它收集指標資料並將其寫入時間序列資料庫。
- **時間序列資料庫**（Time-series database）。這將指標資料儲存為時間序列。它通常提供自訂查詢介面，用於分析和彙總大量時間序列資料。它維護標籤索引，以方便通過標籤快速查找時間序列資料。
- **查詢服務**（Query service）。查詢服務使從時間序列資料庫中查詢和檢索資料變得容易。如果我們選擇一個好的時間序列資料庫，這應該是一個非常薄的封裝層。它也可以完全被時間序列資料庫自身的查詢介面所取代。
- **告警系統**（Alerting system）。這會向各種告警接收端發送告警通知。
- **可視化系統**（Visualization system）。這以各種圖形/圖表的形式展示指標。

## 拉取模型 vs 推送模型 (Pull vs push models)

采集指標資料有兩種方式：拉取（pull）或推送（push）。關於哪種方式更好一直存在常見的爭論點，且沒有標準答案。讓我們仔細研究一下。

### 拉取模型 (Pull model)

圖 8 顯示了通過 HTTP 進行拉取模型的資料采集。我們有專門的指標收集器，定期從運行中的應用程式 中拉取指標值。

![Figure5.8.png](../images/v2/chapter05/Figure5.8.png)

圖 8 拉取模型

在這種方法中，指標收集器需要知道要從中拉取資料的所有服務端點（service endpoints）的完整列表。一種初級的方法是在「指標收集器」伺服器上使用一個檔案來保存每個服務端點的 DNS/IP 資訊。雖然這個想法很簡單，但在大規模環境下很難維護，因為伺服器會被頻繁地添加或移除，而我們要確保指標收集器不會錯過從任何新伺服器采集指標。好消息是，我們有一種可靠、可擴展且可維護的解決方案，即通過 **服務發現（Service Discovery）** 來實現，該功能由 etcd [16]、ZooKeeper [17] 等提供；其中，服務註冊其可用性，指標收集器可以在服務端點列表發生變化時由服務發現元件通知。

服務發現包含有關何時以及何處采集指標的配置規則，如圖 9 所示。

![Figure5.9.png](../images/v2/chapter05/Figure5.9.png)

圖 9 服務發現

圖 10 詳細解釋了拉取模型。

![Figure5.10.png](../images/v2/chapter05/Figure5.10.png)

圖 10 拉取模型細節

1. 指標收集器從服務發現中獲取服務端點的配置中繼資料。中繼資料包括拉取間隔、IP 位址、超時和重試參數等。
2. 指標收集器通過預定義的 HTTP 端點（例如 `/metrics`）拉取指標資料。為了暴露該端點，通常需要在服務中添加一個用戶端庫。在圖 10 中，該服務是 Web 伺服器。
3. （可選）指標收集器可以在服務發現中註冊變更事件通知，以便在服務端點發生變化時接收更新。或者，指標收集器可以定期輪詢端點變更。

在我們的規模下，單個指標收集器將無法處理成千上萬台伺服器。我們必須使用一個指標收集器池來處理需求。當存在多個收集器時，一個常見的問題是多個實例可能會嘗試從同一個資源拉取資料並產生重複資料。實例之間必須存在某種協調方案（coordination scheme）來避免這種情況。

一種潛在的方法是在一致性哈希環（consistent hash ring）中為每個收集器指定一個範圍，然後將每個被監控的伺服器通過其唯一的名稱映射到哈希環上。這確保了一台指標源伺服器僅由一個收集器處理。讓我們看一個例子。

如圖 11 所示，有四個收集器和六台指標源伺服器。每個收集器負責從一組不同的伺服器收集指標。收集器 2 負責從伺服器 1 和伺服器 5 收集指標。

### 推送模型 (Push model)

如圖 12 所示，在推送模型中，各種指標源（如 Web 伺服器、資料庫伺服器等）直接將指標發送給指標收集器。

![Figure5.12.png](../images/v2/chapter05/Figure5.12.png)

圖 12 推送模型

在推送模型中，通常在每台被監控的伺服器上安裝一個**收集代理**（collection agent）。收集代理是一個長期運行的軟體，它從伺服器上運行的服務中收集指標，並定期將這些指標推送給指標收集器。收集代理還可以在將指標發送給指標收集器之前，在本地對指標進行彙總（特別是簡單的計數器）。

彙總是減少發送到指標收集器的資料量的有效方法。如果推送流量很高，且指標收集器因錯誤拒絕了推送，代理可以在本地保留一個小型的資料緩衝區（可能通過將其儲存在本地磁碟上），並在稍後重試。然而，如果伺服器處於經常輪換的自動擴縮容組（auto-scaling group）中，那麼當指標收集器延遲時，本地保留資料（即使是暫時的）可能會導致資料丟失。

為了防止指標收集器在推送模型中延遲，指標收集器應該處於一個前面掛有負載均衡器的自動擴縮容叢集中（圖 13）。該叢集應根據指標收集器伺服器的 CPU 負載進行橫向擴縮容。

![Figure5.13.png](../images/v2/chapter05/Figure5.13.png)

圖 13 負載均衡器

### 拉取還是推送？ (Pull or push?)

那麼，哪一個對我們來說是更好的選擇呢？就像生活中的許多事情一樣，沒有明確的答案。雙方都有廣泛採用的實際用例。

- 拉取架構的範例包括 Prometheus。
- 推送架構的範例包括 Amazon CloudWatch [18] 和 Graphite [19]。

了解每種方法的優缺點比在面試中為它們分出高下更重要。表 3 比較了推送和拉取架構的優缺點 [20] [21] [22] [23]。

| | 拉取 (Pull) | 推送 (Push) |
| :--- | :--- | :--- |
| **易於除錯** | 應用程式伺服器上的 `/metrics` 端點可用於隨時查看指標。你甚至可以在筆記型電腦上執行此操作。**拉取模式勝出。** | |
| **健康檢查** | 如果應用程式伺服器對拉取沒有回應，你可以快速確定該應用程式伺服器是否已宕機。**拉取模式勝出。** | 如果指標收集器沒有收到指標，問題可能是由網路問題引起的。 |
| **短壽命任務** | | 一些批處理作業（batch jobs）可能是短壽命的，持續時間不足以被拉取。**推送模式勝出。** 這可以通過為拉取模型引入推送閘道（push gateways）來解決 [24]。 |
| **防火牆或複雜的網路設置** | 拉取指標的伺服器需要所有指標端點都是可達的。這在多資料中心設置中可能會有問題。它可能需要更複雜的網路基礎設施。 | 如果指標收集器配置了負載均衡器和自動擴縮容組，則可以從任何地方接收資料。**推送模式勝出。** |
| **效能** | 拉取方法通常使用 TCP。 | 推送方法通常使用 UDP。這意味著推送方法提供了更低延遲的指標傳輸。這裡對應的觀點是，建立 TCP 連接的開銷與發送指標負載相比是很小的。 |
| **資料真實性** | 要從中收集指標的應用程式伺服器是在配置檔案中預先定義的。從這些伺服器收集的指標保證是真實的。 | 任何類型的用戶端都可以將指標推送給指標收集器。這可以通過將接受指標的伺服器列入白名單，或通過要求身份驗證來解決。 |

表 3 拉取 vs 推送

如上所述，拉取還是推送是一個常規的辯論話題，沒有明確的答案。一個大型組織可能需要同時支援兩者，特別是考慮到如今無伺服器（serverless）[25] 的流行。在某些情況下，可能根本無法安裝用於推送資料的代理。

## 擴展指標傳輸流水線 (Scale the metrics transmission pipeline)

![Figure5.14.png](../images/v2/chapter05/Figure5.14.png)

讓我們放大觀察指標收集器和時間序列資料庫。無論你使用推送還是拉取模型，指標收集器都是一個伺服器叢集，該叢集接收海量資料。無論是推送還是拉取，指標收集器叢集都配置為自動擴縮容，以確保有足夠數量的收集器實例來處理需求。

然而，如果時間序列資料庫不可用，則存在資料丟失的風險。為了緩解這個問題，我們引入了一個佇列元件，如圖 15 所示。

![Figure5.15.png](../images/v2/chapter05/Figure5.15.png)

圖 15 添加佇列

在此設計中，指標收集器將指標資料發送到像 Kafka 這樣的佇列系統。然後，消費者或串流處理服務（如 Apache Storm、Flink 和 Spark）處理資料並將其推送給時間序列資料庫。這種方法有幾個優點：

- Kafka 被用作高度可靠且可擴展的分散式訊息平台。
- 它將資料采集服務和資料處理服務解耦。
- 通過將資料保留在 Kafka 中，可以在資料庫不可用時輕鬆防止資料丟失。

### 通過 Kafka 進行擴展 (Scale through Kafka)

我們可以利用 Kafka 內置的分區機制（partition mechanism）通過幾種方式來擴展我們的系統。

- 根據吞吐量需求配置分區數量。
- 根據指標名稱對指標資料進行分區，以便消費者可以按指標名稱彙總資料。

![Figure5.16.png](../images/v2/chapter05/Figure5.16.png)

圖 16 Kafka 分區

- 利用標籤/標記（tags/labels）對指標資料進行進一步分區。
- 對指標進行分類並確定優先順序，以便優先處理重要的指標。

### Kafka 的替代方案 (Alternative to Kafka)

維護一個生產規模的 Kafka 系統绝非易事。你可能會收到面試官關於此點的質疑。目前已有大規模的監控攝取系統在不使用中間佇列的情況下運行。Facebook 的 Gorilla [26] 記憶體時間序列資料庫就是一個典型的例子；它被設計為在發生局部網路故障時仍能保持寫入的高可用性。可以認為，這樣的設計與擁有像 Kafka 這樣的中間佇列一樣可靠。

### 彙總可以在哪裡發生 (Where aggregations can happen)

指標可以在不同的地方進行彙總：在收集代理（用戶端）、攝取流水線（寫入儲存之前）以及查詢端（寫入儲存之後）。讓我們仔細研究一下它們。

**收集代理（Collection agent）**。安裝在用戶端的收集代理僅支援簡單的彙總邏輯。例如，在將計數器發送給指標收集器之前，每分鐘對其進行一次彙總。

**攝取流水線（Ingestion pipeline）**。要在將資料寫入儲存之前彙總資料，我們通常需要諸如 Flink 之類的串流處理引擎。由於僅將計算結果寫入資料庫，寫入量將顯著減少。然而，處理延遲到達的事件（late-arriving events）可能是一個挑戰，另一個缺點是我們失去了資料精度和一些靈活性，因為我們不再儲存原始資料。

**查詢端（Query side）**。原始資料可以在查詢時在給定的時間段內進行彙總。使用這種方法不會丟失資料，但查詢速度可能會較慢，因為查詢結果是在查詢時計算的，並且需要針對整個資料集運行。

## 查詢服務 (Query service)

查詢服務由一個查詢伺服器叢集組成，它們存取時間序列資料庫並處理來自可視化系統或告警系統的請求。擁有專門的一組查詢伺服器可以將時間序列資料庫與用戶端（可視化和告警系統）解耦。這使我們能夠根據需要隨時靈活地更改時間序列資料庫或可視化及告警系統。

### 快取層 (Cache layer)

為了減輕時間序列資料庫的負載並使查詢服務更具效能，添加了快取伺服器來儲存查詢結果，如圖 17 所示。

![Figure5.17.png](../images/v2/chapter05/Figure5.17.png)

### 不使用查詢服務的理由 (The case against query service)

實際上並沒有迫切的需求來引入我們自己的抽象（查詢服務），因為大多數工業級規模的可視化和告警系統都有強大的插件來與市場上知名的時間序列資料庫對接。而且如果選擇了合適的時間序列資料庫，通常也不需要額外添加我們自己的快取。

### 時間序列資料庫查詢語言 (Time-series database query language)

大多數流行的指標監控系統（如 Prometheus 和 InfluxDB）不使用 SQL，而是擁有自己的查詢語言。一個主要原因是 SQL 很難用於查詢時間序列資料。例如，正如這裡 [27] 所提到的，在 SQL 中計算指數移動平均（exponential moving average）可能如下所示：

```sql
select id,
    temp,
    avg (temp) over ( partition by group_nr order by time_read) as rolling_avg
from (
    select id,
    time,
    time_read,
    interval_group,
    id - row_number() over (partition by interval_group order by time_read) as group_nr
    from (
        select time_read,
        "epoch ":: timestamp + "900 seconds ":: interval * (
        extract ( epoch from time_read ):: int4 / 900) as interval_group,
        temp,
        from readings,
    ) t1
) t2
order by time_read;
```

而在 Flux（一種針對時間序列分析優化的語言，用於 InfluxDB）中，它看起來像這樣。如你所見，它要容易理解得多。

```flux
from(db:"telegraf")
|> range(start:-1h)
|> filter(fn: (r) => r._measurement == "foo")
|> exponentialMovingAverage(size:-10s)
```

## 儲存層 (Storage layer)

現在讓我們深入探討儲存層。

### 仔細選擇時間序列資料庫 (Choose a time-series database carefully)

根據 Facebook 發表的一篇研究論文 [26]，運維資料儲存中至少 85% 的查詢是針對過去 26 小時內收集的資料。如果我們使用的時間序列資料庫能夠利用這一特性，將對系統的整體效能產生重大影響。如果你對儲存引擎的設計感興趣，請參考 InfluxDB 儲存引擎的設計文檔 [28]。

### 空間優化 (Space optimization)

正如在高層需求中所解釋的，需要儲存的指標資料量是巨大的。這裡有幾種解決策略。

#### 資料編碼和壓縮 (Data encoding and compression)

資料編碼和壓縮可以顯著減小資料的大小。這些功能通常內置在優秀的時間序列資料庫中。下面是一個簡單的例子。

**雙重差分編碼 (Double-delta Encoding)**

![Figure5.18.png](../images/v2/chapter05/Figure5.18.png)

圖 18 資料編碼

如上圖所示，1610087371 和 1610087381 僅相差 10 秒，這僅需 4 位（bits）即可表示，而無需使用 32 位的完整時間戳。因此，與其儲存絕對值，不如儲存數值的差值（delta）以及一個基準值，例如：*1610087371, 10, 10, 9, 11*。

#### 降採樣 (Downsampling)

降採樣是將高解析度資料轉換為低解析度資料以減少整體磁碟佔用的過程。由於我們的資料保留期為 1 年，我們可以對舊資料進行降採樣。例如，我們可以讓工程師和資料科學家為不同的指標定義規則。下面是一個例子：

- 保留期：7 天，不採樣
- 保留期：30 天，降採樣至 1 分鐘解析度
- 保留期：1 年，降採樣至 1 小時解析度

讓我們看另一個具體的範例。它將 10 秒解析度的資料彙總（aggregates）為 30 秒解析度的資料。

| cpu | 2021-10-24T19:00:10Z | host-a | 16 |
| cpu | 2021-10-24T19:00:20Z | host-a | 20 |
| cpu | 2021-10-24T19:00:30Z | host-a | 30 |
| cpu | 2021-10-24T19:00:40Z | host-a | 20 |
| cpu | 2021-10-24T19:00:50Z | host-a | 30 |

冷儲存是對不常用且處於非活躍狀態的資料進行的儲存。冷儲存的財務成本要低得多。

## 告警系統 (Alert system)

出於面試的目的，讓我們來看看圖 19 所示的告警系統。

![Figure5.19.png](../images/v2/chapter05/Figure5.19.png)

圖 19 告警系統

告警流程如下：

1. 將配置檔案載入快取伺服器。規則以磁碟上的配置檔案形式定義。YAML [29] 是定義規則的常用格式。下面是一個告警規則的範例：

```yaml
- name: instance_down
  rules:

  # 為任何無法存取超過 5 分鐘的執行個體 (instance) 發送告警 (Alert for)。
  - alert: instance_down
    expr: up == 0
    for: 5m
    labels:
      severity: page
```

2. 告警管理器從快取中獲取告警配置。
3. 基於配置規則，告警管理器以預定義的時間間隔調用查詢服務。如果數值違反了閾值，則建立一個告警事件。告警管理器負責以下工作：

- 對告警進行過濾、合併和去重。
- 範例：合併在一個執行個體 (instance1) 內觸發的告警。

![Figure5.20.png](../images/v2/chapter05/Figure5.20.png)

圖 20 合併告警

- 存取控制：將某些告警管理操作的存取限制在授權個人範圍內。
- 重試：通過檢查告警狀態，確保通知至少發送一次。

4. 告警儲存是一個鍵值資料庫，例如 Cassandra，它保存所有告警的狀態（非活躍、待定、正在觸發、已解決）。它確保通知至少發送一次。
5. 符合條件的告警被插入 Kafka。
6. 告警消費者從 Kafka 拉取告警事件。
7. 告警消費者處理來自 Kafka 的告警事件，並向不同的管道發送通知，如電子郵件、簡訊、PagerDuty 或 HTTP 端點。

### 告警系統 - 自建還是購買 (Alert system - build vs buy)

市面上有很多現成的工業級告警系統，而且大多數都提供了與流行時間序列資料庫的緊密整合。這些告警系統中的許多都能與現有的通知管道（如電子郵件和 PagerDuty）很好地整合。在現實世界中，很難找到充分的理由來自己建構告警系統。在面試場景中，尤其是對於高級職位，要準備好為你的決定辩解。
