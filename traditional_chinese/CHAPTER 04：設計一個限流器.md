# 第04章：設計一個限流器


在網路系統中，限流器被用於控制用戶端或伺服端發送流量的速率。在HTTP世界中，限流器限制在指定時間內允許發送用戶端請求數。 如果API請求次數超過了限流器設定的閾值，則所有超出的呼叫都會被阻止。 以下是一些範例：

* 單個使用者每秒最多可以發布2則貼文
* 每天同一IP位址最多可以建立10個帳戶。
* 同一裝置每週最多可以領取5次獎勵。

在本章中，您需要設計一個限流器。在開始設計之前，我們先看看使用 API 限流器的好處：

* 防止拒絕服務 (DoS) 攻擊造成的資源匱乏 \[1]。 幾乎大型科技公司發布的所有 API 都強制執行某種形式的速率限制。 例如，Twitter將每3小時的推文數量限制為300條\[2]。 Google docs APIs有如下預設限制：每個使用者每60秒讀取請求為300次\[3]。 限流器透過阻止多餘的呼叫來防止DoS攻擊，無論是有意還是無意的。
* 降低成本。限制過多的請求意味著減少伺服器數量，將更多資源分配給高優先級的API。 速率限制對於使用付費第三方API的公司來說極為重要。 例如，你對以下外部API的呼叫是按次數收費的：檢查信用、付款、檢索健康記錄等。限制呼叫次數對減少成本至關重要。
* 防止伺服器過載。為了減少伺服器的負荷，使用限流器來過濾由機器人或使用者的不當行為造成的過量請求。

### 第1步：了解問題並確定設計範圍

速率限制可以透過不同的演算法來實現，每一種演算法都有其優點和缺點。 面試官和候選人之間的互動有助於闡明我們試圖建立的限流器的類型。

候選人：我們要設計什麼樣的限流器？是用戶端的限流器還是伺服端的API限流器？

面試官：好問題，我們專注於伺服端的API限流器。

候選人：限流器是否根據IP、使用者ID或其他屬性來限制API請求？

面試官：限流器應該足夠靈活，能夠支援不同的限流規則集。

候選人：該系統的規模是多少？它是為新創企業還是擁有龐大使用者群的大公司建立的？

面試官：該系統必須能夠處理大量的請求。

候選人：系統會在分散式環境下工作嗎？

面試官：是的。

候選人：限流器是一個單獨的服務還是應該在應用程式程式碼中實現？

面試官：這是一個由你決定的設計問題。

候選人：是否需要通知被限流的使用者？

面試官：是的。

**要求**

以下是對系統要求的概述

* 準確地限制過多的請求
* 低延遲：限流器不能降低Http回應時間。
* 盡可能的佔用更少的記憶體。
* 分散式速率限制，要求可以在多個伺服器或程序之間共用。
* 異常處理，當使用者的請求受到限制時，向使用者顯示明確的異常資訊。
* 高容錯性。如果限流器出現任何問題（例如，一個快取伺服器離線），它不會影響整個系統。

### 第2步：提出高層次的設計方案並獲得認同

* 用戶端實現。 一般來說，用戶端是執行速率限制的不可靠場所，因為用戶端請求很容易被惡意使用者偽造。 此外，我們可能無法控制用戶端的實現。
*   伺服端實現。圖4-1顯示了一個放在伺服端的限流器。

![](images/chapter4/figure4-2.jpg)

讓我們使用圖 4-3 中的範例來說明這種設計中的限流器是如何工作的。 假設我們的 API 允許每秒 2 個請求，而用戶端在一秒內發送了3個請求到伺服器。 前兩個請求被路由到 API 伺服器。 但是，限流器中介軟體會限制第三個，請求並返回 HTTP 狀態代碼 429。 HTTP 429 回應狀態代碼表示使用者發送了過多的請求。

![](images/chapter4/figure4-3.jpg)

雲微服務\[4]已經變得非常流行，限流通常在一個叫做API閘道的元件中實現。API閘道是一個完全託管的服務，支援限速、SSL、認證、IP白名單、服務靜態內容等。現在，我們只需要知道API閘道是一個支援限流的中介軟體。

在設計限流器時，要問自己的一個重要問題是：限流器應該在哪裡實現，在伺服端還是在閘道中？這沒有絕對的答案。這取決於你公司目前的技術棧、工程資源、優先級、目標等。這裡有一些通用的指南：

* 評估你目前的技術棧，如程式語言、快取服務等。確保你目前的程式語言能夠有效地在伺服端實現速率限制。
* 確定適合你業務需求的速率限制演算法。當你在伺服端實施一切時，你可以完全控制演算法。然而，如果你使用第三方閘道，你的選擇可能是有限的。
* 如果你已經使用了微服務架構，並在設計中包含了一個API閘道來執行認證、IP白名單等，你可以在API閘道上新增一個限流器。
* 建立你自己的速率限制服務需要時間。如果你沒有足夠的工程資源來實現限流器，商業API閘道是一個更好的選擇。

#### 限流演算法

限流可以使用不同的演算法來實現，每種演算法都有其獨特的優缺點。儘管本章並不關注演算法，但在高層次上了解它們有助於選擇正確的演算法或演算法組合來適應我們的使用情況。

下面是一個流行演算法的列表：

* 令牌桶（Token bucket）
* 漏桶演算法（Leaking bucket）
* 固定視窗計數器（Fixed window counter）
* 滑動視窗日誌（Sliding window log）
* 滑動視窗計數器（Sliding window counter）

**令牌桶演算法**

令牌桶（Token bucket）演算法廣泛用於速率限制。 它簡單易懂，常被網際網路公司採用。 Amazon \[5] 和 Stripe \[6] 都使用這個演算法來限制他們的 API 請求。

**令牌桶演算法的工作原理如下：**

* 令牌桶是一個具有預定義容量的容器。 令牌會以預定的速率定期放入桶中， 一旦桶滿了，就不再新增令牌。 如圖4-4所示，令牌桶容量為4，注入裝置每秒向桶中放入2個令牌，一旦桶滿了，多餘的令牌就會溢出。 ![](images/chapter4/figure4-4.jpg)
*   每個請求都會消耗一個令牌。當一個請求到達時，我們檢查桶中是否有足夠的令牌。圖4-5解釋了它是如何工作的。

    * 如果有足夠的令牌，我們會為每個請求取出一個令牌，然後請求通過。
    * 如果沒有足夠的令牌，則該請求被丟棄。

    ![](images/chapter4/figure4-5.jpg)

圖 4-6 說明了令牌消耗、重新填充和速率限制邏輯的工作原理。 在此範例中，令牌桶大小為 4，重新填充速率為每 1 分鐘 4 個。

![](images/chapter4/figure4-6.jpg)

令牌桶演算法需要兩個參數：

1. 桶大小：桶內允許的最大令牌數。
2. 填充速率：每秒放入到桶內的令牌數量。

我們需要多少個桶？這取決於限流規則，並且會有所不同。 以下是幾個例子。

* 通常需要為不同的API端點使用不同的桶。例如，如果一個使用者被允許每秒發1則貼文，每天新增150個好友，並且每秒鐘按讚5則貼文，則每個使用者需要3個桶。
* 如果我們需要根據IP位址對請求進行限流，每個IP位址都需要一個桶。
* 如果系統允許每秒最多10,000個請求，則有一個全域桶供所有請求共享是有意義的。

**優點**

* 演算法容易實現
* 佔用記憶體少
* 令牌桶允許在短時間內進行突發流量。只要有剩餘的令牌，請求就可以通過。

**缺點**

* 演算法中有兩個參數，即桶的大小和令牌的補充速率。然而，正確調整它們可能具有挑戰性。

**漏桶演算法**

漏桶（Leaking bucket）演算法與令牌桶類似，不同之處在於請求是以固定速率處理的。它通常用先入先出（FIFO）佇列來實現。

該演算法的工作原理如下：

* 當請求到達時，系統會檢查佇列是否已滿。如果佇列未滿，則將請求新增到佇列中。
* 否則，將丟棄該請求。
* 請求會在固定的時間間隔內從佇列中取出並進行處理。

圖4-7解釋了該演算法的工作原理：

![](images/chapter4/figure4-7.jpg)

漏桶演算法需要以下兩個參數

* 桶的大小：它等於佇列的大小。佇列容納了要以固定速度處理的請求。
* 流出率：定義了在固定速率下可以處理多少請求，通常以秒為單位。

Shopify，一家電子商務公司，使用漏桶來限制速度\[7]。

**優點：**

* 鑒於佇列大小有限，記憶體效率高。
* 請求以固定的速率處理，因此它適用於需要穩定流出速率的用例。

**缺點：**

* 突發的流量使佇列中充滿了舊的請求，如果這些請求沒有得到及時處理，最近的請求將受到速率限制。
* 演算法中有兩個參數，要適當地調整它們可能並不容易。

**固定視窗計數器演算法**

固定視窗計數器（Fixed window counter）演算法工作原理如下：

* 該演算法將時間軸劃分為固定大小的時間視窗，並為每個視窗分配一個計數器。
* 每個請求將計數器增加一。
* 一旦計數器達到預定的閾值，新的請求就會被放棄，直到新的時間視窗開始。

讓我們用一個具體的例子來看看它是如何工作的。在圖4-8中，時間單位是1秒，系統允許每秒鐘最多有3個請求。在每個秒視窗中，如果收到的請求超過3個，額外的請求就會被放棄，如圖4-8所示： ![](images/chapter4/figure4-8.jpg)

該演算法的一個主要問題是，在時間視窗邊緣的突發流量可能會導致超過允許配額的請求。考慮以下情況：

![](images/chapter4/figure4-9.jpg)

在圖4-9中，系統允許每分鐘最多有5個請求，可用的配額在整點分鐘時重置。如圖所示，在 2:00:00 和 2:01:00 之間有5個請求，在 2:01:00 和 2:02:00 之間又有五個請求。 在 2:00:30 和 2:01:30 之間的1分鐘視窗，有10個請求通過了。這是允許請求數量的兩倍。

**優點：**

* 記憶體高效
* 容易理解
* 在單位時間視窗結束時重新設定可用配額，適合某些使用情況

**缺點：**

* 視窗邊緣的流量激增可能導致超過允許配額的請求被通過（有突刺）

**滑動視窗日誌演算法**

如前所述，固定視窗計數（Fixed window log）演算法有一個主要問題：它允許更多的請求在視窗的邊緣通過。滑動視窗日誌演算法解決了這個問題。 它的工作原理如下：

* 該演算法對請求的時間戳進行跟蹤。時間戳資料通常儲存在快取中，如Redis的sorted\[8] 。
* 當一個新的請求進來時，刪除所有過期的時間戳。過時的時間戳被定義為比當前時間視窗的開始時間更早的時間戳。
* 將新請求的時間戳新增到日誌中
* 如果日誌大小與允許的計數相同或更低，則接受請求。否則，它將被拒絕

我們用圖4-10所示的一個例子來解釋該演算法。

![](images/chapter4/figure4-10.jpg)

在這個例子中，限流器允許每分鐘2個請求。通常情況下，Linux的時間戳會儲存在日誌中。然而，在我們的例子中，為了提高可讀性，使用了人類可讀的時間表示法。

* 當一個新的請求在 $$1:00:01$$ 到達時，該日誌是空的。因此，該請求被允許。
* 一個新的請求在 $$1:00:30$$ 到達，時間戳 $$1:00:30$$ 被插入到日誌中。插入後，日誌大小為2，不大於允許的數量，因此，該請求被允許。
* 一個新的請求在 $$1:00:50$$ 到達，時間戳被插入到日誌中。插入後，日誌大小為3，大於允許的大小2。因此，這個請求被拒絕，儘管時間戳仍然在日誌中。
* 一個新的請求在 $$1:01:40$$ 到達。在 $$\left [1:00:40,1:01:40 \right]$$ 範圍內的請求是在最新的時間範圍內，但在 $$1:00:40$$ 之前發送的請求是過時的。
* 兩個過期的時間戳 $$1:00:01$$ 和 $$1:00:30$$ 被從日誌中刪除。在刪除操作之後，日誌大小變成了2；因此，請求被接受。

**優點：**

* 這種演算法實現的速率**限制是非常準確**的。在任何滾動視窗中，請求都不會超過速率限制。

**缺點：**

* 該演算法消耗了大量的記憶體，因為即使一個請求被拒絕，其時間戳仍可能被儲存在記憶體中

**滑動視窗計數器演算法**

滑動視窗計數器(Sliding window counter)演算法是一種混合方法，結合了固定視窗計數器和滑動視窗日誌。 該演算法可以透過兩種不同的方法來實現。我們將在本節中解釋一種實現方法，並在本節末尾提供另一種實現方法的參考。

圖4-11說明了這種演算法的工作原理：

![](images/chapter4/figure4-11.jpg)

假設限流器允許每分鐘最多有7個請求，在上一分鐘有5個請求，當前一分鐘有3個請求。對於在當前分鐘內到達30%位置的新請求，滾動視窗中的請求數用以下公式計算：

* 當前視窗中的請求數量 + 上一個視窗中的請求數量 \* 滾動視窗和上一個視窗的重疊百分比
* 使用這個公式，我們得到 $$3 + 5 \times 0.7 \% = 6.5$$ 個請求。根據不同的使用情況，這個數字可以向上或向下取整。在我們的例子中，它被向下四捨五入為6。

由於限流器每分鐘最多允許7個請求，當前的請求可以通過。然而，再收到一個請求後，就會達到限制。

由於篇幅所限，我們在此不討論其他的實現。有興趣的讀者可以參考參考資料\[9]。這種演算法並不完美。它有優點也有缺點。

**優點：**

* 它平滑了流量的峰值，因為速率是基於前一個視窗的平均速率。
* 記憶體高效

**缺點：**

* 它只適用於不太嚴格的回看視窗。它是實際速率的近似值，因為它假設前一個視窗的請求是均勻分佈的。然而，這個問題可能並不像它看起來那麼糟糕。 根據Cloudflare\[10]所做的實驗，在4億個請求中，只有0.003%的請求被錯誤地允許或限制速率

#### 高層次的架構

限流演算法的基本思想很簡單。在高層次上，我們需要一個計數器來跟蹤來自同一使用者、IP位址等的多少個請求。如果計數器大於限制值，則請求被禁止。

我們應該在哪裡儲存計數器？由於磁碟存取速度慢，使用資料庫並不是一個好主意。選擇記憶體快取是因為它速度快並且支援基於時間的過期策略。 例如，Redis\[11]是實現速率限制的一個流行選擇。它是一個記憶體中的儲存，提供兩個命令：`INCR`和`EXPIRE`

* `INCR`：它使儲存的計數器加1。
* `EXPIRE`：它為計數器設定一個超時。如果超時過後，計數器會被自動刪除。

圖4-12顯示了速率限制的高層結構，其工作原理如下：

![](images/chapter4/figure4-12.jpg)

* 用戶端向限流中介軟體發送請求
* 限流中介軟體從Redis中相應的桶中獲取計數器，並檢查是否達到限制
  * 如果達到限制，則拒絕該請求
  * 如果沒有達到限制，請求會被發送到API伺服器。同時，系統會增加計數器並將其儲存回Redis。

#### 限流規則

`Lyft`開源了他們的速率限制元件\[12]。我們將窺探該元件的內部情況，並看看一些速率限制規則的例子。

```yaml
domain: messaging
descriptors:
  - key: message_type
    Value: marketing
    rate_limit:
      unit: day
      requests_per_unit: 5
```

在上述例子中，系統被設定為每天最多允許5條行銷訊息。下面是另一個例子：

```yaml
domain: auth
descriptors:
  - key: auth_type
    Value: login
    rate_limit:
      unit: minute
      requests_per_unit: 5
```

這個規則顯示，客戶不允許在1分鐘內登入超過5次。規則一般寫在設定檔中並儲存在磁碟上。

#### 超過速率限制

如果一個請求被限制了速率，API會向用戶端返回一個HTTP回應代碼429（請求太多）。根據不同的使用情況，我們可能會將速率受限的請求排隊等候以後處理。 例如，如果一些訂單由於系統過載而受到速率限制，我們可以保留這些訂單以便以後處理。

#### 限流器請求頭

一個客戶如何知道它是否被節流？用戶端如何知道在被節流之前允許的剩餘請求的數量？答案就在HTTP回應頭中。限流器向用戶端返回以下 HTTP 標頭：

* X-Ratelimit-Remaining：視窗內允許請求的剩餘數量
* X-Ratelimit-limit：它表示用戶端在每個時間視窗可以進行多少次呼叫
* X-Ratelimit-Retry-After：等待的秒數，直到你可以再次提出請求而不被節流。

當使用者發送過多請求時，將向用戶端返回 429 too many requests 錯誤和 X-Ratelimit-Retry-After 標頭。

#### 詳細設計

圖 4-13 給出了系統的詳細設計。

![](images/chapter4/figure4-13.jpg)

* 規則被儲存在磁碟上。工作者經常從磁碟中提取規則，並將其儲存在高速快取中。
* 當用戶端向伺服器發送請求時，該請求首先被發送到限流中介軟體。
* 限流中介軟體從快取中載入規則。它從Redis快取中獲取計數器和最後一次請求的時間戳。根據回應，限流器決定：
  * 如果請求沒有速率限制，它將被轉發到API伺服器。
  * 如果請求受到速率限制，限流器會向用戶端返回 429 too many requests 錯誤。 同時，請求被丟棄或轉發到佇列。

#### 分散式環境下的限流器

構建一個在單個伺服器環境下執行的限流器並不困難。然而，將系統擴展到支援多個伺服器和並發執行緒是另一回事。這裡有兩個挑戰：

* 競爭條件
* 同步問題

**競爭條件**

如前所述，限流器在高層的工作原理如下


- 從Redis讀取計數器的值
- 檢查 ( 計數器 + 1 ) 是否超過閾值
- 如果不是，則將 Redis 中的計數器值加 1


如圖4-14所示，在高度並發的環境中會發生競爭條件。

![](images/chapter4/figure4-14.jpg)

假設 Redis 中的計數器值為 3。如果兩個請求在其中一個請求寫回值之前同時讀取計數器值，則每個請求都會將計數器加 1 並在不檢查另一個執行緒的情況下將其寫回。 兩個請求（執行緒）都認為它們具有正確的計數器值 4。但是，正確的計數器值應該是 5

鎖是解決競爭條件最顯然的解決方案。 但是，鎖會顯著降低系統速度。 通常使用兩種策略來解決這個問題： Lua 腳本 [[13](#ref13)] 和 Redis [[8](#ref8)] 中的 sorted sets 資料結構。 對這些策略感興趣的讀者可以參考相應的參考資料[[8](#ref8)] [[13](#ref13)]。

**同步問題**

同步是分散式環境中要考慮的另一個重要因素。 要支援數百萬使用者，一個限流器伺服器可能不足以處理流量。 當使用多個限流器伺服器時，需要同步。 例如，在圖 4-15 的左側，用戶端 1 將請求發送到限流器 1，用戶端 2 將請求發送到限流器 2。 由於 Web 層是無狀態的，用戶端可以將請求發送到不同的限流器，如圖所示，在圖 4-15 的右側。 如果沒有發生同步，限流器 1 不包含有關用戶端 2 的任何資料。因此，限流器無法正常工作。 ![](images/chapter4/figure4-15.jpg)

一種可能的解決方案是使用黏性會話，允許用戶端將流量發送到相同的限流器。 這個解決方案是不可取的，因為它既不可擴展也不靈活。 更好的方法是使用像 Redis 這樣的集中式資料儲存。

#### 效能優化

效能優化是系統設計面試中的一個常見話題。我們將涉及兩個方面的改進。

首先，多資料中心的設定對限流器至關重要，因為對於遠離資料中心的使用者來說，延遲很高。大多數雲端服務提供商在世界各地建立了許多邊緣伺服器位置。 例如，截至2020年5月20日，Cloudflare有194個地理上分佈的邊緣伺服器\[14]。流量被自動路由到最近的邊緣伺服器，以減少延時。

#### 監控

限流器到位後，最重要的是要收集分析資料以檢查限流器是否有效。

首先，我們要確保：

* 限流演算法是有效的
* 限流規則是有效的

例如，如果速率限制規則過於嚴格，則會丟棄許多有效請求。 在這種情況下，我們想稍微放寬規則。 在另一個範例中，我們注意到當流量突然增加時（如搶購），我們的限流器變得無效。 在這種場景下，我們可能會更換演算法來支援突發流量。 令牌桶很適合這種場景。

* 固定視窗計數器

* 滑動視窗計數器

然後，我們討論了系統架構、分散式環境中的限流器、效能優化和監控。 與任何系統設計面試問題類似，如果時間允許，您可以提及其他談話要點:

* 硬體 與 軟體限流對比
  * 硬體：請求的數量不能超過閾值
  * 軟體：請求可以在短時間內超過閾值
* 不同級別的限流。在本章中，我們只討論了應用程式級別（HTTP：第 7 層）的限流，可以在其他層應用限流。例如，你可以使用Iptables\[15]（IP：第3層）按IP位址應用速率限制。 注意：開放系統互連模型（OSI模型）有7層\[16]。第1層：物理層，第2層：資料鏈路層，第3層：網路層，第4層：傳輸層，第5層：會話層，第6層：表示層，第7層：應用層。
* 避免被限速，使用最佳實踐設計你的用戶端：
  * 使用用戶端快取，以避免頻繁呼叫API
  * 了解限制，不要在短時間內發送太多請求
  * 包含捕獲異常或錯誤的代碼，以便用戶端可以從異常中恢復正常
  * 為重試邏輯增加足夠的回退時間
