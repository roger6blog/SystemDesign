# 第06章：鍵值儲存設計


鍵值儲存，也稱為鍵值資料庫，是一種非關聯式資料庫。每個唯一識別碼都儲存為一個鍵及其關聯值，這種資料配對稱為「鍵值」對。

在鍵值對中，鍵必須是唯一的，通過鍵可以存取與鍵關聯的值。鍵可以是純文字或雜湊值。出於效能原因，短鍵效果更好。鍵是什麼樣子的？

* 純文字鍵：「last\_logged\_in\_at」
* 雜湊鍵：253DDEC4

鍵值對中的值可以是字串、列表、物件等。在鍵值儲存中，值通常被視為不透明的物件，如Amazon dynamo \[1], Memcached \[2], Redis \[3], 等。

![](images/chapter6/table6-1.jpg)

在本章中，你需要設計一個支援以下操作的鍵值儲存：

* `put(key, value)` // 插入與「key」關聯的「value」
* `get(key)` // 取得與「key」關聯的「value」

### 了解問題並確定設計範圍

這裡沒有完美的設計。每種設計都在讀取、寫入和記憶體使用方面取得了特定的權衡。必須在一致性和可用性之間做出另一個權衡。

在本章中，我們設計了一個包含以下特徵的鍵值儲存：

* 鍵值對的大小很小：不到 10 KB。
* 有能力儲存大資料。
* 高可用性：系統回應迅速，即使在出現故障時也是如此。
* 高可擴展性：系統可以擴展以支援大型資料集。
* 自動縮放：伺服器的新增/刪除應該根據流量自動進行。
* 可調整的一致性。
* 低延遲。

### 單一伺服器的鍵值儲存

開發一個駐留在單一伺服器中的鍵值儲存很容易，一種直觀的方法是將鍵值對儲存在雜湊表中，該雜湊表將所有內容保存在記憶體中。

為了在一個伺服器中容納更多的資料，可以做兩個優化措施：

```
- 資料壓縮
- 只在記憶體中儲存經常使用的資料，其餘的儲存在磁碟上
```

即使進行了這些優化，單一伺服器也可以很快達到其容量。需要分散式鍵值儲存來支援大資料

### 分散式鍵值儲存

分散式鍵值儲存也稱為分散式雜湊表，它將鍵值對分佈在許多伺服器上。在設計分散式系統時，了解 CAP（C一致性、A可用性、P分區容錯性）定理很重要。

CAP 定理指出，分散式系統不可能同時提供以下三種保證中的兩種以上：一致性、可用性和分區容錯性。讓我們熟悉一些定義。

**一致性**：一致性意味著所有用戶端無論連接到哪個節點，都在同一時間看到相同的資料。

**可用性**：可用性意味著即使某些節點已關閉，任何請求資料的用戶端都會得到回應。

**分區容忍度**：分區表示兩個節點之間的通訊中斷，分區容錯意味著系統在網絡分區的情況下繼續運行。

CAP 定理指出，必須犧牲三個屬性之一來支援 3 個屬性中的 2 個，如圖 6-1 所示：

如今，鍵值儲存根據它們支援的兩個 CAP 特性進行分類：

![](images/chapter6/figure6-1.jpg)

**CP（一致性和分區容錯）系統**：CP 鍵值儲存在犧牲可用性的同時支援一致性和分區容錯。

**AP（可用性和分區容錯）系統**：AP 鍵值儲存支援可用性和分區容錯，同時犧牲一致性

**CA（一致性和可用性）系統**：CA 鍵值儲存支援一致性和可用性，同時犧牲分區容錯性。由於網路故障是不可避免的，分散式系統必須容忍網路分區。因此，CA 系統不能存在於現實世界的應用程式中。

您在上面閱讀的內容主要是定義部分。為了更容易理解，讓我們看一些具體的例子。在分散式系統中，資料通常會被複製多次。假設資料被複製到三個副本節點n1、n2和n3上，如圖6-2所示。

*   **理想情況**

    在理想世界中，網路分區永遠不會發生。寫入 n1 的資料會自動複製到 n2 和 n3。實現了一致性和可用性。

    ![](images/chapter6/figure6-2.jpg)
*   **真實世界的分散式系統**

    在分散式系統中，分區是不可避免的，當出現分區時，我們必須在一致性和可用性之間做出選擇。圖6-3中，n3宕機，無法與n1、n2通訊。如果用戶端將資料寫入 n1 或 n2，則資料無法傳播到 n3。如果資料寫入 n3 但尚未傳播到 n1 和 n2，則 n1 和 n2 將具有陳舊資料。

    ![](images/chapter6/figure6-3.jpg)

    如果我們選擇一致性大於可用性（CP系統），我們必須阻止所有對n1和n2的寫操作，以避免這三個伺服器之間的資料不一致，這使得系統不可用。銀行系統通常有極高的一致性要求。例如，對於銀行系統來說，顯示最新的餘額資訊是至關重要的。如果由於網路分區而發生不一致，在不一致問題解決之前，銀行系統會返回一個錯誤。

    然而，如果我們選擇可用性大於一致性（AP系統），系統就會一直接受讀取，即使它可能返回陳舊的資料。對於寫，n1和n2將繼續接受寫，當網路分區解決後，資料將被同步到n3。

    選擇正確的 CAP 以確保適合你的用例是構建分散式鍵值儲存的重要一步。你可以與面試官討論這個問題並相應地設計系統

在本節中，我們將討論以下用於構建鍵值儲存的核心元件和技術：

* 寫入路徑

下面的內容主要基於三個流行的鍵值儲存系統：Dynamo \[4]、Cassandra \[5] 和 BigTable \[6]。

### 資料分區

對於大型應用程式，將完整的資料集放在單一伺服器中是不可行的。實現這一點的最簡單方法是將資料拆分為更小的分區並將它們儲存在多個伺服器中。分區資料時有兩個挑戰：

* 跨多個伺服器平均分配資料。
* 當節點被新增或刪除時，盡量減少資料移動。

第 5 章中討論的一致性哈希是解決這些問題的一種很好的技術。讓我們重新審視一致性哈希在高層次上的工作原理。

* 首先，伺服器被放置在哈希環上。在圖 6-4 中，8 個伺服器，分別用 s0、s1、...、s7 表示，放在哈希環上。
*   接下來，將一個鍵散列到同一個環上，並將其儲存在順時針方向移動時遇到的第一個伺服器上。例如，key0 使用此邏輯儲存在 s1 中。

    ![](images/chapter6/figure6-4.jpg)

使用一致性哈希對資料進行分區有以下優點：

* 自動縮放：可以根據負載自動新增和刪除伺服器
* 異質性：伺服器的虛擬節點數與伺服器容量成正比。例如，容量越大的伺服器分配的虛擬節點越多。

### 資料複製

為了實現高可用性和可靠性，必須在 N 個伺服器上非同步複製資料，其中 N 是一個可設定的參數。這N台伺服器的選擇邏輯如下：將key映射到哈希環上的某個位置後，從該位置順時針走，選擇環上的前N台伺服器儲存資料副本。在圖 6-5（N = 3）中，key0 被複製到 s1、s2 和 s3。

![](images/chapter6/figure6-5.jpg)

對於虛擬節點，環上的前 N 個節點可能由少於 N 個物理伺服器擁有。為避免此問題，我們在執行順時針行走邏輯時僅選擇唯一的伺服器。

由於停電、網路問題、自然災害等原因，同一資料中心內的節點經常同時發生故障。為了更好的可靠性，副本被放置在不同的資料中心，資料中心之間通過高速網路連接。

### 一致性

由於資料在多個節點進行複製，因此必須跨副本同步。Quorum 共識可以保證讀寫操作的一致性。 讓我們先建立幾個定義。

N = 副本數

W = 大小為 W 的法定寫入人數。要將寫入操作視為成功，必須從 W 個副本確認寫入操作。

R = 大小為 R 的讀取法定人數。為了使讀取操作被認為是成功的，讀取操作必須等待至少R個副本的回應。

考慮以下圖 6-6 中所示的範例，其中 N = 3。

![](images/chapter6/figure6-6.jpg)

W = 1 並不意味著資料寫在一台伺服器上。 例如，對於圖 6-6 中的設定，資料被複製到 s0、s1 和 s2。 W = 1 表示協調器必須至少收到一個確認才能認為寫操作成功。例如，如果我們收到来自 s1 的確認，我們就不再需要等待来自 s0 和 s2 的確認。 協調器充當用戶端和節點之間的代理。

W、R和N的設定是一個典型的延遲和一致性之間的權衡。如果W = 1 或R = 1，操作會很快返回，因為協調器只需要等待来自一個副本的回應。 如果 W 或 R > 1，系統提供更好的一致性； 但是，查詢會變慢，因為協調器必須等待最慢副本的回應。

如果W+R>N，就能保證強一致性，因為至少有一個重疊的節點擁有最新的資料，以保證一致性。

如何設定N、W和R以適應我們的使用情況？

下面是一些可能的設定：

* 如果R=1，W=N，系統被優化為快速讀取
* 如果W=1，R=N，系統被優化為快速寫入
* 如果W+R>N，就可以保證強一致性（通常N=3，W=R=2）。
* 如果W+R<=N，則不能保證強一致性

根據要求，我們可以調整W、R、N的值，以達到理想的一致性水平。

### 一致性模型

一致性模型是設計鍵值儲存時要考慮的另一個重要因素。 一致性模型定義了資料一致性的程度，並且存在多種可能的一致性模型：

* 強一致性：任何讀取操作都會返回一個與最新的寫資料項的結果相對應的值。用戶端永遠不會看到過期的資料
* 弱一致性：後續的讀取操作可能看不到最新的值。
* 最終一致性：這是弱一致性的一種特殊形式。只要有足夠的時間，所有的更新都會被傳播，而且所有的副本都是一致的。

強一致性通常是通過強迫一個副本不接受新的讀/寫，直到每個副本都同意當前的寫來實現的。這種方法對於高可用系統來說並不理想，因為它可能會阻塞新的操作。Dynamo和Cassandra採用最終一致性，這是我們推薦的鍵值儲存的一致性模型。

從並發寫入來看，最終一致性允許不一致的值進入系統，並強迫用戶端讀取這些值來進行調和。下一節將解釋調和是如何與版本管理一起工作的。

### 不一致的解決方法：版本控制

複製提供了高可用性，但會導致副本之間的不一致。 **版本控制**和**向量時鐘**用於解決不一致問題。版本化意味著將每一次資料修改都視為一個新的不可更改的資料版本。在我們談論版本控制之前，讓我們用一個例子來解釋不一致是如何發生的：

如圖6-7所示，副本節點n1和n2的值相同。 讓我們稱這個值為原始值。 server 1 和 server 2 透過 get(「name」) 操作取得相同的值。

![](images/chapter6/figure6-7.jpg)

接下來，server 1 將名稱更改為「johnSanFrancisco」，server 2 將名稱更改為「johnNewYork」，如圖 6-8 所示。 這兩個更改是同時執行的。 現在，我們有衝突的值，稱為版本 v1 和 v2。

![](images/chapter6/figure6-8.jpg)

在此範例中，可以忽略原始值，因為修改是基於它的。 但是，沒有明確的方法來解決最後兩個版本的衝突。 為了解決這個問題，我們需要一個可以檢測衝突並協調衝突的版本控制系統。

**向量時鐘**是解決此問題的常用技術。

讓我們來看看向量時鐘是如何工作的。

向量時鐘是與資料項關聯的鍵值 \[server, version] 對。 它可用於檢查一個版本是否先於、後於或與其他版本衝突。

假設一個向量時鐘用 D(\[S1, v1], \[S2, v2], ..., \[Sn, vn]) 表示，其中 D 是資料項，v1 是版本計數器，s1 是伺服器數字等。如果資料項 D 被寫入伺服器 Si，系統必須執行以下任務之一：

* 如果 \[Si, vi] 存在，則增加 vi。
* 否則，建立一個新的條目\[Si, 1]。

上面的抽象邏輯用一個具體的例子來解釋，如圖6-9所示：

![](images/chapter6/figure6-9.jpg)

1. 用戶端向系統寫入資料項 D1，寫入由伺服器 Sx 處理，伺服器現在具有向量時鐘 D1\[(Sx, 1)]。
2. 另一個用戶端讀取最新的 D1，將其更新為 D2，然後寫回。 D2 繼承自 D1，因此它會覆蓋 D1。 假設寫入由同一個伺服器 Sx 處理，該伺服器現在具有向量時鐘 D2(\[Sx, 2])。
3. 另一個用戶端讀取最新的 D2，將其更新為 D3，然後寫回。 假設寫操作由伺服器 Sy 處理，它現在有向量時鐘 D3(\[Sx, 2], \[Sy, 1]))。
4. 另一個用戶端讀取最新的 D2，將其更新為 D4，然後寫回。 假設寫入由伺服器 Sz 處理，它現在有 D4(\[Sx, 2], \[Sz, 1]))。
5. 當另一個用戶端讀取D3和D4時，發現衝突，這是由於資料項D2被Sy和Sz同時修改造成的。 衝突由用戶端解決，並將更新的資料發送到伺服器。 假設寫入由 Sx 處理，它現在有 D5(\[Sx, 3], \[Sy, 1], \[Sz, 1])。 我們將很快解釋如何檢測衝突。

使用向量時鐘，如果Y的向量時鐘中的每個參與者的版本計數器大於或等於版本X中的版本計數器，則很容易判斷版本X是版本Y的祖先（即無衝突）。例如，向量時鐘 D(\[s0, 1], \[s1, 1])] 是 D(\[s0, 1], \[s1, 2]) 的祖先。因此，未記錄任何衝突。

類似地，如果 Y 的向量時鐘中有任何參與者的計數器小於其在 X 中對應的計數器，則可以判斷版本 X 是 Y 的兄弟版本（即存在衝突）。例如，以下兩個 向量時鐘表示存在衝突：D(\[s0, 1], \[s1,2]) 和 D(\[s0, 2], \[s1, 1])

儘管向量時鐘可以解決衝突，但也有兩個明顯的缺點。 首先，向量時鐘增加了用戶端的複雜性，因為它需要實現衝突解決邏輯。

其次，向量時鐘中的 \[server: version] 對可能會快速增長。為了解決這個問題，我們為長度設定了一個閾值，如果超過了限制，則刪除最舊的對。這可能導致協調效率低下，因為後代關係無法準確確定。然而，基於Dynamo論文\[4]，亞馬遜在生產中還沒有遇到這個問題；因此，這可能是大多數公司可以接受的解決方案。

### 故障處理

與任何大規模的系統一樣，故障不僅是不可避免的，而且是常見的。處理故障情況是非常重要的。在本節中，我們首先介紹檢測故障的技術。然後，我們將介紹常見的故障解決策略。

*   故障檢測

    在分散式系統中，僅因為另一台伺服器這樣說就認為一台伺服器已宕機是不夠的。 通常，至少需要兩個獨立的資訊來源才能將伺服器標記為宕機。

    如圖 6-10 所示，all-to-all 多播是一種直接的解決方案。 但是，當系統中有很多伺服器時，這是低效的。

    ![](images/chapter6/figure6-10.jpg)

    一個更好的解決方案是使用分散的故障檢測方法，如`gossip`協議。`gossip`協議的工作原理如下：

    * 每個節點維護一個節點成員列表，其中包含成員ID和心跳計數器。
    * 每個節點定期增加它的心跳計數器
    * 每個節點定期向一組隨機節點發送心跳，然後再傳播到另一組節點上
    * 一旦節點收到心跳，成員名單就會更新到最新資訊。
    * 如果心跳沒有增加超過預定的時間，該成員被認為是離線的。

    ![](images/chapter6/figure6-11.jpg)

    如圖6-11所示：

    * 節點s0維護一個節點成員列表，如左側所示
    * 節點s0注意到節點s2（成員ID=2）的心跳計數器很長時間沒有增加。
    * 節點s0向一組隨機節點發送包括s2的資訊的心跳。一旦其他節點確認s2的心跳計數器長時間沒有更新，節點s2就會被標記下來，這個資訊會傳播給其他節點。
*   處理暫時性故障

    通過`gossip`協議檢測到故障後，系統需要部署一定的機制來確保可用性。 在嚴格的仲裁方法(`quorum`)中，讀取和寫入操作可以被阻塞，如仲裁共識部分所示。

    一種稱為「草率仲裁(`sloppy quorum`)」\[4] 的技術用於提高可用性。 系統不會強制執行法定人數要求，而是選擇前 W 個健康的伺服器進行寫入，並選擇前 R 個健康的伺服器進行哈希環上的讀取。 離線伺服器將被忽略。

    如果由於網路或伺服器故障導致伺服器不可用，將由另一台伺服器臨時處理請求，當宕機伺服器啟動時，更改將被推回以實現資料一致性。這個過程稱為**暗示交接（hinted handoff）**。由於圖6-12中s2不可用，讀寫暫時交由s3處理，當 s2 重新上線時，s3 會將資料交還給 s2。

    ![](images/chapter6/figure6-12.jpg)
*   處理永久性故障

    提示交接用於處理暫時故障。 如果副本永久不可用怎麼辦？

    為了處理這種情況，我們實施了一個**反熵協議（anti-entropy protocol）** 來保持副本同步。 反熵需要比較副本上的每條資料，並將每個副本更新為最新版本。

    Merkle樹用於檢測不一致，並盡量減少傳輸的資料量。

    引自維基百科 \[7]：「哈希樹或 Merkle 樹是一棵樹，其中每個非葉節點都標有其子節點的標籤或值（如果是葉子）的哈希值。 哈希樹允許對大型資料結構的內容進行高效和安全的驗證」。

    假設鍵空間是從 1 到 12，下面的步驟展示了如何構建 Merkle 樹，突出顯示的框表示不一致。

    * 第1步：將密鑰空間劃分為桶（在我們的例子中為4個），如圖6-13所示。 一個桶被用作根級節點，以保持樹的有限深度

    ![](images/chapter6/figure6-13.jpg)

    * 第2步：一旦建立了桶，使用統一的散列方法對桶中的每個密鑰進行散列（圖6-14）。

    ![](images/chapter6/figure6-14.jpg)

    * 第3步：為每個桶建立一個哈希節點（圖6-15）

    ![](images/chapter6/figure6-15.jpg)

    * 第4步：通過計算子代的哈希值，向上建立樹，直到根（圖6-16）。

    ![](images/chapter6/figure6-16.jpg)

    要比較兩個Merkle樹，首先要比較根哈希值。如果根哈希值匹配，則兩個伺服器有相同的資料。如果根哈希值不一致，那麼就比較左邊的子哈希值，然後是右邊的子哈希值。你可以遍歷該樹，找到哪些桶沒有被同步，並只同步這些桶。

    使用Merkle樹，需要同步的資料量與兩個副本之間的差異成正比，而不是它們包含的資料量。在現實世界的系統中，桶的大小是相當大的。例如，一個可能的設定是每十億個鍵有一百萬個桶，所以每個桶只包含1000個鍵。
*   處理資料中心的中斷故障

    資料中心的中斷可能是由於停電、網路中斷、自然災害等原因造成的。為了建立一個能夠處理資料中心中斷的系統，在多個資料中心之間**複製資料**是非常重要的。即使一個資料中心完全離線，使用者仍然可以通過其他資料中心存取資料。

架構的主要特點列舉如下：

* 用戶端通過簡單的API與鍵值儲存通訊：`get(key)` 和 `put(key, value)`。
* 協調器是一個節點，在用戶端和鍵值儲存之間充當代理。
* 節點採用一致性hash的散列方式分佈在一個環上。

* 不存在單點故障，因為每個節點都有相同的職責。

### 寫入路徑

圖 6-19 解釋了將寫請求定向到特定節點後會發生什麼。 請注意，建議的寫/讀路徑設計主要基於 Cassandra \[8] 的架構。

1. 寫入請求持久保存在提交日誌檔案中。
2. 資料保存在記憶體快取中。
    ![](images/chapter6/figure6-19.jpg)
3. 當記憶體快取已滿或達到預定義的閾值時，資料將刷新到磁碟上的 SSTable \[9]。 注意：排序字串表 (SSTable) 是 \<key, value> 對的排序列表。 有興趣進一步了解 SStable 的讀者，請參閱參考資料 \[9]。

### 讀取路徑

讀取請求被引導到一個特定的節點後，它首先檢查資料是否在記憶體快取中。如果是，資料就會被返回給用戶端，如圖6-20所示。

![](images/chapter6/figure6-20.jpg)

如果資料不在記憶體中，就會從磁碟中檢索出來。我們需要一個有效的方法來找出哪個SSTable中包含了該鍵。**布隆過濾器**\[10]通常被用來解決這個問題。

當資料不在記憶體中時，讀取路徑如圖6-21所示。

![](images/chapter6/figure6-21.jpg)

1. 系統首先檢查資料是否在記憶體中。如果沒有，就轉到第2步。
2. 如果資料不在記憶體中，系統會檢查Bloom過濾器。
3. Bloom過濾器被用來計算哪些SSTables可能包含密鑰。
4. SSTables會返回資料集的結果。
5. 資料集的結果被返回給用戶端。

### 總結

本章涵蓋了許多概念和技術。為了加深記憶，下表總結了分散式鍵值儲存的特點和相應的技術。

| 目標/問題              | 技術                                       |
| ------------------ | ---------------------------------------- |
| 儲存大資料的能力           | 使用一致性哈希將負載分散到多個伺服器上                      |
| 高可用性讀取             | 資料複製 多資料中心設定                             |
| 高可用性寫入             | 使用向量時鐘（vector clocks）進行版本控制和衝突解決         |
| 資料分區               | 一致性哈希                                    |
| 增量可擴展性             | 一致性哈希                                    |
| 異質性（heterogeneity） | 一致性哈希                                    |
| 處理暫時性故障            | 草率仲裁（sloppy quorum）和暗示交接（hinted handoff） |
| 處理永久性故障            | Merkle 樹                                 |
| 處理資料中心中斷           | 跨資料中心複製                                  |
