# 第11章：設計一個資訊推送系統


在本章中，您需要設計一個資訊推送系統。 什麼是資訊推送？ 根據 Facebook 說明頁面，「動態是位於首頁中間不斷更新的動態列表。動態包括您在 Facebook 上關注的使用者、公共主頁和小組發布的狀態更新、照片、影片、連結、應用程式事件和按讚。」\[1]。 這是一個流行的面試問題。 類似的常見問題有：設計 Facebook 資訊推送、Instagram 推送、Twitter 時間線等。

![](images/chapter11/figure11-1.jpg)

### 第1步：了解問題並確定設計範圍

第一組解釋問題是為了了解當面試官要求你設計一個資訊推送系統時，她的想法是什麼。最起碼，你應該弄清楚要支援哪些功能。下面是一個候選人與面試官互動的例子。

候選人：這是一個行動應用程式嗎？還是一個網路應用？或者兩者都是？

面試官：都是

候選人：哪些是重要的特徵？

面試官：使用者可以發布貼文，並在資訊流頁面上看到她朋友的貼文。

候選人：資訊是按倒序時間順序排序，還是按任何特定順序，如主題得分？例如，你的親密朋友的貼文有更高的分數。

面試官：為了簡單起見，讓我們假設推送是按倒序時間順序排序的。

候選人：一個使用者可以有多少個朋友？

面試官：5000

候選人：業務流量是多少？

面試官：1000萬DAU

候選人：推送可以包含圖片、影片，還是只有文字？

面試官：它可以包含媒體檔案，包括圖片和影片。

現在你已經收集了需求，我們把重點放在設計系統上。

### 第2步：提出高層次的設計方案並獲得認同

該設計分為兩個流程：資訊流發布和資訊流構建：

* 資訊發布（Feed publishing）：當使用者發布貼文時，相應的資料被寫入快取和資料庫。貼文被推送到她朋友的動態中。
* 資訊流構建（Newsfeed building）：為簡單起見，我們假設資訊推送是通過按時間倒序聚合朋友的貼文來構建的。

#### 資訊流發布 API

資訊流 API是客戶與伺服器通訊的主要方式。這些API是基於HTTP的，允許客戶執行操作，其中包括發布狀態、檢索資訊流、新增朋友等。

我們討論兩個最重要的API：資訊流發布 API 和資訊流檢索 API。

*   資訊流發布 API

    要發布一個貼文，將向伺服器發送一個HTTP POST請求。該API顯示如下。

    ```
    POST /v1/me/feed
    ```

    參數：

    * `content`：貼文內容的文字。
    * `auth_token`：它用於驗證API請求。
*   資訊流檢索 API

    檢索資訊流的API如下：

    ```
    GET /v1/me/feed
    ```

    參數：

    * `auth_token`：它用於驗證API請求。

#### 資訊發布（Feed publishing）

圖11-2顯示了發布流程的高層設計。

![](images/chapter11/figure11-2.jpg)

* **User（使用者）**：使用者可以在瀏覽器或行動應用程式上查看資訊流。一個使用者通過API發布內容為「你好」的貼文： `/v1/me/feed?content=Hello&auth_token={auth_token}`
* Load balancer（負載平衡器）：將流量分配給網路伺服器。
* Web servers（網路伺服器）：網路伺服器將流量重新導向到不同的內部服務。
* Post service（貼文服務）：在資料庫和快取中持久保存貼文。
* Fanout service（扇出服務）：推送新內容到朋友的資訊流。資訊流資料儲存在快取中，以便快速檢索。
* 通知服務：通知朋友有新內容，並發送推送通知。

在這一節中，我們將討論資訊流是如何在幕後構建的。

圖11-3顯示了高層設計。

* User（使用者）：一個使用者發送了一個請求來檢索她的資訊。 該請求看起來像這樣：`/v1/me/feed`。
* Load balancer（負載平衡器）：將流量分配給網路伺服器。
* Web servers（網路伺服器）：網路伺服器將請求路由到資訊發布服務。
* Newsfeed service（資訊饋送服務）：資訊饋送服務從快取中取得資訊。
* Newsfeed cache（資訊流快取）：儲存渲染資訊流所需的資訊ID。

### 第3步：深入設計

高層設計簡要地涵蓋了兩個流程：資訊發布和資訊流構建。在這裡，我們更深入地討論這些主題。

#### 資訊發布深入研究

圖 11-4 概述了資訊發布的詳細設計。我們已經討論了高層設計中的大部分元件，我們將重點關注兩個元件：Web 伺服器和扇出服務。

![](images/chapter11/figure11-4.jpg)

**web 服務**

除了與用戶端通訊外，Web 伺服器還執行身份驗證和速率限制。只有使用有效 auth\_token 登入的使用者才可以發文。該系統限制使用者在一定時期內可以發布的貼文數量，這對於防止垃圾郵件和濫用內容至關重要。

**扇出服務**

Fanout 是將貼文傳遞給所有朋友的過程。兩種類型的扇出模型是：**寫扇出（也稱為推模型）和讀扇出（也稱為拉模型）**。兩種模型各有利弊。我們解釋他們的工作流程並探索支援我們系統的最佳方法。

**寫扇出**

通過這種方法，資訊流在寫的時候就被預先計算了一個新的貼文在發布後會立即被送到朋友的快取中。

優點：

* 動態消息是即時生成的，可以第一時間推送給朋友。
* 取得資訊流的速度很快，因為資訊流是在寫的時候預先計算的。

缺點：

* 如果一個使用者有很多朋友，取得朋友列表並為所有朋友生成資訊流是很慢的，而且很耗時間。這被稱為熱鍵問題。
* 對於不活躍的使用者或那些很少登入的使用者，預先計算的資訊流會浪費計算資源。

**讀扇出**

資訊源是在閱讀時間內產生的。這是一個按需分配的模式。當使用者載入她的主頁時，最近的貼文被拉出。

優點：

* 對於不活躍的使用者或那些很少登入的使用者，讀取時的扇出效果更好，因為它不會在他們身上浪費計算資源。
* 資料不會被推送給朋友，所以不存在熱鍵的問題。

缺點：

* 取得資訊源的速度很慢，因為資訊源不是預先計算的。

我們採用了一種混合方法，以獲得兩種方法的好處並避免其中的缺點。由於快速取得資訊流是至關重要的，我們對大多數使用者使用推送模式。對於名人或有很多朋友/粉絲的使用者，我們讓粉絲按需提取資訊內容以避免系統過載。一致性哈希是緩解熱鍵問題的一個有用技術，因為它有助於更均勻地分配請求/資料。

讓我們仔細看看圖11-5中所示的扇出服務。

![](images/chapter11/figure11-5.jpg)

扇出服務的工作原理如下：

1. 從圖形資料庫中取得朋友 ID。 圖形資料庫適用於管理朋友關係和朋友推薦。 希望了解更多有關此概念的感興趣的讀者應參閱參考資料 \[2]。
2. 從使用者快取中取得朋友資訊。然後，系統根據使用者設定過濾出朋友。例如，如果你把某人調成靜音，她的貼文將不會顯示在你的資訊流中，儘管你們仍然是朋友。貼文可能不顯示的另一個原因是，使用者可以有選擇地與特定的朋友分享資訊或對其他人隱藏資訊。
3. 將好友列表和新貼文 ID 發送到訊息佇列。
4. Fanout worker 從訊息佇列中取得資料並將資訊流資料儲存在資訊流快取中。 你可以將資訊流快取視為一個 `<post_id, user_id>` 結構的映射表。 每當發布新貼文時，新貼文將被附加到資訊流表中，如圖 11-6 所示。 如果我們將整個使用者和貼文物件儲存在快取中，記憶體消耗會變得非常大。 因此，僅儲存 ID。 為了保持較小的記憶體大小，我們設定了一個可設定的限制。 使用者滾動瀏覽資訊流中數千個貼文的機會很小。 大多數使用者只對最新的內容感興趣，所以快取未命中率低。
5. 將 `<post_id, user_id>` 儲存在資訊流快取中。 圖 11-6 顯示了快取中資訊流的範例。

    ![](images/chapter11/figure11-6.jpg)

#### 資訊源檢索深入研究

圖 11-7 說明了資訊檢索的詳細設計。

![](images/chapter11/figure11-7.jpg)

如圖11-7所示，媒體內容（圖片、影片等）儲存在CDN中，便於快速檢索。 讓我們看看用戶端如何檢索資訊流。

1. 一個使用者發送了一個請求來檢索她的資訊流。該請求看起來像這樣：`/v1/me/feed`
2. 負載平衡將請求重新分配給網路伺服器。
3. Web 伺服器呼叫資訊流服務（News Feed Service）取得資訊流。
4. 資訊流服務從資訊流快取中獲得一個貼文ID列表。
5. 使用者的資訊流不僅僅是 feed ID 列表。 它包含使用者名稱、個人資料圖片、貼文內容、貼文圖片等。因此，資訊流服務從快取（使用者快取和貼文快取）中取得完整的使用者和貼文物件，以構建完全整合的資訊流。
6. 完全整合的資訊流以JSON格式返回到用戶端進行渲染。

*   **快取架構**

    快取對於資訊流系統非常重要。 我們將快取層分為 5 層，如圖 11-8 所示。

    ![](images/chapter11/figure11-8.jpg)

    * News Feed：它儲存了資訊的ID。
    * Content：它儲存每個貼文的資料。受歡迎的內容被儲存在熱快取中。
    * Social Graph：它儲存使用者關係資料。
    * Action：它儲存有關使用者是否喜歡貼文、回覆貼文或對貼文執行其他操作的資訊。
    * Counters：它儲存按讚、回覆、關注者、關注等的計數器。

### 第4步：總結

在本章中，我們設計了一個資訊推送系統。 我們的設計包含兩個流程：資訊發布和資訊檢索。

與任何系統設計面試問題一樣，沒有完美的系統設計方法。 每個公司都有其獨特的限制，您必須設計一個系統來適應這些限制。 了解您的設計和技術選擇的權衡很重要。 如果還剩幾分鐘，您可以討論可擴展性問題。 為避免重複討論，下面僅列出高層次的談話要點。

資料庫擴展：

* 垂直擴展 vs 水平擴展
* SQL vs NoSQL
* 主從複製
* 讀寫分離
* 一致性模型
* 資料庫分片

其他談話要點：

* 保持網路層的無狀態
* 盡可能多地快取資料
* 支援多個資料中心
* 使用訊息佇列降低耦合
* 監控關鍵指標。 例如，高峰時段的 QPS 和使用者刷新資訊流時的延遲值得監控。
