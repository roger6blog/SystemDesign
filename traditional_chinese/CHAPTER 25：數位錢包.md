# 數位錢包

支付平台通常為客戶提供數位錢包服務，這樣他們可以將錢存入錢包並在以后使用。例如，你可以從銀行卡向數位錢包充值，當你在線購買產品時，可以選擇使用錢包中的錢進行支付。

![Figure 12.1: 數位錢包](../images/v2/chapter12/Figure_12.1.png)

花錢並不是數位錢包提供的唯一功能。對於像 PayPal 這樣的支付平台，我們可以直接將錢轉移到同一支付平台上其他人的錢包中。與銀行間轉帳相比，數位錢包之間的直接轉帳速度更快，最重要的是，通常不會收取額外費用。

![Figure 12.2: 跨錢包餘額轉帳](../images/v2/chapter12/Figure_12.2.png)

假設我們被要求設計一個支援跨錢包餘額轉帳操作的數位錢包應用程式的後端。在面試開始時，我們會通過澄清問題來確定需求。

## 第一步 - 理解問題並確定設計範圍

**候選人**：我們是否只需要關注兩個數位錢包之間的餘額轉帳操作？是否需要考慮其他功能？

**面試官**：我們只關注餘額轉帳操作。

**候選人**：系統需要支援每秒多少筆交易（TPS）？

**面試官**：假設需要支援 1,000,000 TPS。

**候選人**：數位錢包對正確性有嚴格要求。我們可以假設事務性保證 [1] 足夠嗎？

**面試官**：聽起來不錯。

**候選人**：我們需要證明正確性嗎？

**面試官**：這是個好問題。正確性通常只能在交易完成後驗證。一種驗證方法是將我們的內部記錄與銀行的對帳單進行比較。對帳的局限性在於它只能顯示差異，無法說明差異是如何產生的。因此，我們希望設計一個具有可重現性的系統，這意味著我們始終可以通過從最開始重放資料來重建歷史餘額。

**候選人**：我們可以假設可用性要求為 99.99% 嗎？

**面試官**：聽起來不錯。

**候選人**：是否需要考慮外匯？

**面試官**：不需要，這超出了範圍。

### 總結

總結一下，我們的數位錢包需要支援以下內容：

- 支援兩個數位錢包之間的餘額轉帳操作。
- 支援 1,000,000 TPS。
- 可靠性至少為 99.99%。
- 支援交易。
- 支援可重現性。

### 粗略估算

當我們談論 TPS 時，意味著將使用交易性資料庫。如今，運行在典型資料中心節點上的關聯式資料庫可以支援每秒幾千筆交易。例如，參考文獻 [2] 包含一些流行的事務性資料庫伺服器的效能基準。假設一個資料庫節點可以支援 1,000 TPS。為了達到 100 萬 TPS，我們需要 1,000 個資料庫節點。

然而，這個計算略有偏差。每筆轉帳命令需要兩個操作：從一個帳戶扣款和向另一個帳戶存款。為了支援每秒 100 萬筆轉帳，系統實際上需要處理高達 200 萬 TPS，這意味著我們需要 2,000 個節點。

| 單節點 TPS | 節點數量 |
| ---------- | -------- |
| 100        | 20,000   |
| 1,000      | 2,000    |
| 10,000     | 200      |

**表 12.1: 單節點 TPS 與節點數量的映射**

## 第二步 - 提出高層設計並獲得認可

在本節中，我們將討論以下內容：

- API 設計
- 三種高層設計
  1. 簡單的記憶體解決方案
  2. 基於資料庫的分散式事務解決方案
  3. 具有可重現性的事件溯源解決方案

### API 設計

我們將使用 RESTful API 規範。對於這次面試，我們只需要支援一個 API：

| API                              | 詳情                             |
| -------------------------------- | -------------------------------- |
| POST /v1/wallet/balance_transfer | 將餘額從一個錢包轉移到另一個錢包 |

**請求參數如下：**

| 欄位           | 描述          | 類型                 |
| -------------- | ------------- | -------------------- |
| from_account   | 扣款帳戶      | string               |
| to_account     | 收款帳戶      | string               |
| amount         | 金額          | string               |
| currency       | 貨幣類型      | string (ISO 4217[3]) |
| transaction_id | 用於去重的 ID | uuid                 |

**範例回應體：**

```json
{
  "status": "success",
  "transaction_id": "81589980-2664-11ec-9621-0242ac130002"
}
```

值得一提的是，「amount」欄位的資料類型是「string」，而不是「double」。我們在第 11 章支付系統中解釋了原因（[第 320 頁](./CHAPTER 11：Payment System.md#APIs-for-payment-service)）。

在實踐中，許多人仍然選擇浮點數或雙精度數表示，因為它幾乎被所有程式設計語言和資料庫支援。只要我們了解精度丟失的潛在風險，這是一個合適的選擇。

### 記憶體分片解決方案

錢包應用程式為每個使用者帳戶維護一個帳戶餘額。表示這種 <使用者, 餘額> 關係的一個良好資料結構是映射（map），也稱為哈希表（hash table）或鍵值儲存（key-value store）。

對於記憶體儲存，一個流行的選擇是 Redis。一個 Redis 節點不足以處理 100 萬 TPS。我們需要建立一個 Redis 節點叢集，並將使用者帳戶均勻分布在其中。這個過程稱為分區或分片。

為了將鍵值資料分布在 n 個分區中，我們可以計算鍵的哈希值並將其除以 n。餘數就是分區的目標。以下偽程式碼展示了分片過程：

```java
String accountID = "A";
Int partitionNumber = 7;
Int myPartition = accountID.hashCode() % partitionNumber;
```

分區數量和所有 Redis 節點的位址可以儲存在一個集中的地方。我們可以使用 ZooKeeper [4] 作為高可用性配置儲存解決方案。

該解決方案的最後一個元件是處理轉帳命令的服務。我們稱之為錢包服務，它有幾個關鍵職責。

1. 接收轉帳命令
2. 驗證轉帳命令
3. 如果命令有效，它會更新參與轉帳的兩個使用者的帳戶餘額。在叢集中，帳戶餘額可能位於不同的 Redis 節點中

錢包服務是無狀態的。它很容易水平擴展。圖 12.3 展示了記憶體解決方案。

![Figure 12.3: 記憶體解決方案](../images/v2/chapter12/Figure_12.3.png)

在這個例子中，我們有 3 個 Redis 節點。有三個用戶端 A、B 和 C。他們的帳戶餘額均勻分布在這三個 Redis 節點中。這個例子中有兩個錢包服務節點處理餘額轉帳請求。當其中一個錢包服務節點收到將 1 美元從用戶端 A 轉移到用戶端 B 的轉帳命令時，它會向兩個 Redis 節點發出兩個命令。對於包含用戶端 A 帳戶的 Redis 節點，錢包服務從帳戶中扣除 1 美元。對於用戶端 B，錢包服務向帳戶中添加 1 美元。

**候選人**：在這個設計中，帳戶餘額分布在多個 Redis 節點中。ZooKeeper 用於維護分片資訊。無狀態的錢包服務使用分片資訊定位用戶端的 Redis 節點並相應地更新帳戶餘額。

**面試官**：這個設計可行，但它不符合我們的正確性要求。錢包服務為每筆轉帳更新兩個 Redis 節點。無法保證兩個更新都會成功。例如，如果錢包服務節點在第一次更新完成後但在第二次更新完成之前崩潰，將導致轉帳不完整。這兩個更新需要在一個原子交易中完成。

### 分散式事務

#### 資料庫分片

我們如何使對兩個不同儲存節點的更新具有原子性？第一步是將每個 Redis 節點替換為交易性關聯式資料庫節點。圖 12.4 展示了這種架構。這次，用戶端 A、B 和 C 被分區到 3 個關聯式資料庫中，而不是 3 個 Redis 節點中。

![Figure 12.4: 關聯式資料庫](../images/v2/chapter12/Figure_12.4.png)

使用交易性資料庫只能解決部分問題。正如上一節提到的，一個轉帳命令很可能需要更新兩個不同資料庫中的兩個帳戶。無法保證兩個更新操作會同時處理。如果錢包服務在更新第一個帳戶餘額後立即重啟，我們如何確保第二個帳戶也會被更新？

### 分散式事務：兩階段提交

在分散式系統中，一個交易可能涉及多個節點上的多個進程。為了使交易具有原子性，分散式事務可能是答案。有兩種實現分散式事務的方式：低級別解決方案和高階解決方案。我們將分別探討它們。

低級別解決方案依賴於資料庫本身。最常用的演算法稱為兩階段提交（2PC）。顧名思義，它有兩個階段，如圖 12.5 所示。

![Figure 12.5: 兩階段提交（來源 [5]）](../images/v2/chapter12/Figure_12.5.png)

1. 協調者（在我們的例子中是錢包服務）像往常一樣在多個資料庫上執行讀寫操作。如圖 12.5 所示，資料庫 A 和 C 都被鎖定。
2. 當應用程式準備提交交易時，協調者會要求所有資料庫準備交易。
3. 在第二階段，協調者收集所有資料庫的回覆並執行以下操作：
   (a) 如果所有資料庫都回覆「是」，協調者會要求所有資料庫提交它們收到的交易。
   (b) 如果有任何一個資料庫回覆「否」，協調者會要求所有資料庫中止交易。

這是一個低級別解決方案，因為準備步驟需要對資料庫交易進行特殊修改。例如，有一個 X/Open XA [6] 標準，用於協調異構資料庫以實現 2PC。2PC 的最大問題是效能不佳，因為在等待其他節點的消息時，鎖可能會被長時間持有。2PC 的另一個問題是協調者可能成為單點故障，如圖 12.6 所示。

![Figure 12.6: 協調者崩潰](../images/v2/chapter12/Figure_12.6.png)

### 分散式事務：嘗試-確認/取消（TC/C）

TC/C 是一種補償性事務 [7]，分為兩個步驟：

1. 在第一階段，協調者要求所有資料庫為交易預留資源。
2. 在第二階段，協調者收集所有資料庫的回覆：
   (a) 如果所有資料庫都回覆「是」，協調者要求所有資料庫確認操作，即嘗試-確認過程。
   (b) 如果有任何一個資料庫回覆「否」，協調者要求所有資料庫取消操作，即嘗試-取消過程。

需要注意的是，2PC 的兩個階段被包裝在同一個交易中，而在 TC/C 中，每個階段都是一個獨立的事務。

#### TC/C 範例

通過一個實際例子更容易解釋 TC/C 的工作原理。假設我們想從帳戶 A 轉帳 1 美元到帳戶 C。表 12.2 總結了 TC/C 在每個階段的執行情況。

| 階段 | 操作 | 帳戶 A 的餘額變化 | 帳戶 C 的餘額變化 |
| ---- | ---- | ----------------- | ----------------- |
| 1    | 嘗試 | -$1               | 無操作            |
| 2    | 確認 | 無操作            | +$1               |
|      | 取消 | +$1               | 無操作            |

**表 12.2: TC/C 範例**

假設錢包服務是 TC/C 的協調者。在分散式事務開始時，帳戶 A 的餘額為 1 美元，帳戶 C 的餘額為 0 美元。

##### 第一階段：嘗試

在嘗試階段，錢包服務（作為協調者）向兩個資料庫發送兩個交易命令：

1. 對於包含帳戶 A 的資料庫，協調者啟動一個本地交易，將帳戶 A 的餘額減少 1 美元。
2. 對於包含帳戶 C 的資料庫，協調者發送一個無操作（NOP）命令。資料庫對 NOP 命令不執行任何操作，並始終向協調者回覆成功消息。

嘗試階段如圖 12.7 所示。粗線表示交易持有的鎖。

![Figure 12.7: 嘗試階段](../images/v2/chapter12/Figure_12.7.png)

##### 第二階段：確認

如果兩個資料庫都回覆「是」，錢包服務進入確認階段。

帳戶 A 的餘額已在第一階段更新，錢包服務無需在此階段更改其餘額。然而，帳戶 C 在第一階段尚未收到來自帳戶 A 的 1 美元。在確認階段，錢包服務需要將 1 美元添加到帳戶 C 的餘額中。

確認過程如圖 12.8 所示。

![Figure 12.8: 確認階段](../images/v2/chapter12/Figure_12.8.png)

##### 第二階段：取消

如果嘗試階段失敗怎么辦？在上面的例子中，我們假設帳戶 C 的 NOP 操作始終成功，但在實踐中它可能會失敗。例如，帳戶 C 可能是一個非法帳戶，監管機構禁止資金流入或流出該帳戶。在這種情況下，分散式事務必須取消，我們需要清理。

由於帳戶 A 的餘額已在嘗試階段的交易中更新，錢包服務無法取消已完成的事務。它所能做的是啟動另一個事務，以撤銷嘗試階段事務的效果，即將 1 美元加回帳戶 A。

由於帳戶 C 在嘗試階段未被更新，錢包服務只需向帳戶 C 的資料庫發送一個 NOP 操作。

取消過程如圖 12.9 所示。

![Figure 12.9: 取消階段](../images/v2/chapter12/Figure_12.9.png)

### 2PC 與 TC/C 的比較

表 12.3 顯示 2PC 和 TC/C 有許多相似之處，但也存在差異。在 2PC 中，當第二階段開始時，所有本地事務都未完成（仍被鎖定），而在 TC/C 中，所有本地事務都已完成（解鎖）。換句話說，2PC 的第二階段是關於完成未完成的事務（如中止或提交），而在 TC/C 中，第二階段是關於在發生錯誤時使用反向操作來抵銷之前的事務結果。下表總結了它們的差異。

|      | 第一階段           | 第二階段：成功   | 第二階段：失敗         |
| ---- | ------------------ | ---------------- | ---------------------- |
| 2PC  | 本地事務未完成     | 提交所有本地事務 | 取消所有本地事務       |
| TC/C | 所有本地事務已完成 | 執行新的本地事務 | 撤銷已提交事務的副作用 |

**表 12.3: 2PC 與 TC/C 的比較**

TC/C 也被稱為補償性分散式事務。它是一種高階解決方案，因為補償（也稱為「撤銷」）是在業務邏輯中實現的。這種方法的優點是它與資料庫無關。只要資料庫支援交易，TC/C 就可以工作。缺點是必須在應用層的業務邏輯中管理分散式事務的細節並處理其複雜性。

#### 階段狀態表

我們還沒有回答之前提出的問題：如果錢包服務在 TC/C 中間重啟怎麼辦？當它重啟時，所有之前的操作歷史可能會丟失，系統可能不知道如何恢復。

解決方案很簡單。我們可以將 TC/C 的進度作為階段狀態儲存在交易性資料庫中。階段狀態至少包括以下資訊：

- 分散式事務的 ID 和內容。
- 每個資料庫的嘗試階段狀態。狀態可能是「未發送」、「已發送」和「已收到回覆」。
- 第二階段名稱。可能是「確認」或「取消」。可以使用嘗試階段的結果計算。
- 第二階段的狀態。
- 一個亂序標誌（稍後在「亂序執行」部分解釋）。

階段狀態表應該放在哪裡？通常，我們將階段狀態儲存在包含扣款錢包帳戶的資料庫中。更新後的架構圖如圖 12.10 所示。

![Figure 12.10: 階段狀態表](../images/v2/chapter12/Figure_12.10.png)

#### 不平衡狀態

不知你是否注意到，在嘗試階段結束時，1 美元消失了（圖 12.11）？

假設一切順利，在嘗試階段結束時，1 美元從帳戶 A 中扣除，這比 TC/C 開始時少了。這違反了會計的基本原則，即交易後總和應保持不變。

好消息是，TC/C 仍然保持了事務性保證。TC/C 由多個獨立的本地事務組成。由於 TC/C 是由應用程式驅動的，應用程式本身能夠看到這些本地事務之間的中間結果。另一方面，資料庫交易或 2PC 版本的分散式事務由資料庫維護，對高級應用程式是不可見的。

在分散式事務執行期間，總會出現資料不一致的情況。這些不一致可能對我們來說是透明的，因為資料庫等低級系統已經修復了這些不一致。如果沒有，我們必須自己處理（例如 TC/C）。

#### 有效操作順序

嘗試階段有三種選擇：

| 嘗試階段選擇 | 帳戶 A | 帳戶 C |
| ------------ | ------ | ------ |
| 選擇 1       | -$1    | 無操作 |
| 選擇 2       | NOP    | +$1    |
| 選擇 3       | -$1    | +$1    |

**表 12.4: 嘗試階段選擇**

這三種選擇看起來都合理，但有些是無效的。

對於選擇 2，如果帳戶 C 的嘗試階段成功，但帳戶 A 的嘗試階段失敗（NOP)，錢包服務需要進入取消階段。有可能其他人會介入並將 1 美元從帳戶 C 中移走。當錢包服務嘗試從帳戶 C 中扣除 1 美元時，發現沒有剩餘資金，這違反了分散式事務的事務性保證。

對於選擇 3，如果 1 美元從帳戶 A 中扣除並同時添加到帳戶 C 中，會引入很多複雜性。例如，1 美元被添加到帳戶 C，但未能從帳戶 A 中扣除。在這種情況下，我們該怎麼辦？

因此，選擇 2 和選擇 3 是有缺陷的選擇，只有選擇 1 是有效的。

### 分散式事務：Saga

#### 線性順序執行

另一種流行的分散式事務解決方案稱為 Saga [8]。Saga 是微服務架構中的事實標準。Saga 的理念很簡單：

1. 所有操作按順序排列。每個操作都是其自身資料庫上的獨立事務。
2. 操作從第一個到最後一個依次執行。當一個操作完成後，觸發下一個操作。
3. 當某個操作失敗時，整個過程從目前操作開始，按相反順序回滾到第一個操作，使用補償事務。因此，如果一個分散式事務有 n 個操作，我們需要準備 2n 個操作：n 個操作用於正常情況，另外 n 個操作用於回滾期間的補償事務。

通過一個例子更容易理解這一點。圖 12.13 展示了從帳戶 A 轉帳 1 美元到帳戶 C 的 Saga 工作流程。頂部水平線顯示正常執行順序。兩條垂直線顯示系統在遇到錯誤時應執行的操作。當遇到錯誤時，轉帳操作會回滾，用戶端會收到錯誤消息。正如我們在第 352 頁的「有效操作順序」部分提到的，我們必須將扣款操作放在加款操作之前。

![Figure 12.13: Saga 工作流程](../images/v2/chapter12/Figure_12.13.png)

我們如何協調這些操作？有兩種方法：

1. **編排（Choreography）**。在微服務架構中，所有參與 Saga 分散式事務的服務通過訂閱其他服務的事件來完成自己的工作。因此，這是一種完全去中心化的協調方式。
2. **編排（Orchestration）**。一個單一的協調者指示所有服務按正確順序完成工作。

選擇哪種協調模型取決於業務需求和目標。編排解決方案的挑戰在於服務以完全非同步的方式通訊，因此每個服務必須維護一個內部狀態機，以便在其他服務發出事件時知道該做什麼。當有許多服務時，管理起來可能會變得困難。編排解決方案很好地處理了複雜性，因此它通常是數位錢包系統中的首選解決方案。

1. 如果沒有延遲要求，或者服務非常少（例如我們的轉帳範例），我們可以選擇其中任何一種。如果我們想順應微服務架構的趨勢，選擇 Saga。
2. 如果系統對延遲敏感並且包含許多服務/操作，TC/C 可能是更好的選擇。

### 事件溯源（Event Sourcing）

#### 背景

在現實生活中，數位錢包提供商可能會被審計。這些外部審計員可能會提出一些具有挑戰性的問題，例如：

1. 我們是否知道任何時間點的帳戶餘額？
2. 我們如何知道歷史和目前帳戶餘額是正確的？
3. 我們如何證明在代碼更改後系統邏輯是正確的？

一種系統回答這些問題的設計理念是事件溯源，這是一種在領域驅動設計（DDD）[9] 中開發的技術。

#### 定義

事件溯源中有四個重要術語：

1. **命令（Command）**
2. **事件（Event）**
3. **狀態（State）**
4. **狀態機（State Machine）**

##### 命令

命令是來自外部世界的預期操作。例如，如果我們想從用戶端 A 轉帳 1 美元到用戶端 C，這個轉帳請求就是一個命令。

在事件溯源中，一切都有順序非常重要。因此，命令通常被放入一個 FIFO（先進先出）佇列中。

##### 事件

命令是一種意圖，而不是事實，因為某些命令可能無效且無法執行。例如，如果轉帳後帳戶餘額變為負數，轉帳操作將失敗。

在執行任何操作之前，必須驗證命令。一旦命令通過驗證，它就是有效的並且必須執行執行的結果稱為事件。

命令和事件之間有兩個主要區別：

1. 事件必須被執行，因為它們代表已驗證的事實。在實踐中，我們通常使用過去時態來描述事件。如果命令是「從 A 轉帳 1 美元到 C」，則相應的事件是「從 A 轉帳了 1 美元到 C」。
2. 命令可能包含隨機性或 I/O，但事件必須是確定性的。事件代表歷史事實。

事件生成過程有兩個重要屬性：

1. 一個命令可能生成任意數量的事件。它可能生成零個或多個事件。
2. 事件生成可能包含隨機性，這意味著不能保證一個命令總是生成相同的事件。事件生成可能包含外部 I/O 或隨機數。我們將在本章末尾更詳細地重新討論這一屬性。

事件的順序必須遵循命令的順序。因此，事件也儲存在 FIFO 佇列中。

##### 狀態

狀態是在應用事件時將更改的內容。在錢包系統中，狀態是帳戶名稱或 ID，值是帳戶餘額。狀態可以看作是一個鍵值儲存，其中鍵是主鍵，值是表行。

##### 狀態機

狀態機驅動事件溯源過程。它有兩個主要功能：

1. 驗證命令並生成事件。
2. 應用事件以更新狀態。

事件溯源要求狀態機的行為是確定性的。因此，狀態機本身不應包含任何隨機性。例如，它不應使用 I/O 從外部讀取任何隨機內容，或使用任何隨機數。當它將事件應用於狀態時，它應始終生成相同的結果。

圖 12.14 展示了事件溯源架構的靜態視圖。狀態機負責將命令轉換為事件並應用事件。由於狀態機有兩個主要功能，我們通常繪製兩個狀態機，一個用於驗證命令，另一個用於應用事件。

![Figure 12.14: 事件溯源的靜態視圖](../images/v2/chapter12/Figure_12.14.png)

對於錢包服務，命令是餘額轉帳請求。這些命令被放入 FIFO 佇列中。命令佇列的一個流行選擇是 Kafka [10]。命令佇列如圖 12.16 所示。

![Figure 12.16: 命令佇列](../images/v2/chapter12/Figure_12.16.png)

假設狀態（帳戶餘額）儲存在關聯式資料庫中。狀態機按 FIFO 順序逐個檢查每個命令。對於每個命令，它檢查帳戶是否有足夠的餘額。如果有，狀態機為每個帳戶生成一個事件。例如，如果命令是「A→$1→C」，狀態機生成兩個事件：「A:-$1」和「C:+$1」。

3. 驗證命令。如果有效，為每個帳戶生成兩個事件。
4. 讀取下一個事件。
5. 通過更新資料庫中的餘額來應用事件。

#### 可重現性

事件溯源相對於其他架構的最大優勢是可重現性。

在前面提到的分散式事務解決方案中，錢包服務將更新後的帳戶餘額（狀態）保存到資料庫中。很難知道帳戶餘額為何發生變化。同時，歷史餘額資訊在更新操作期間丟失。在事件溯源設計中，所有更改首先作為不可變的歷史記錄保存。資料庫僅用作任何給定時間點餘額的更新視圖。

我們始終可以通過從最開始重放事件來重建歷史餘額狀態。由於事件列表是不可變的，並且狀態機邏輯是確定性的，因此可以保證每次重放生成的歷史狀態是相同的。

圖 12.18 展示了如何通過重放事件來重現錢包服務的狀態。

![Figure 12.18: 重現狀態](../images/v2/chapter12/Figure_12.18.png)

可重現性幫助我們回答本節開頭審計員提出的難題。我們在此重複這些問題：

1. 我們是否知道任何時間點的帳戶餘額？
2. 我們如何知道歷史和目前帳戶餘額是正確的？
3. 我們如何證明在代碼更改後系統邏輯是正確的？

對於第一個問題，我們可以通過從開始重放事件到我們想知道帳戶餘額的時間點來回答。

對於第二個問題，我們可以通過從事件列表中重新計算帳戶餘額來驗證其正確性。

對於第三個問題，我們可以針對事件運行不同版本的代碼，並驗證它們的結果是否相同。

由於審計能力，事件溯源通常被選為錢包服務的事實解決方案。

### 命令查詢責任分離（CQRS）

到目前為止，我們已經設計了錢包服務以高效地將資金從一個帳戶轉移到另一個帳戶。然而，用戶端仍然不知道帳戶餘額是多少。需要有一種方法來發布狀態（餘額資訊），以便事件溯源架構外部的用戶端可以知道狀態是什麼。

直觀地說，我們可以建立資料庫（歷史狀態）的只讀副本並與外部世界共用。事件溯源以一种略有不同的方式回答這個問題：與其發布狀態（餘額資訊），不如發布所有事件。這種理念稱為 CQRS [11]。

在 CQRS 中，有一個狀態機負責狀態的寫入部分，但有多個只讀狀態機負責生成狀態的視圖。這些視圖可用於查詢。

這些只讀狀態機可以從事件佇列中派生出不同的狀態表示。例如，用戶端可能想知道他們的餘額，一個只讀狀態機可以將狀態保存在資料庫中以服務餘額查詢。另一個狀態機可以為特定時間段構建狀態，以幫助調查可能的重複扣款等問題。狀態資訊是一個審計追蹤，可以幫助對帳財務記錄。

只讀狀態機在一定程度上滯後，但最終會趕上。架構設計是最終一致的。

### 高效能事件溯源

在之前的例子中，我們使用 Kafka 作為命令和事件儲存，使用資料庫作為狀態儲存。現在讓我們探討一些優化方法。

### 基於檔案的命令和事件列表

第一個優化是將命令和事件保存到本地磁碟，而不是像 Kafka 這樣的遠端儲存。這樣可以避免網路傳輸時間。事件列表使用僅追加的資料結構。追加是一種順序寫入操作，通常非常快。即使對於機械硬碟，它也表現良好，因為作業系統對順序讀寫進行了高度優化。根據這篇文章 [12]，在某些情況下，順序磁碟存取可能比隨機記憶體存取更快。

第二個優化是將最近的命令和事件緩存在記憶體中。正如我們之前解釋的那樣，我們在命令和事件持久化後立即處理它們。我們可以將它們緩存在記憶體中，以節省從本地磁碟載入的時間。

我們將探討一些實現細節。一種稱為 mmap [13] 的技術非常適合實現上述優化。Mmap 可以同時寫入本地磁碟並將最近的內容緩存在記憶體中。它將磁碟檔案映射為記憶體中的陣列。作業系統將檔案的某些部分緩存在記憶體中，以加速讀寫操作。對於僅追加的檔案操作，幾乎可以保證所有資料都保存在記憶體中，這非常快。

### 快照

一旦所有內容都基於檔案，我們考慮如何加速可重現性過程。當我們第一次介紹可重現性時，狀態機每次都必須從頭開始處理事件。我們可以優化的方法是定期停止狀態機並將目前狀態保存到檔案中。這稱為快照。

快照是歷史狀態的不可變視圖。一旦保存了快照，狀態機就不必再從頭開始了。它可以從快照中讀取資料，驗證上次停止的位置，並從那裡恢復處理。

對於錢包服務等金融應用，財務團隊通常要求在 00:00 拍攝快照，以便他們可以驗證當天發生的所有交易。當我們第一次介紹事件溯源的 CQRS 時，解決方案是設置一個只讀狀態機，從頭開始讀取，直到達到指定時間。使用快照後，只讀狀態機只需載入包含資料的快照。

快照是一個巨大的二進位檔案，常見的解決方案是將其保存在物件儲存解決方案中，例如 HDFS [16]。

圖 12.22 展示了基於檔案的事件溯源架構。當所有內容都基於檔案時，系統可以充分利用電腦硬體的最大 I/O 吞吐量。

![](../images/v2/chapter12/Figure_12.22.png)

**候選人**：我們可以重構事件溯源的設計，使命令列表、事件列表、狀態和快照都保存在檔案中。事件溯源架構以線性方式處理事件列表，這與硬碟和作業系統快取的設計非常契合。

**面試官**：本地基於檔案的解決方案的效能優於需要從遠端 Kafka 和資料庫存取資料的系統。然而，還有一個問題：由於資料保存在本地磁碟上，伺服器現在是有狀態的，並成為單點故障。我們如何提高系統的可靠性？

### 可靠的高效能事件溯源

在解釋解決方案之前，讓我們先分析系統中需要可靠性保證的部分。

#### 可靠性分析

從概念上講，節點所做的一切都圍繞兩個概念：資料和計算。只要資料是持久的，就可以通過在另一個節點上運行相同的代碼來恢復計算結果。這意味著我們只需要擔心資料的可靠性，因為如果資料丟失，它將永遠丟失。系統的可靠性主要取決於資料的可靠性。

我們的系統中有四種類型的資料：

1. 基於檔案的命令
2. 基於檔案的事件
3. 基於檔案的狀態
4. 狀態快照

我們來仔細看看如何確保每種類型資料的可靠性。

狀態和快照總是可以通過重放事件列表來重新生成。為了提高可靠性。

現在我們來檢查命令。表面上看，事件是從命令生成的。我們可能認為為命令提供強可靠性保證就足夠了。乍一看這似乎是正確的，但它忽略了一些重要的東西。事件生成不一定是確定性的，它可能包含隨機因素，例如隨機數、外部 I/O 等。因此，命令不能保證事件的可重現性。

現在是時候仔細看看事件了。事件代表引入狀態（帳戶餘額）變化的歷史事實。事件是不可變的，可以用來重建狀態。

通過這個分析，我們得出結論：事件資料是唯一需要高可靠性保證的資料。我們將在下一節中解釋如何實現這一點。

### 共識

為了提供高可靠性，我們需要在多個節點之間複製事件列表。在複製過程中，我們必須保證以下屬性：

1. 無資料丟失。
2. 日誌檔案中的資料相對順序在節點之間保持一致。

為了實現這些保證，基於共識的複製是一個很好的選擇。共識演算法確保多個節點就事件列表的內容達成一致。讓我們以 Raft [17] 共識演算法為例。

Raft 演算法保證只要超過半數的節點線上，它們上的僅追加列表就具有相同的資料。例如，如果我們有 5 個節點並使用 Raft 演算法同步它們的資料，只要至少 3 個（超過 1/2）節點線上，如圖 12.23 所示，系統仍然可以正常工作：

![Figure 12.23: Raft](../images/v2/chapter12/Figure_12.23.png)

在 Raft 演算法中，一個節點可以扮演三種不同的角色：

1. **領導者（Leader）**
2. **候選者（Candidate）**
3. **跟隨者（Follower）**

我們可以在 Raft 論文中找到 Raft 演算法的實現。我們在這裡只涵蓋高層次的概念，不深入細節。在 Raft 中，最多有一個節點是叢集的領導者，其餘節點是跟隨者。領導者負責接收外部命令並在叢集中的節點之間可靠地複製資料。

使用 Raft 演算法，只要大多數節點線上，系統就是可靠的。例如，如果有 3 個節點，系統可以容忍 1 個節點的故障；如果有 5 個節點，系統可以容忍 2 個節點的故障。

### 可靠解決方案

通過複製，我們的基於檔案的事件溯源架構中不會有單點故障。讓我們看看實現細節。圖 12.24 展示了具有可靠性保證的事件溯源架構。

![Figure 12.24: Raft 節點組](../images/v2/chapter12/Figure_12.24.png)

在圖 12.24 中，我們設置了 3 個事件溯源節點。這些節點使用 Raft 演算法可靠地同步事件列表。

領導者從外部使用者接收傳入的命令請求，將它們轉換為事件，並將事件追加到本地事件列表中。Raft 演算法將新添加的事件複製到跟隨者。

所有節點（包括跟隨者）處理事件列表並更新狀態。Raft 演算法確保領導者和跟隨者具有相同的事件列表，而事件溯源保證只要事件列表相同，所有狀態都相同。

一個可靠的系統需要優雅地處理故障，因此讓我們探討如何處理節點崩潰。

如果領導者崩潰，Raft 演算法會自動從剩餘的健康節點中選擇一個新的領導者。這個新選舉的領導者負責接受外部使用者的命令。保證當節點宕機時，叢集整體仍能提供服務。

當領導者崩潰時，崩潰可能發生在命令列表轉換為事件之前。在這種情況下，用戶端會通過超時或收到錯誤回應注意到問題。用戶端需要將相同的命令重新發送給新選舉的領導者。

相比之下，跟隨者崩潰的處理要簡單得多。如果跟隨者崩潰，發送給它的請求將失敗。Raft 通過無限重試來處理故障，直到崩潰的節點重新啟動或被新節點替換。

**候選人**：在這個設計中，我們使用 Raft 共識演算法在多個節點之間複製事件列表。領導者接收命令並將事件複製到其他節點。

### 分散式事件溯源

在上一節中，我們解釋了如何實現可靠的高效能事件溯源架構。它解決了可靠性問題，但有兩個限制：

1. 當數位錢包更新時，我們希望立即收到更新結果。但在 CQRS 設計中，請求/回應流程可能很慢。這是因為用戶端不知道數位錢包何時更新，用戶端可能需要依賴定期輪詢。
2. 單個 Raft 組的容量有限。在某個規模下，我們需要對資料進行分片並實現分散式事務。

讓我們看看如何解決這兩個問題。

### 拉取與推送

在拉取模型中，外部使用者定期從只讀狀態機輪詢執行狀態。這種模型不是即時的，如果輪詢頻率設定過高，可能會使錢包服務過載。圖 12.25 展示了拉取模型。

![Figure 12.25: 定期拉取](../images/v2/chapter12/Figure_12.25.png)

通過在外部使用者和事件溯源節點之間添加反向代理 [18]，可以改進簡單的拉取模型。在這種設計中，外部使用者發送命令，反向代理定期輪詢執行狀態。這種設計簡化了用戶端邏輯，但通訊仍然不是即時的。

圖 12.26 展示了添加反向代理的拉取模型。

![Figure 12.26: 帶反向代理的拉取模型](../images/v2/chapter12/Figure_12.26.png)

一旦我們有了反向代理，我們可以通過修改只讀狀態機來加快回應速度。正如我們之前提到的，只讀狀態機可以有他自己的行為。例如，一種行為可以是一直讀狀態機在收到事件後立即將執行狀態推回反向代理。這將給使用者一種即時回應的感覺。

### 分散式事務

我們可以重用分散式事務解決方案，TC/C 或 Saga。假設我們通過將鍵的哈希值除以 2 來對資料進行分區。

圖 12.28 展示了更新後的設計。

![Figure 12.28: 最終設計](../images/v2/chapter12/Figure_12.28.png)

讓我們看看在最終的分散式事件溯源架構中，轉帳操作是如何工作的。為了更容易理解，我們使用 Saga 分散式事務模型，並且只解釋沒有回滾的成功路徑。

轉帳操作包含 2 個分散式操作：A:-$1 和 C:+$1。Saga 協調者協調執行，如圖 12.29 所示：

1. 使用者 A 向 Saga 協調者發送一個分散式事務。它包含兩個操作：A:-$1 和 C:+$1。
2. Saga 協調者在階段狀態表中建立一條記錄以追蹤事務的狀態。
3. Saga 協調者檢查操作順序並確定需要先處理 A:-$1。協調者將 A:-$1 作為命令發送到包含帳戶 A 資訊的分區 1。
4. 分區 1 的 Raft 領導者接收 A:-$1 命令並將其儲存在命令列表中。然後驗證命令。如果有效，則將其轉換為事件。Raft 共識演算法用於在不同節點之間同步資料。事件（從 A 的帳戶餘額中扣除 1 美元）在同步完成後執行。
5. 事件同步後，分區 1 的事件溯源框架使用 CQRS 將資料同步到讀取路徑。讀取路徑重建狀態和執行狀態。
6. 分區 1 的讀取路徑將狀態推回事件溯源框架的調用者，即 Saga 協調者。
7. Saga 協調者從分區 1 接收到成功狀態。
8. Saga 協調者在階段狀態表中建立一條記錄，指示分區 1 中的操作成功。
9. 由於第一個操作成功，Saga 協調者執行第二個操作，即 C:+$1。協調者將 C:+$1 作為命令發送到包含帳戶 C 資訊的分區 2。
10. 分區 2 的 Raft 領導者接收 C:+$1 命令並將其保存到命令列表中。如果有效，則將其轉換為事件。Raft 共識演算法用於在不同節點之間同步資料。事件（向 C 的帳戶添加 1 美元）在同步完成後執行。
11. 事件同步後，分區 2 的事件溯源框架使用 CQRS 將資料同步到讀取路徑。讀取路徑重建狀態和執行狀態。
12. 分區 2 的讀取路徑將狀態推回事件溯源框架的調用者，即 Saga 協調者。
13. Saga 協調者從分區 2 接收到成功狀態。
14. Saga 協調者在階段狀態表中建立一條記錄，指示分區 2 中的操作成功。
15. 此時，所有操作都成功，分散式事務完成。Saga 協調者向其調用者返回結果。

![Figure 12.29: 最終設計的編號序列](../images/v2/chapter12/Figure_12.29.png)

# 第四步 - 總結

在本章中，我們設計了一個能夠每秒處理超過 100 萬條支付命令的錢包服務。經過粗略估算，我們得出結論：需要數千個節點來支援這樣的負載。

在第一個設計中，提出了使用 Redis 等記憶體鍵值儲存的解決方案。這個設計的問題是資料不具備持久性。

在第二個設計中，記憶體快取被交易性資料庫取代。為了支援多個節點，提出了不同的分散式事務協定，如 2PC、TC/C 和 Saga。基於事務的解決方案的主要問題是無法輕鬆進行資料審計。

接下來，我們引入了事件溯源。我們首先使用外部資料庫和佇列實現了事件溯源，但效能不佳。通過將命令、事件和狀態儲存在本地節點中，我們提高了效能。

單節點意味著單點故障。為了提高系統的可靠性，我們使用 Raft 共識演算法將事件列表複製到多個節點上。

我們做的最後一個增強是採用事件溯源的 CQRS 特性。我們為外部使用者提供了一個非同步的讀取路徑。TC/C 或 Saga 協定用於協調多個節點組之間的命令執行。

恭喜你堅持到這裡！現在給自己一個鼓勵吧。幹得好！
