# 第 7 章 飯店預訂系統

在本章中，我們將設計一個飯店預訂系統，例如萬豪國際（Marriott International）。本章中使用的設計和技術也適用於其他流行的與預訂相關的面試題：

- 設計 Airbnb
- 設計航班預訂系統
- 設計電影票預訂系統

## 第 1 步 - 了解問題並確定設計範圍

飯店預訂系統非常複雜，其元件因業務用例而異。在深入設計之前，你應該向面試官詢問澄清問題以縮小範圍。

**候選人**：系統的規模有多大？
**面試官**：假設我們是為一個擁有 5,000 家飯店的連鎖飯店建立網站，共 100 萬間客房。

**候選人**：客戶是在預訂時支付，還是在到達飯店時支付？
**面試官**：為了簡單起見，他們在預訂時全額支付。

**候選人**：客戶僅通過飯店網站預訂客房嗎？我們需要支援其他預訂方式（如電話預訂）嗎？
**面試官**：假設人們可以通過飯店網站或 App 預訂客房。

**候選人**：客戶可以取消預訂嗎？
**面試官**：可以。

**候選人**：還有其他需要考慮的事情嗎？
**面試官**：是的，我們允許 10% 的超額預訂（overbooking）。如果你不知道的話，超額預訂意味著飯店出售的房間數超過了其實際擁有的房間數。飯店這樣做是考慮到部分客戶會取消預訂。

**候選人**：由於時間有限，我假設飯店搜尋不在範圍內。我們專注於以下功能：

- 展示飯店相關頁面。
- 展示客房相關詳情頁。
- 預訂客房。
- 用於添加/刪除/更新飯店或房間資訊的後台管理面板。
- 支援超額預訂功能。

**面試官**：聽起來不錯。

**面試官**：還有一件事，飯店價格是動態變化的。客房價格取決於該飯店在給定日期的預計入住率。對於本次面試，我們可以假設每天的價格可能不同。
**候選人**：我會記住這一点的。

接下來，你可能想談談最重要的非功能性需求。

### 非功能性需求 (Non-functional requirements)

- 支援高並發。在旺季或重大活動期間，一些熱門飯店可能會有很多客戶嘗試預訂同一間房。
- 適度的延遲。使用者進行預訂時，最好能有快速的回應時間，但系統處理預訂請求花費幾秒鐘也是可以接受的。

### 估算 (Back-of-the-envelope estimation)

- 總計 5,000 家飯店和 100 萬間客房。
- 假設 70% 的房間被佔用，平均入住時長為 3 天。
- 預估每日預訂量：(100 萬 × 0.7) / 3 = 233,333（向上取整為 ~240,000）。
- 每秒預訂數 = 240,000 / $10^5$ 秒（一天約 $10^5$ 秒）= ~3。如我們所見，平均每秒預訂事務數 (TPS) 並不高。

接下來，讓我們粗略計算一下系統中所有頁面的 QPS。典型的客戶流程有三個步驟：

1. 查看飯店/客房詳情頁。使用者流覽此頁面（查詢）。
2. 查看預訂頁。使用者在預訂前確認預訂詳情，如日期、客人數、支付資訊（查詢）。
3. 預訂房間。使用者點擊「預訂」按鈕預訂房間，房間被預訂（事務）。

讓我們假設大約 10% 的使用者會進入下一步，而 90% 的使用者在到達最後一步之前會流失。我們還可以假設沒有實現預取（prefetching）功能（即在使用者到達下一步之前預取內容）。圖 1 展示了不同步驟 QPS 的粗略估算。我們知道最終預訂的 TPS 是 3，因此我們可以沿漏斗逆向推導。訂單確認頁面的 QPS 為 30，詳情頁的 QPS 為 300。

發起新預訂是一個非常重要的功能。發起新預訂（POST /v1/reservations）的請求參數可能如下所示：

```json
{
  "startDate": "2021-04-28",
  "endDate": "2021-04-30",
  "hotelID": "245",
  "roomID": "U12354673389",
  "reservationID": "U12354673390"
}
```

請注意，`reservationID` 被用作**冪等鍵 (idempotency key)**，以防止重複預訂（double booking）。重複預訂是指在同一天為同一間客房進行了多次預訂。詳細資訊將在「深度探究」章節的「並發問題」部分進行解釋。

## 資料模型 (Data model)

在決定使用哪種資料庫之前，讓我們先仔細審視資料存取模式。對於飯店預訂系統，我們需要支援以下查詢：

查詢 1：查看飯店的詳細資訊。
查詢 2：在給定日期範圍內查找可用的房型。
查詢 3：記錄預訂資訊。
查詢 4：查找某筆預訂或預訂的歷史記錄。

從估算中我們知道，系統的規模並不大，但我們需要為重大活動期間的流量激增做好準備。考慮到這些需求，我們選擇關聯式資料庫，原因如下：

- 關聯式資料庫非常適合讀密集型（read-heavy）且寫頻率較低（write less frequently）的工作負載。這是因為存取飯店網站/App 的使用者數量比實際進行預訂的使用者數量高出幾個數量級。NoSQL 資料庫通常針對寫入進行了優化，而關聯式資料庫對於讀密集型工作負載表現得足夠好。
- 關聯式資料庫提供 ACID 保證。ACID 屬性對於預訂系統至關重要。如果沒有這些屬性，就不容易防止諸如負餘額、重複扣費、重複預訂等問題。ACID 屬性使應用程式碼變得簡單得多，並使整個系統更容易被理解和推導。關聯式資料庫通常提供這些保證。
- 關聯式資料庫可以輕鬆地對資料進行建模。業務資料的結構非常清晰，不同實體（飯店、客房、房型等）之間的關係也很穩定。關聯式資料庫可以輕鬆地對這種資料模型進行建模。

既然我們選擇了關聯式資料庫作為我們的資料儲存，那麼讓我們來探討一下表結構設計（schema design）。圖 2 展示了一個簡單的表結構設計，它是許多候選人對飯店預訂系統進行建模時最自然的方式。

## 高層設計 (High-level design)

我們為這個飯店預訂系統採用微服務架構。在過去的幾年中，微服務架構變得非常流行。使用微服務的公司包括亞馬遜、Netflix、Uber、Airbnb、Twitter 等。如果你想了解更多關於微服務架構優勢的資訊，可以查看一些優秀的資源 [1] [2]。

我們的設計採用微服務架構進行建模，高層設計圖如圖 4 所示。

![Figure7.4 High-level design png](../images/v2/chapter07/Figure7.4.png)

圖 4 高層設計

我們將從上到下簡要介紹系統的每個元件。

- **使用者 (User)**：使用者通過手機或電腦預訂飯店房間。
- **管理員（飯店員工）(Admin)**：授權的飯店員工執行行政操作，如為客戶退款、取消預訂、更新客房資訊等。
- **CDN (內容分發網路)**：為了獲得更好的載入時間，CDN 被用於快取所有靜態資源，包括 JavaScript 套件、圖片、視頻、HTML 等。
- **公共 API 閘道 (Public API Gateway)**：這是一個完全託管的服務，支援限流、認證等功能。API 閘道配置為根據端點將請求路由到特定的服務。例如，載入飯店主頁的請求被定向到飯店服務 (Hotel Service)，而預訂飯店客房的請求被路由到預訂服務 (Reservation Service)。
- **內部 API (Internal APIs)**：這些 API 僅供授權的飯店員工使用。它們通過內部軟體或網站存取。通常受 VPN（虛擬專用網路）的進一步保護。
- **飯店服務 (Hotel Service)**：提供有關飯店和客房的詳細資訊。飯店和客房資料通常是靜態的，因此可以輕鬆快取。
- **房費服務 (Rate Service)**：提供未來不同日期的房價。飯店行業的一個有趣事實是，房間的價格取決於該飯店在特定日期的預計入住率。
- **預訂服務 (Reservation Service)**：接收預訂請求並預訂飯店房間。該服務還會在房間被預訂或預訂被取消時管理客房庫存。
- **支付服務 (Payment Service)**：執行客戶的付款操作，並在支付交易成功時將預訂狀態更新為「已支付 (paid)」，如果交易失敗則更新為「被拒絕 (rejected)」。
- **飯店管理服務 (Hotel Management Service)**：僅供授權的飯店員工使用。飯店員工可以使用以下功能：查看即將到來的預訂記錄、為客戶預訂房間、取消預訂等。

為了清晰起見，圖 4 省略了微服務之間互動的許多箭頭。例如，如圖 5 所示，預訂服務與房費服務之間應該有一個箭頭。預訂服務向房費服務查詢房費，這用於計算預訂的總房費。另一個例子是，飯店管理服務與大多數其他服務之間應該有許多連接箭頭。當管理員通過飯店管理服務進行更改時，請求會被轉發到實際擁有該資料的服務來處理這些更改。

![Figure7.5 Connections between services png](../images/v2/chapter07/Figure7.5.png)

圖 5 服務之間的連接

### 改進的資料模型

如高層設計中所述，當我們預訂飯店房間時，我們實際上預訂的是一種房型，而不是一間特定的房間。我們需要對 API 和表結構進行哪些更改來適應這一點？

對於預訂 API，請求參數中的 `roomID` 被替換為 `roomTypeID`。發起預訂的 API 如下所示：

POST /v1/reservations
請求參數：

```json
{
  "startDate": "2021-04-28",
  "endDate":"2021-04-30",
  "hotelID":"245",
  "roomTypeID":"12354673389",
  "roomCount":"3",
  "reservationID":"U12354673390"
}
```

我們將簡要介紹一些最重要的表。

- **room**：包含有關房間的資訊。
- **room_type_rate**：儲存特定房型在未來日期的價格資料。
- **reservation**：記錄房客預訂資料。
- **room_type_inventory**：儲存飯店房間的庫存資料。這張表對預訂系統非常重要，所以讓我們仔細看看每一列。
  - **hotel_id**：飯店的 ID。
  - **room_type_id**：房型的 ID。
  - **date**：單個日期。
  - **total_inventory**：房間總數減去那些臨時從庫存中移除的房間。有些房間可能會因為維護而下架。
  - **total_reserved**：指定 hotel_id、room_type_id 和日期下已預訂的客房總數。

設計 `room_type_inventory` 表還有其他方法，但每個日期一行可以使管理日期範圍內的預訂和查詢變得容易。如圖 6 所示，(hotel_id, room_type_id, date) 是複合主鍵。表中的行是通過查詢未來 2 年內所有日期的庫存資料預先填充的。我們有一個預定的每日作業，當日期進一步推進時，該作業會預先填充庫存資料。

現在我們已經完成了表結構設計，讓我們對儲存容量做一些估算。正如在估算部分提到的，我們有 5,000 家飯店。假設每家飯店有 20 種房型。那麼就是 (5,000 家飯店 x 20 種房型 x 2 年 x 365 天) = 7,300 萬行。7,300 萬行資料量並不大，單個資料庫足以儲存這些資料。然而，單台伺服器意味著單點故障。為了實現高可用性，我們可以跨多個區域或可用區設置資料庫複製。

`room_type_inventory` 表用於檢查客戶是否可以預訂特定類型的房間。預訂的輸入和輸出可能如下所示：

- 輸入：startDate (2021-07-01), endDate (2021-07-03), roomTypeID, hotelId, numberOfRoomsToReserve
- 輸出：如果指定的房型有庫存且使用者可以預訂，則為 True。否則，返回 false。

從 SQL 的角度來看，它包含以下兩個步驟：

1. 選擇日期範圍內的行

```sql
SELECT date, total_inventory, total_reserved
FROM room_type_inventory
WHERE room_type_id = ${roomTypeID} AND hotel_id = ${hotelId}
AND date between ${startDate} and ${endDate}
```

2. 對於每一條記錄，應用程式檢查以下條件：

```java
if (total_reserved + ${numberOfRoomsToReserve}) <= total_inventory
```

如果所有記錄的條件都返回 true，則表示日期範圍內的每個日期都有足夠的房間。

其中一個需求是支援 10% 的超額預訂。有了新的表結構，這很容易實現：

```java
if (total_reserved + ${numberOfRoomsToReserve}) <= 110% * total_inventory
```

此時，面試官可能會問一個後續問題：「如果預訂資料量對於單個資料庫來說太大了，你會怎麼做？」有幾種策略：

- 僅儲存目前和未來的預訂資料。預訂歷史記錄不經常被存取。因此，它們可以被歸檔，有些甚至可以移至冷儲存。
- 資料庫分片。最頻繁的查詢包括進行預訂或按名稱查找預訂。在兩種查詢中，我們都需要先選擇飯店，這意味著 `hotel_id` 是一個很好的分片鍵（sharding key）。資料可以通過 `hash(hotel_id) % number_of_servers` 進行分片。

**場景 2：如果當只剩下一間客房時，多個使用者同時預訂同一種房型會發生什麼？** 讓我們考慮如圖 11 所示的場景。

![Figure7.11 Race condition](../images/v2/chapter07/Figure7.11.png)

圖 11 競態條件

讓我們假設資料庫隔離級別不是串列化（serializable）[5]。使用者 1 和使用者 2 嘗試同時預訂同一種房型，但只剩下一間客房。我們將使用者 1 的執行稱為「事務 1」，使用者 2 的執行稱為「事務 2」。此時，飯店共有 100 間客房，其中 99 間已被預訂。

事務 2 通過檢查 `(total_reserved + rooms_to_book) <= total_inventory` 來確認是否有足夠的剩餘客房。由於還剩 1 間房，它返回 true。

事務 1 同樣通過檢查 `(total_reserved + rooms_to_book) <= total_inventory` 來確認是否有足夠的客房。由於還剩 1 間房，它也返回 true。

事務 1 預訂了房間並更新了庫存：`reserved_room` 變為 100。

然後事務 2 也預訂了房間。ACID 中的**隔離性（isolation）**屬性意味著資料庫事務必須獨立於其他事務完成其任務。因此，在事務 1 完成（提交）之前，事務 1 所做的資料更改對事務 2 是不可見的。所以事務 2 看到的 `total_reserved` 仍然是 99，並通過更新庫存預訂了房間：`reserved_room` 變為 100。這導致系統允許兩個使用者都預訂了房間，儘管只剩下一間客房。

事務 1 成功提交了更改。

事務 2 成功提交了更改。

解決這個問題通常需要某種形式的鎖機制。我們探討以下技術：

- 悲觀鎖（Pessimistic locking）
- 樂觀鎖（Optimistic locking）
- 資料庫約束（Database constraints）

在著手修復之前，讓我們先看看用於預訂房間的 SQL 偽程式碼。該 SQL 由兩部分組成：

- 檢查客房庫存
- 預訂房間

```sql
# 步驟 1：檢查客房庫存
SELECT date, total_inventory, total_reserved
FROM room_type_inventory
WHERE room_type_id = ${roomTypeID} AND hotel_id = ${hotelId}
AND date between ${startDate} and ${endDate}

# 對於步驟 1 返回的每一條記錄
if ((total_reserved + ${numberOfRoomsToReserve}) > 110% * total_inventory) {
    Rollback
}

# 步驟 2：預訂房間
UPDATE room_type_inventory
SET total_reserved = total_reserved + ${numberOfRoomsToReserve}
WHERE room_type_id = ${roomTypeID}
AND date between ${startDate} and ${endDate}

Commit
```

### 方案 1：悲觀鎖 (Pessimistic locking)

悲觀鎖 [6]，也稱為悲觀並發控制，通過在使用者開始更新記錄時立即對其加鎖來防止同時更新。其他嘗試更新該記錄的使用者必須等待，直到第一個使用者釋放鎖（提交更改）。

對於 MySQL，「SELECT ... FOR UPDATE」語句通過鎖定查詢選中的行來工作。讓我們假設一個事務由「事務 1」啟動。其他事務必須等待事務 1 完成後才能開始另一個事務。詳細說明如圖 12 所示。

![Figure7.12 Pessimistic locking](../images/v2/chapter07/Figure7.12.png)

圖 12 悲觀鎖

在圖 12 中，事務 2 的「SELECT ... FOR UPDATE」語句會等待事務 1 完成，因為事務 1 鎖定了這些行。在事務 1 完成後，`total_reserved` 變為 100，這意味著使用者 2 已無房可訂。

**優點：**

- 防止應用程式更新正在被更改——或已經被更改——的資料。
- 易於實現，且通過序列化更新來避免衝突。當資料競爭激烈時，悲觀鎖非常有用。

**缺點：**

- 當鎖定多個資源時可能會發生死鎖。編寫無死鎖的應用程式碼可能具有挑戰性。
- 該方案不可擴展。如果一個事務被鎖定太久，其他事務將無法存取該資源。這對資料庫效能有重大影響，尤其是當事務耗時較長或涉及大量實體時。

由於這些限制，我們不推薦在預訂系統中使用悲觀鎖。

### 方案 2：樂觀鎖 (Optimistic locking)

樂觀鎖 [7]，也稱為樂觀並發控制，允許多個並發使用者嘗試更新同一個資源。

實現樂觀鎖有兩種常用方法：版本號和時間戳。版本號通常被認為是一個更好的選擇，因為伺服器時鐘可能會隨時間產生偏差。我們通過版本號來解釋樂觀鎖的工作原理。

圖 13 展示了一個成功案例和一個失敗案例。

![Figure7.13 Optimistic locking](../images/v2/chapter07/Figure7.13.png)

圖 13 樂觀鎖

1. 資料庫表中添加了一個名為「version」的新列。
2. 在使用者修改資料庫行之前，它會讀取該行的版本號。
3. 當使用者更新該行時，它會將版本號加 1 並寫入該版本號。
4. 資料庫會進行一次驗證檢查；下一個版本號應該比目前版本號大 1。如果驗證失敗，事務將中止，使用者從步驟 2 開始重試。

樂觀鎖通常比悲觀鎖快，因為我們不需要鎖定資料庫。然而，當並發量很高時，樂觀鎖的效能會大幅下降。

為了理解其中的原因，考慮許多用戶端嘗試同時預訂同一家特定飯店客房的情況。由於對有多少用戶端可以讀取可用房間數沒有限制，所有用戶端讀取到的可用房間數和目前版本號都是相同的。當不同的用戶端進行預訂並將結果寫回資料庫時，只有其中一個會成功，其餘用戶端都會收到版本檢查失敗的訊息。這些用戶端必須重試。在後續的重試輪次中，依然只有一個用戶端能成功，其餘的仍需重試。雖然最終結果是正確的，但反覆的重試會導致非常糟糕的使用者體驗。

**優點：**

- 它防止應用程式編輯陳舊資料 (stale data)。
- 我們不需要鎖定資料庫資源。從資料庫的角度來看，實際上沒有鎖定。這完全取決於應用程式如何處理版本號邏輯。
- 樂觀鎖通常用於資料競爭較低的情況。當衝突很少發生時，事務可以在不支付管理鎖的開銷的情況下完成。

**缺點：**

- 當資料競爭激烈時，效能較差。

由於預訂系統的 QPS 通常不高，因此樂觀鎖是飯店預訂系統的一個不錯選擇。

### 方案 3：資料庫約束 (Database constraints)

這種方法與樂觀鎖非常相似。讓我們探討它是如何工作的。在 `room_type_inventory` 表中，添加以下約束：

`CONSTRAINT check_room_count CHECK((total_inventory - total_reserved) >= 0)`

使用與圖 14 所示相同的範例，當使用者 2 嘗試預訂房間時，`total_reserved` 變為 101，這違反了 `total_inventory (100) - total_reserved (101) >= 0` 的約束。事務隨後被回滾。

![Figure7.14 Database constraint](../images/v2/chapter07/Figure7.14.png)

圖 14 資料庫約束

**優點：**

- 易於實現。
- 在資料競爭極小時效果良好。

**缺點：**

- 與樂觀鎖類似，當資料競爭激烈時，可能會導致大量的失敗。使用者可能會看到仍有房間可用，但當他們嘗試預訂時，卻得到「無房可用」的回應。這種體驗會讓使用者感到沮喪。
- 資料庫約束不像應用程式碼那樣容易進行版本控制。
- 並非所有資料庫都支援約束。當我們從一個資料庫方案遷移到另一個資料庫方案時，可能會導致問題。

由於這種方法易於實現，且飯店預訂的資料競爭通常不高（低 QPS），因此它是飯店預訂系統的另一個不錯選擇。

## 擴展性 (Scalability)

通常，飯店預訂系統的負載並不高。然而，面試官可能會提出一個後續問題：「如果飯店預訂系統不僅用於一家連鎖飯店，而是用於像 booking.com 或 expedia.com 這樣熱門的旅遊網站呢？」在這種情況下，QPS 可能會高出 1,000 倍。

當系統負載很高時，我們需要了解什麼可能成為瓶頸。我們所有的服務都是無狀態的，因此可以通過添加更多伺服器來輕鬆擴展。然而，資料庫是帶狀態的，不能僅通過添加更多資料庫來擴展。讓我們探討如何擴展資料庫。

### 資料庫分片 (Database sharding)

擴展資料庫的一種方法是應用資料庫分片。其核心思想是將資料拆分到多個資料庫中，以便每個資料庫僅包含部分資料。

當我們對資料庫進行分片時，我們需要考慮如何分布資料。正如我們在資料模型部分所看到的，大多數查詢需要按 `hotel_id` 進行過濾。因此，一個自然的結論是我們按 `hotel_id` 對資料進行分片。在圖 15 中，負載被分散到 16 個分片中。假設 QPS 為 30,000。經過資料庫分片後，每個分片處理 30,000 / 16 = 1875 QPS，這處於單個 MySQL 伺服器的負載能力範圍內。

### 快取 (Caching)

飯店庫存資料有一個有趣的特性：只有目前和未來的飯店庫存資料是有意義的，因為客戶只能預訂近期未來的房間。

因此，對於儲存選擇，理想情況下我們希望有一種**生存時間 (Time-to-live, TTL)** 機制來自動刪除過期資料。歷史資料可以在不同的資料庫中查詢。Redis 是一個不錯的選擇，因為 TTL 和**最近最少使用 (Least Recently Used, LRU)** 快取淘汰策略可以幫助我們優化記憶體利用率。

如果載入速度和資料庫可擴展性成為問題（例如，我們正在設計 booking.com 或 expedia.com 規模的系統），我們可以在資料庫之上添加一個快取層，並將「檢查客房庫存」和「預訂房間」邏輯移動到快取層，如圖 16 所示。在這種設計中，只有一小部分請求會到達庫存資料庫，因為大多數大多數請求都被庫存快取攔截了。值得一提的一點是，即使 Redis 中顯示有足夠的庫存，出於預防考慮，我們仍然需要在資料庫端重新檢查庫存。資料庫是庫存資料的**事實來源 (source of truth)**。

![圖 16 快取](../images/v2/chapter07/Figure7.16.png)

讓我們首先介紹此系統中的每個元件。

**預訂服務 (Reservation service)**：支援以下庫存管理 API：

- 查詢給定房型和日期範圍內的可用房間數。
- 通過執行 `total_reserved + 1` 來預訂房間。
- 當使用者取消預訂時更新庫存。

**庫存快取 (Inventory cache)**：所有庫存管理查詢操作都移動到庫存快取 (Redis) 中，我們需要將庫存資料預先填充到快取。快取是一個具有以下結構的鍵值對儲存：

- `key`: `hotelID_roomTypeID_{date}`
- `value`: 給定飯店 ID、房型 ID 和日期的可用房間數。

對於飯店預訂系統，讀操作（檢查客房庫存）的量比寫操作高出一個數量級。大多數讀操作由快取回答。

**庫存資料庫 (Inventory DB)**：儲存庫存資料，作為事實來源。

#### 快取帶來的新挑戰 (New challenges posed by the cache)

添加快取層顯著提高了系統的可擴展性和吞吐量，但也引入了新的挑戰：如何維護資料庫和快取之間的資料一致性。

當使用者預訂房間時，在正常流程 (happy path) 中會執行兩個操作：

1. 查詢客房庫存，查明是否有足夠的房間剩餘。該查詢在**庫存快取**上運行。
2. 更新庫存資料。首先更新**庫存資料庫**。然後將更改非同步傳播到快取。這種非同步快取更新可以由應用程式碼調用，在資料保存到資料庫後更新庫存快取。它也可以使用**變更資料捕獲 (Change Data Capture, CDC)** [8] 進行傳播。CDC 是一種從資料庫讀取資料更改並將更改應用到另一個資料系統的機制。一個常見的解決方案是 Debezium [9]。它使用源連接器從資料庫讀取更改，並將其應用到 Redis [10] 等快取解決方案。

因為庫存資料首先在資料庫上更新，所以快取可能無法反映最新的庫存資料。例如，當資料庫說沒有房間剩餘時，快取可能會報告仍有空房，反之亦然。

如果你仔細思考，你會發現庫存快取和資料庫之間的一致性實際上並不重要，只要資料庫進行最終的庫存驗證檢查即可。

讓我們看一個範例。假設快取狀態顯示仍有空房，但資料庫顯示沒有。在這種情況下，當使用者查詢客房庫存時，他們發現仍有房間可用，因此他們嘗試預訂。當請求到達庫存資料庫時，資料庫執行驗證並發現沒有房間剩餘。在這種情況下，用戶端收到一個錯誤，表明在他們之前已經有其他人預訂了最後一間房。當使用者重新整理網站時，他們可能會看到沒有房間剩餘，因為資料庫在他們點擊重新整理按鈕之前已經將庫存資料同步到了快取。

**優點：**

- 減輕資料庫負載。由於查詢請求由快取層回答，資料庫負載顯著降低。
- 高效能。讀取查詢非常快，因為結果是從記憶體中獲取的。

**缺點：**

- 維護資料庫和快取之間的資料一致性很難。我們需要仔細思考這種不一致性如何影響使用者體驗。

### 服務間的資料一致性 (Data consistency among services)

在傳統的**單體架構 (monolithic architecture)** [11] 中，使用共用關聯式資料庫來確保資料一致性。在我們的微服務設計中，我們選擇了一種混合方法，讓預訂服務同時處理預訂和庫存 API，以便庫存和預訂資料庫表儲存在同一個關聯式資料庫中。如「並發問題」部分所述，這種安排允許我們利用關聯式資料庫的 ACID 屬性來優雅地處理預訂流程中出現的許多並發問題。

然而，如果你的面試官是一個微服務純粹主義者，他們可能會挑戰這種混合方法。在他們看來，在微服務架構中，每個微服務都有自己的資料庫，如圖 17 右側所示。

![Figure7.17 Monolithic vs microservice](../images/v2/chapter07/Figure7.17.png)

圖 17 單體 vs 微服務

這種純粹的設計引入了許多資料一致性問題。由於這是我們第一次涉及微服務，讓我們解釋一下它是如何發生的以及為什麼會發生。為了更易於理解，本次討論僅使用兩個服務。在現實世界中，一家公司可能有數百個微服務。在單體架構中，如圖 18 所示，不同的操作可以包裝在一個事務中以確保 ACID 屬性。

![Figure7.18 Monolithic architecture](../images/v2/chapter07/Figure7.18.png)

圖 18 單體架構

然而，在微服務架構中，每個服務都有自己的資料庫。一個邏輯上的原子操作可能跨越多個服務。這意味著我們不能使用單個事務來確保資料一致性。如圖 19 所示，如果更新操作在預訂資料庫中失敗，我們需要回滾庫存資料庫中的預留客房計數。通常，正常路徑（happy path）只有一條，但可能導致資料不一致的失敗情況卻有很多。

![Figure7.19 Microservice architecture](../images/v2/chapter07/Figure7.19.png)

圖 19 微服務架構

為了解決資料不一致問題，以下是業界認可的技術的高層總結。如果你想閱讀細節，請參考參考資料。

- **二階段提交 (Two-phase commit, 2PC)** [12]。2PC 是一種用於保證跨多個節點進行原子事務提交的資料庫協定，即所有節點要麼全部成功，要麼全部失敗。由於 2PC 是一種阻塞協定，單個節點的故障會阻塞整個進程直到該節點恢復。它的效能並不理想。
- **Saga**。Saga 是一系列本地事務的序列。每個事務更新並發布一條訊息，以觸發下一個事務步驟。如果某個步驟失敗，Saga 會執行補償事務（compensating transactions），以撤銷之前事務所做的更改 [13]。2PC 作為一個單一提交來執行 ACID 事務，而 Saga 由多個步驟組成並依賴於**最終一致性（eventual consistency）**。

值得注意的是，解決微服務之間的資料不一致需要一些複雜的機制，這會大大增加整體設計的複雜性。作為架構師，你需要決定增加的複雜性是否值得。針對這個問題，我們認為不值得這樣做，因此採取了更務實的方法，即將預訂和庫存資料儲存在同一個關聯式資料庫下。

## 第 4 步 - 總結

在本章中，我們展示了一個飯店預訂系統的設計。我們從收集需求和進行估算開始，以了解規模。在高層設計中，我們展示了 API 設計、資料模型的初稿以及系統架構圖。在深度探究中，由於我們意識到預訂應該在房型級別（room type-level）進行，而不是針對具體的房間，因此我們探索了替代的資料庫表結構設計。我們深入討論了競態條件（race conditions）並提出了幾種潛在的解決方案：

- 悲觀鎖 (Pessimistic locking)
- 樂觀鎖 (Optimistic locking)
- 資料庫約束 (Database constraints)

接著，我們討論了擴展系統的不同方法，包括資料庫分片（database sharding）和使用 Redis 快取。最後，我們應對了微服務架構中的資料一致性問題，並簡要介紹了幾種解決方案。
