# 第15章：設計 Google Drive

近年來，Google Drive、Dropbox、Microsoft OneDrive、Apple iCloud 等雲端儲存服務非常流行。在本章中，你需要設計 Google Drive。

在開始設計之前，讓我們花點時間了解一下 Google Drive。 Google Drive 是一種檔案儲存和同步服務，可幫助您在雲端儲存文件、照片、影片和其他檔案。您可以從任何電腦、智慧型手機和平板電腦存取您的檔案。你可以輕鬆地與朋友、家人和同事共享這些檔案 \[1]。圖 15-1 和 15-2 分別顯示了 Google Drive 在瀏覽器和行動應用程式中的樣子。

### 第1步：了解問題並確定設計範圍

設計 Google Drive 是一項大工程，因此提出問題以縮小範圍很重要。

候選人：最重要的特徵是什麼？

面試官：上傳下載檔案，檔案同步，通知。

候選人：這是一個行動應用程式、一個網路應用程式，還是兩者兼而有之？

面試官：都是。

考生：支援的檔案格式有哪些？

面試官：任何檔案類型。

候選人：檔案需要加密嗎？

面試官：是的，儲存中的檔案必須加密。

候選人：檔案大小有限制嗎？

面試官：是的，檔案必須為 10 GB 或更小。

候選人：產品有多少使用者？

面試官：10M DAU

在本章中，我們重點關注以下功能：

* 新增檔案。新增檔案的最簡單方法是將檔案拖放到 Google Drive 中。
* 下載檔案。
* 跨多個裝置同步檔案。將檔案新增到一台裝置後，它會自動同步到其他裝置。
* 檢視檔案修訂內容。
* 與您的朋友、家人和同事共享檔案
* 當檔案被編輯、刪除或與您共享時發送通知。

本章未討論的功能包括：

* 可靠性。可靠性對於儲存系統來說極其重要，資料遺失是不可接受的。
* 同步速度快。如果檔案同步花費太多時間，使用者會變得不耐煩並放棄該產品。
* 頻寬使用。如果產品佔用大量不必要的網路頻寬，使用者會不高興，特別是當他們使用行動數據時。
* 可擴展性。該系統應該能夠處理大量的流量。
* 高可用性。當某些伺服器離線、速度變慢或出現意外網路錯誤時，使用者應該仍然可以使用該系統。

#### 粗略估算

* 假設該應用程式有 5000 萬註冊使用者和 1000 萬 DAU。
* 使用者獲得10 GB 的可用空間。
* 假設使用者每天上傳2 個檔案。平均檔案大小為 500 KB。
* 1:1 的讀寫比。
* 分配的總空間： $$5000 萬 \times 10 GB = 500 PB$$
* 上傳 API 的 QPS： $$1000 萬 \times 2 次上傳/24 小時/3600 秒 \approx 240$$
* 峰值 QPS : $$QPS \times 2 = 480$$

#### API

API 是什麼樣子的？我們主要需要 3 個 API：上傳檔案、下載檔案和取得檔案修訂。

1.  上傳檔案到 Google Drive

    支援兩種類型的上傳：

    * 簡單上傳。當檔案較小時使用此上傳類型。
    * 可恢復上傳。當檔案很大並且網路中斷的可能性很高時，請使用此上傳類型。

    以下是可恢復上傳 API 的範例： `https://api.example.com/files/upload?uploadType=resumable`

    參數：

    * `uploadType=resumable`
    * `data`: 待上傳的本地檔案

    可恢復上傳通過以下 3 個步驟 \[2] 實現：

    * 發送初始請求以檢索可恢復 URL。
    * 上傳資料並監控上傳狀態。
    * 如果上傳受到干擾，請繼續上傳。
2.  從Google Drive下載檔案

    範例 API：`https://api.example.com/files/download`

    參數：

    * `path`：下載檔案路徑

    範例參數：

    ```json
    {
      "path": "/recipes/soup/best_soup.txt"
    }
    ```
3.  取得檔案修訂

    範例 API：`https://api.example.com/files/list_revisions`

    參數：

    * `path`：要取得修訂歷史記錄的檔案的路徑
    * 要返回的最大修訂數。

    範例參數：

    ```json
    {
      "path": "/recipes/soup/best_soup.txt",
      "limit": 20
    }
    ```

    所有 API 都需要使用者身份驗證並使用 HTTPS。安全通訊端層 (SSL) 保護用戶端和後端伺服器之間的資料傳輸。

你通宵達旦地設定了資料庫分片，並密切監控。一切又順利地運行了。你已經阻止了火災的發生，但你仍然擔心在儲存伺服器中斷的情況下可能出現的資料損失。你四處打聽，你的後台大師朋友法蘭克告訴你，許多領先的公司如Netflix和Airbnb都使用Amazon S3進行儲存。「亞馬遜簡單儲存服務（Amazon S3）是一種物件儲存服務，提供業界領先的可擴展性、資料可用性、安全性和效能」 \[3]。你決定做一些研究，看看它是否合適。

大量閱讀後，您對 S3 儲存系統有了很好的了解，並決定將檔案儲存在 S3 中。 Amazon S3 支援同區域和跨區域複製。一個區域是指亞馬遜網路服務（AWS）擁有資料中心的地理區域。如圖15-6所示，資料可以在同區域（左側）和跨區域（右側）複製。冗餘檔案儲存在多個區域，以防止資料遺失並確保可用性。儲存桶就像檔案系統中的資料夾。

![](../images/chapter15/figure15-6.png)

把檔案放到S3後，你終於可以睡個好覺了，不用擔心資料遺失。為了阻止類似的問題在未來發生，你決定對可以改進的地方做進一步研究。以下是你發現的幾個方面：

* 負載平衡器：新增負載平衡器來分配網路流量。負載平衡器確保流量均勻分佈，如果 Web 伺服器出現故障，它將重新分配流量。
* Web 伺服器：新增負載平衡器後，可以根據流量負載輕鬆新增/刪除更多Web 伺服器。
* 中繼資料資料庫：將資料庫移出伺服器以避免單點故障。同時，設定資料複製和分片以滿足可用性和可擴展性要求。
* 檔案儲存：Amazon S3 用於檔案儲存。為確保可用性和持久性，檔案被複製到兩個不同的地理區域。

#### 同步衝突

對於像 Google Drive 這樣的大型儲存系統，同步衝突時有發生。當兩個使用者同時修改同一個檔案或資料夾時，就會發生衝突。我們如何解決衝突？這是我們的策略：先處理的版本獲勝，後處理的版本接收衝突。圖 15-8 顯示了同步衝突的範例。

![](../images/chapter15/figure15-8.png)

在圖 15-8 中，使用者 1 和使用者 2 試圖同時更新同一個檔案，但是使用者 1 的檔案首先被我們的系統處理。使用者 1 的更新操作通過，但使用者 2 發生同步衝突。我們如何解決使用者 2 的衝突？我們的系統提供同一檔案的兩個副本：使用者 2 的本地副本和來自伺服器的最新版本（圖 15-9）。使用者 2 可以選擇合併兩個檔案或用另一個版本覆蓋一個版本。

![](../images/chapter15/figure15-9.png)

當多個使用者同時編輯同一個文件時，保持文件同步是一項挑戰。有興趣的讀者可以參考參考資料\[4] \[5]。

#### 高層設計

圖 15-10 說明了建議的高級設計。讓我們檢查系統的每個元件。

![](../images/chapter15/figure15-10.png)

**User**：使用者通過瀏覽器或行動應用程式使用該應用程式。

**Block servers**：區塊伺服器將區塊上傳到雲端儲存。區塊儲存，簡稱區塊級儲存，是一種在基於雲端的環境中儲存資料檔案的技術。一個檔案可以分成幾個區塊，每個區塊都有一個唯一的雜湊值，儲存在我們的中繼資料資料庫中。每個區塊都被視為一個獨立的物件並儲存在我們的儲存系統 (S3) 中。為了重建檔案，區塊以特定順序連接。至於區塊大小，我們參考了Dropbox：它把一個區塊的最大大小設定為4MB \[6]。

**Cloud storage**：檔案被分割成更小的區塊並儲存在雲端儲存中。

**Cold storage**：冷儲存是一種為儲存非活動資料而設計的計算機系統，意味著檔案在很長一段時間內不會被存取。

**Load balancer**：負載平衡器在 API 伺服器之間平均分配請求。

**API servers**：這些伺服器負責除上傳流程以外的幾乎所有工作。API伺服器用於使用者認證、管理使用者資料、更新檔案中繼資料等。

**Metadata database**：它儲存使用者、檔案、區塊、版本等中繼資料。請注意，檔案儲存在雲端，中繼資料資料庫僅包含中繼資料。

**Metadata cache**：一些中繼資料被快取以便快速檢索。

**Notification service**：它是一個發布者/訂閱者系統，允許在某些事件發生時將資料從通知服務傳輸到用戶端。在我們的具體案例中，通知服務會在其他地方新增/編輯/刪除檔案時通知相關客戶，以便他們可以拉取最新的更改。

**Offline backup queue**：如果用戶端離線並且無法拉取最新的檔案更改，離線備份佇列會儲存資訊，以便在用戶端在線時同步更改。我們已經在高層討論了 Google Drive 的設計。有些元件很複雜，值得仔細檢查；我們將在深入討論中詳細討論這些。

### 第3步：深入設計

在本節中，我們將仔細研究以下內容：區塊伺服器、中繼資料資料庫、上傳流程、下載流程、通知服務、節省儲存空間和故障處理。

#### 區塊伺服器

對於定期更新的大檔案，在每次更新時發送整個檔案會消耗大量頻寬。提出了兩種優化來最小化傳輸的網路流量：

* 增量同步。修改檔案時，使用同步演算法 \[7] \[8] 僅同步修改的區塊而不是整個檔案。
* 壓縮。對區塊進行壓縮可以顯著減少資料大小。因此，根據檔案類型，使用壓縮演算法對區塊進行壓縮。例如，gzip和bzip2是用來壓縮文字檔案的。壓縮圖像和影片則需要不同的壓縮演算法。

在我們的系統中，區塊伺服器負責上傳檔案的繁重工作。區塊伺服器通過將檔案分成區塊、壓縮每個區塊並加密它們來處理從用戶端傳遞的檔案。不是將整個檔案上傳到儲存系統，而是只傳輸修改過的區塊。

圖 15-11 顯示了新增檔案時區塊伺服器的工作方式。

![](../images/chapter15/figure15-11.png)

* 一個檔案被分割成更小的區塊。
* 每個區塊都使用壓縮演算法進行壓縮。
* 為了確保安全，每個區塊在發送到雲端儲存之前都經過加密。
* 區塊被上傳到雲端儲存。

圖 15-12 說明了增量同步，這意味著只有修改過的區塊才會傳輸到雲端儲存。突出顯示的區塊「區塊 2」和「區塊 5」表示已更改的區塊。使用增量同步，只有這兩個區塊被上傳到雲端儲存。

![](../images/chapter15/figure15-12.png)

區塊伺服器允許我們通過提供增量同步和壓縮來節省網路流量。

**高一致性要求**

我們的系統預設要求強一致性。一個檔案同時被不同的用戶端顯示不同是不可接受的。系統需要為中繼資料快取和資料庫層提供強一致性。

記憶體快取預設採用最終一致性模型，這意味著不同的副本可能有不同的資料。要實現強一致性，我們必須確保以下幾點：

* 快取副本和主伺服器中的資料是一致的。
* 在資料庫寫入時使快取無效，以確保快取和資料庫保持相同的值。

在關聯式資料庫中實現強一致性很容易，因為它維護了 ACID（原子性、一致性、隔離性、持久性）屬性 \[9]。但是，NoSQL 資料庫預設不支援 ACID 屬性。 ACID 屬性必須以程式化方式合併到同步邏輯中。在我們的設計中，我們選擇關聯式資料庫，因為 ACID 是原生支援的。

#### 上傳流程

讓我們討論一下用戶端上傳檔案時會發生什麼。為了更好地理解流程，我們繪製了如圖 15-14 所示的時序圖。

![](../images/chapter15/figure15-14.png)

在圖 15-14 中，並行發送了兩個請求：新增檔案中繼資料和將檔案上傳到雲端儲存。兩個請求都來自用戶端 1。

* 新增檔案中繼資料
  1. 用戶端1發送了一個請求，要求新增新檔案的中繼資料。
  2. 將新的檔案中繼資料儲存在中繼資料資料庫中，並將檔案上傳狀態改為 `pending`。
  3. 通知通知服務正在新增一個新的檔案。
  4. 通知服務通知相關用戶端（用戶端2）有檔案正在上傳
* 上傳檔案到雲端儲存
  1. 用戶端 1 將檔案內容上傳到區塊伺服器。
  2. 區塊伺服器將檔案分塊成區塊，壓縮，加密區塊，並將它們上傳到雲端儲存。
  3. 檔案上傳完成後，雲端儲存觸發上傳完成回呼。請求被發送到 API 伺服器。
  4. 在Metadata DB 中，檔案狀態改為 `uploaded`。
  5. 通知通知服務檔案狀態更改為 `uploaded`。
  6. 通知服務通知相關用戶端（用戶端2）檔案已完全上傳。

編輯檔案時，流程類似，不再贅述。

#### 下載流程

當一個檔案在其他地方被新增或編輯時，會觸發下載流。用戶端如何知道一個檔案是由另一個用戶端新增或編輯的？用戶端有兩種方法可以知道：

* 如果用戶端 A 在線，而另一個用戶端更改了檔案，通知服務將通知用戶端 A 某處發生了更改，因此需要拉取最新資料。
* 如果用戶端A在檔案被另一個用戶端修改時處於離線狀態，資料將被儲存到快取中。當離線的用戶端再次上線時，它就會拉出最新的變化

一旦用戶端知道一個檔案被改變，它首先通過API伺服器請求中繼資料，然後下載區塊來構建檔案。圖15-15顯示了詳細的流程。注意，由於空間的限制，圖中只顯示了最重要的部分。

![](../images/chapter15/figure15-15.png)

1. 通知服務通知用戶端某個檔案在其他地方發生了更改。
2. 一旦用戶端 2 知道有新的更新可用，它就會發送一個取得中繼資料的請求。
3. API 伺服器呼叫中繼資料資料庫來取得更改的中繼資料。
4. 中繼資料返回給API伺服器。
5. 用戶端 2 取得中繼資料。
6. 一旦用戶端收到中繼資料，它就會向區塊伺服器發送請求以下載區塊。
7. 區塊伺服器首先從雲端儲存中下載區塊。
8. 雲端儲存返回區塊給區塊伺服器。
9. 用戶端2下載所有新區塊重建檔案

#### 通知服務

為了保持檔案的一致性，本地執行的任何檔案變更都需要通知其他用戶端以減少衝突。通知服務就是為這個目的而構建的。在高層設計中，通知服務允許在事件發生時將資料傳輸到用戶端。這裡有幾個選項：

* 長輪詢。 Dropbox 使用長輪詢 \[10]
* WebSocket。 WebSocket 在用戶端和伺服器之間提供持久連接。溝通是雙向的。

儘管這兩種選擇都很好，但出於以下兩個原因，我們選擇了長輪詢：

* 通知服務的通訊不是雙向的。伺服器向用戶端發送有關檔案更改的資訊，反之亦然。
* WebSocket 適用於即時雙向通訊，例如聊天應用程式。對於 Google 雲端硬碟，通知發送不頻繁，沒有突發資料。

使用長輪詢，每個用戶端都會建立到通知服務的長輪詢連接。如果偵測到對檔案的更改，用戶端將關閉長輪詢連接。關閉連接意味著用戶端必須連接到中繼資料伺服器以下載最新的更改。收到回應或達到連接逾時後，用戶端立即發送新請求以保持連接打開。

#### 節省儲存空間

為了支援檔案版本歷史並確保可靠性，同一檔案的多個版本被儲存在多個資料中心。如果頻繁地備份所有檔案的修訂版，儲存空間就會很快被填滿。提出了三種技術來減少儲存成本：

* 去除重複的資料區塊。在帳戶層面消除多餘的區塊是節省空間的一個簡單方法。如果兩個區塊有相同的雜湊值，那麼它們就是相同的。
* 採用智慧資料備份策略。可以應用兩種優化策略：
  * 設定限制：我們可以為要儲存的版本數量設定限制。如果達到限制，最舊的版本將被新版本替換。
  * 只保留有價值的版本。有些檔案可能會被頻繁地編輯。例如，為一個大量修改的檔案儲存每個編輯過的版本可能意味著該檔案在短時間內被儲存1000多次。為了避免不必要的複製，我們可以限制儲存版本的數量。我們對最近的版本給予更多的權重。實驗有助於找出儲存的最佳版本數
* 將不常用的資料移動到冷儲存。冷資料是數月或數年未活躍的資料。像 Amazon S3 冰川 \[11] 這樣的冷儲存比 S3 便宜得多。

#### 故障處理

在一個大規模的系統中可能會出現故障，我們必須採取設計策略來解決這些故障。你的面試官可能有興趣聽聽你是如何處理以下系統故障的：

* 負載平衡器故障：如果負載平衡器出現故障，輔助節點將變為活動狀態並接收流量。負載平衡器通常使用心跳相互監控，心跳是負載平衡器之間發送的週期性訊號。如果負載平衡器在一段時間內沒有發送心跳，則認為它失敗了。
* 區塊伺服器故障：如果區塊伺服器發生故障，其他伺服器將接管未完成或掛起的作業。
* 雲端儲存故障：S3 bucket在不同地域多次複製。如果檔案在一個區域不可用，可以從不同區域取得它們。
* API伺服器故障：是無狀態服務。如果 API 伺服器發生故障，流量將通過負載平衡器重新導向到其他 API 伺服器。
* 中繼資料快取故障：中繼資料快取伺服器被多次複製。如果一個節點宕機，您仍然可以存取其他節點以取得資料。我們將啟動一個新的快取伺服器來替換發生故障的伺服器。
* 中繼資料資料庫故障
  * Master down：如果master宕機，提升其中一個slave充當新的master，並拉起一個新的slave節點。
  * Slave down：如果一個slave down了，可以用另一個slave進行讀取，帶上另一台資料庫伺服器來代替失效的。
* 通知服務失敗：每個在線使用者與通知伺服器保持一個長輪詢連接。因此，每個通知伺服器都與許多使用者相連。根據 2012 年的 Dropbox 談話 \[6]，每台機器打開超過 100 萬個連接。如果伺服器出現故障，所有長輪詢連接都會丟失，因此用戶端必須重新連接到不同的伺服器。即使一台伺服器可以保持許多打開的連接，它也無法立即重新連接所有丟失的連接。重新連接所有丟失的用戶端是一個相對緩慢的過程。
* 離線備份佇列故障：佇列被多次複製。如果一個佇列發生故障，該佇列的消費者可能需要重新訂閱備用佇列。

### 第4步：總結

在本章中，我們提出了一個支援 Google Drive 的系統設計。強一致性、低網路頻寬和快速同步的結合使設計變得有趣。我們的設計包含兩個流程：管理檔案中繼資料和檔案同步。通知服務是系統的另一個重要組成部分。它使用長輪詢來使用戶端保持最新的檔案更改。

像任何系統設計面試問題一樣，沒有完美的解決方案。每個公司都有其獨特的限制，你必須設計一個系統來適應這些限制。了解你的設計和技術選擇的權衡是很重要的。如果還有幾分鐘的時間，你可以談談不同的設計選擇。

例如，我們可以從用戶端直接向雲端儲存上傳檔案，而不是通過區塊伺服器。這種方法的優點是，它使檔案的上傳速度更快，因為檔案只需要傳輸一次到雲端儲存中。在我們的設計中，一個檔案首先被傳輸到區塊伺服器，然後再傳輸到雲端儲存。然而，這種新方法也有一些缺點：

* 首先，同樣的分區塊、壓縮和加密邏輯必須在不同的平台（iOS、Android、Web）上實現。這是很容易出錯的，需要大量的工程努力。在我們的設計中，所有這些邏輯都在一個集中的地方實現：區塊伺服器。
* 其次，由於用戶端很容易被駭客攻擊或操縱，在用戶端實現加密邏輯並不理想。

該系統的另一個有趣的演變是將在線/離線邏輯轉移到一個單獨的服務中。讓我們稱之為存在服務。通過將在線服務從通知伺服器中移出，在線/離線功能可以很容易地被其他服務整合。

恭喜你走到了這一步！現在給自己一個鼓勵，幹得漂亮！
