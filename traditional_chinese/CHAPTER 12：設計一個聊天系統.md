# 第12章：設計一個聊天系統


在本章中，我們將探討聊天系統的設計，幾乎每個人都使用聊天應用程式。 圖 12-1 顯示了市場上一些最流行的應用程式。

![](images/chapter12/figure12-1.jpg)

聊天應用程式對不同的人執行不同的功能。敲定確切的要求是極其重要的。例如，當面試官想到一對一的聊天時，你不希望設計一個專注於群組聊天的系統。探索功能要求是很重要的。

### 第1步：了解問題並確定設計範圍

就要設計的聊天應用類型達成一致至關重要。市場上有Facebook Messenger、微信和WhatsApp等一對一聊天應用，Slack等專注於群聊的辦公聊天應用，Discord等專注於大型群聊和低語音聊天延遲的遊戲聊天應用。

第一組需要弄清楚的問題是應該明確面試官要求設計一個聊天系統時她的想法到底是什麼。至少要弄清楚你是應該專注於一對一的聊天還是群組聊天應用。

你可以問的一些問題如下：

候選人：我們要設計什麼樣的聊天應用？ 1對1還是群聊？ 面試官：應該支援1對1和群聊。

候選人：這是一個手機APP？還是一個web APP？或者兩者都是？

面試官：都是。

候選人：這個應用程式的規模是多少？是新創公司的應用還是大規模的？

面試官：它應該支援5000萬日活躍使用者（DAU）。

候選人：對於小組聊天，小組成員的限制是什麼？

面試官：最多100人

候選人：對於聊天應用程式來說，哪些功能是重要的？它能支援附件嗎？

面試官：1對1聊天，群聊，在線狀態。系統只支援文字訊息。

候選人：訊息大小有限制嗎？

面試官：是的，文字長度應少於100,000字元。

候選人：是否需要端對端加密？

面試官：暫時不需要，但如果時間允許，我們會討論這個問題。

候選人：我們應將聊天記錄保存多長時間？

面試官：永久。

在這一章中，我們著重於設計一個類似於Facebook messenger的聊天應用，重點是以下功能：

* 一對一的聊天，訊息傳遞延遲低
* 小組聊天（最多100人）。
* 在線狀態
* 支援多個裝置，同一帳戶可以同時登入多個裝置。
* 推送通知

就設計規模達成一致也很重要，我們將設計一個支援 5000 萬 DAU 的系統。

### 第2步：提出高層次的設計方案並獲得認同

為了開發一個高品質的設計，我們應該對客戶和伺服器的通訊方式有一個基本的了解。在一個聊天系統中，用戶端可以是行動應用程式或Web應用程式。用戶端之間並不直接交流。相反，每個用戶端都連接到一個聊天服務，它支援上面提到的所有功能。讓我們專注於基本操作。聊天服務必須支援以下功能：

* 接收來自其他用戶端的訊息。
* 為每條訊息找到合適的收件人，並將訊息轉達給收件人。
* 如果一個收件人不在線，就在伺服器上保留該收件人的訊息，直到她在線。

圖 12-2 顯示了用戶端（發送方和接收方）與聊天之間的關係服務。

![](images/chapter12/figure12-2.jpg)

當客戶打算開始聊天時，它使用一個或多個網路協議連接聊天服務。對於一個聊天服務，網路協議的選擇很重要。讓我們與面試官討論一下這個問題。

對於大多數用戶端/伺服器應用程式，請求由用戶端發起。 對於聊天應用程式的發送方也是如此。

在圖 12-2 中，當發送方通過聊天服務向接收方發送訊息時，它使用久經考驗的 HTTP 協議，這是最常見的 Web 協議。 在此場景中，用戶端打開與聊天服務的 HTTP 連接並發送訊息，通知服務將訊息發送給接收者。 `Keep-Alive` 對此很有效，因為 `Keep-Alive` 標頭允許用戶端與聊天服務保持持久連接。 它還減少了 TCP 握手的次數。 HTTP 在發送端是一個不錯的選擇，許多流行的聊天應用程式（例如 Facebook \[1]）最初使用 HTTP 來發送訊息。

然而，接收方的情況就比較複雜了。由於HTTP是由客戶發起的，因此從伺服器發送訊息並非易事。多年來，許多技術被用來模擬伺服器發起的連接：輪詢（Polling）、長輪詢（Long polling）和 WebSocket。這些都是在系統設計面試中廣泛使用的重要技術，所以讓我們逐一研究。

#### 輪詢

如圖12-3所示，輪詢是一種技術，用戶端定期詢問伺服器是否有訊息可用。根據輪詢的頻率，輪詢的成本可能很高。它可能會消耗寶貴的伺服器資源來回答一個大部分時間都沒有答案的問題。

![](images/chapter12/figure12-3.jpg)

#### 長輪詢

因為輪詢可能是低效的，接下來的是長輪詢（圖12-4）。

![](images/chapter12/figure12-4.jpg)

在長輪詢中，用戶端保持連接打開，直到實際有新訊息可用或達到超時閾值。一旦用戶端收到新訊息，它會立即向伺服器發送另一個請求，重新啟動程序。

長輪詢有一些缺點：

* 發送方和接收方可能不會連接到同一個聊天伺服器。基於HTTP的伺服器通常是無狀態的。如果使用輪詢技術進行負載平衡，接收訊息的伺服器可能沒有與接收訊息的用戶端建立長輪詢連接。
* 伺服器沒有很好的方法來判斷一個客戶是否斷開了連接。
* 它的**效率很低**。如果一個使用者不怎麼聊天，長時間的輪詢仍然會在超時後進行週期性的連接。

#### WebSocket

WebSocket是從伺服器向用戶端發送非同步更新的最常見解決方案。

圖12-5顯示了它的工作原理。

![](images/chapter12/figure12-5.jpg)

WebSocket連接是由用戶端發起的。它是雙向且持久的。它以HTTP連接的形式開始，並可通過一些定義明確的握手方式「升級」為WebSocket連接。通過這種持久的連接，伺服器可以向用戶端發送更新。即使有防火牆，WebSocket連接通常也能工作。這是因為它們使用80或443端口，這些端口也被HTTP/HTTPS連接所使用。

前面我們說過，在發送方使用HTTP是一個很好的協議，但由於WebSocket是雙向的，沒有充分的技術理由不把它也用於發送。

圖12-6顯示了WebSockets（ws）在發送方和接收方的使用情況。

![](images/chapter12/figure12-6.jpg)

通過使用WebSocket進行發送和接收，它簡化了設計，並使用戶端和伺服器上的實現更加直接。由於WebSocket連接是持久的，因此有效的連接管理在伺服器端至關重要。

#### 高層次設計

剛才我們提到，選擇WebSocket作為用戶端和伺服器之間的主要通訊協議，是因為它的雙向通訊，需要注意的是，其他一切都不一定是WebSocket。事實上，聊天應用程式的大多數功能（註冊、登入、使用者資料等）都可以使用HTTP上的傳統請求/回應方法。

讓我們深入了解一下，看看系統的高級元件。

如圖12-7所示，聊天系統被分成三大類：無狀態服務、有狀態服務和第三方整合。

![](images/chapter12/figure12-7.jpg)

*   無狀態的服務

    無狀態服務是傳統的面向公眾的請求/回應用於管理登入、註冊、使用者資料等。這些是許多網站和應用程式中的常見功能。

    無狀態服務位於負載平衡器後面，其工作是根據請求路徑將請求路由到正確的服務。這些服務可以是單體的，也可以是單獨的微服務。我們不需要自己建立許多這樣的無狀態服務，因為市場上有一些服務可以很容易地被整合。

    我們將深入討論的一個服務是服務發現。它的主要工作是給客戶提供一個客戶可以連接到的聊天伺服器的DNS主機名列表。
*   有狀態的服務

    唯一有狀態的服務是聊天服務。該服務是有狀態的，因為每個客戶都與一個聊天伺服器保持持久的網路連接。在這個服務中，只要伺服器仍然可用，客戶通常不會切換到另一個聊天伺服器。服務發現與聊天服務密切協調，以避免伺服器過載。我們將在深入研究中介紹。
*   第三方整合

    對於一個聊天應用程式，推送通知是最重要的第三方整合。它是一種在新訊息到來時通知使用者的方式，即使應用程式沒有運行。正確整合推送通知是至關重要的。更多資訊請參考第10章 設計一個通知系統。

#### 可擴展性

在小規模上，上面列出的所有服務都可以放在一台伺服器中。即使以我們設計的規模，理論上也有可能在一個現代雲伺服器中處理所有的使用者連接。伺服器可以處理的並發連接數很可能是限制因素。在我們的場景中，在 100w 並發使用者的情況下，假設每個使用者連接在伺服器上需要10K記憶體（這是一個非常粗略的數字，非常依賴於語言選擇），則只需要大約10GB的記憶體即可將所有連接保存在一個伺服器上。

如果我們提出一種將所有內容都放在一台伺服器中的設計，這可能會在面試官的腦海中升起一個大大的不好信號。 沒有技術專家會在單一伺服器中設計這樣的規模。由於多種因素，單伺服器設計是交易的障礙，單點失敗是其中最大的。

然而，從單一的伺服器設計開始是完全可以的。只要確保面試官知道這只是一個起點。把我們提到的一切放在一起，圖12-8顯示了調整後的高層設計。

![](images/chapter12/figure12-8.jpg)

在圖12-8中，用戶端與聊天伺服器保持一個持久的 WebSocket 連接，用於即時訊息傳遞。

* 聊天伺服器 (Chat servers) 促進了訊息的發送/接收。
* 在線伺服器 (Presence servers) 管理在線/離線狀態。
* API伺服器 (API servers) 處理一切，包括使用者登入、註冊、更改資料等。
* 通知伺服器 (Notification servers) 發送推送通知。
* 最後，鍵值儲存 (KV store) 用於儲存聊天歷史。當一個離線使用者上線時，她會看到她以前所有的聊天歷史。

#### 儲存

在這一點上，我們已經準備好了伺服器，服務已經開始運行，第三方整合已經完成。在技術棧的深處是資料層。資料層通常需要一些努力才能得到正確的結果。我們必須做出的一個重要決定是，決定使用正確的資料庫類型：關聯式資料庫還是NoSQL資料庫？為了做出一個明智的決定，我們將檢查資料類型和讀/寫模式。

在一個典型的聊天系統中存在兩類資料。

第一類是通用資料，如使用者資料、設定、使用者朋友列表。這些資料被儲存在強大而可靠的關聯式資料庫中。複製和分片是滿足可用性和擴展性要求的常見技術。

第二種是聊天系統特有的：聊天歷史資料。了解讀/寫模式很重要。

* 聊天系統的資料量是巨大的。之前的一項研究\[2]顯示，Facebook 和 Whatsapp 每天要處理60億條訊息。
* 只有最近的聊天記錄被頻繁存取。使用者通常不會查找舊的聊天記錄。
* 雖然在大多數情況下都會查看最近的聊天記錄，但使用者可能會使用需要隨機存取資料的功能，如搜尋、查看您的提及內容、跳轉到特定的訊息等。這些情況應該由資料存取層來支援。
* 一對一聊天應用程式的讀寫比約為 1:1。

選擇正確的儲存系統，支援我們所有的使用案例是至關重要的。我們推薦鍵值儲存，理由如下。

* 鍵值儲存允許容易的水平擴展。
* 鍵值儲存為存取資料提供了非常低的延遲。
* 關聯式資料庫不能很好地處理長尾\[3]的資料。當索引變大時，隨機存取是很昂貴的。
* 鍵值儲存被其他成熟可靠的聊天應用程式所採用。例如，Facebook 和 Discord 都使用鍵值儲存。Facebook 使用 HBase\[4]，而 Discord 使用 Cassandra\[5]。

#### 資料模型

剛才，我們談到了使用鍵值儲存作為我們的儲存層。最重要的資料是訊息資料。讓我們仔細看一下。

*   一對一聊天的訊息表

    圖12-9顯示了1對1聊天的訊息表。主鍵是 `message_id` ，它有助於決定訊息的順序。我們不能依靠 `created_at` 來決定訊息的順序，因為兩條訊息可以同時建立。

    ![](images/chapter12/figure12-9.jpg)
*   群聊訊息表

    圖12-10顯示了群聊的訊息表。複合主鍵是 `(channel_id，message_id)`。頻道和組在此表示相同的含義。`channel_id` 是分割區鍵，因為群聊天中的所有查詢都在一個通道中運行。

    ![](images/chapter12/figure12-10.jpg)
*   訊息ID

    如何生成 `message_id` 是一個值得探索的有趣話題。`message_id` 承擔著確保訊息順序的責任。為了確定訊息的順序，`message_id` 必須滿足以下兩個要求。

    * ID 必須唯一。
    * ID應該可以按時間排序，也就是說，新行的ID要比舊行高。

    我們如何才能實現這兩項保證呢？我想到的第一個想法是 MySQL 中的 `auto_increment` 關鍵字。然而，NoSQL資料庫通常不提供這樣的功能。

    第二種方法是使用像Snowflake\[6]那樣的全域64位序列號產生器。這將在「第七章：在分散式系統中設計一個唯一的ID生成器」中討論。

    最後一種方法是使用本地序列號產生器。本地意味著ID只在一個組內是唯一的。本地ID發揮作用的原因是，在一對一的通道或一個組的通道內維持訊息序列就足夠了。與全域ID的實現相比，這種方法更容易實現。

### 第3步：深入設計

在系統設計面試中，通常希望你能深入了解高層設計中的一些元件。對於聊天系統，服務發現、訊息流和在線/離線值得深入探討。

#### 服務發現

服務發現的主要作用是根據地理位置、伺服器容量等標準，為客戶提供最佳的聊天伺服器。Apache Zookeeper \[7] 是一個流行的服務發現開源解決方案。它註冊了所有可用的聊天伺服器，並根據預定義的標準為客戶挑選最佳聊天伺服器。

圖12-11顯示了服務發現（Zookeeper）是如何工作的。

![](images/chapter12/figure12-11.jpg)

1. 使用者A嘗試登入APP
2. 負載平衡器發送登入請求到API服務
3. 在後端認證使用者後，服務發現為使用者A找到最佳的聊天伺服器。在這個例子中，伺服器2被選中，伺服器資訊被返回給使用者A。
4. 使用者A通過 WebSocket 連接到聊天伺服器2。

#### 訊息流

了解一個聊天系統的端到端流程是很有趣的。在本節中，我們將探討1對1的聊天流程、跨多個裝置的訊息同步和群組聊天流程。

**1對1聊天**

圖12-12解釋了當使用者A向使用者B發送訊息時發生的情況。

![](images/chapter12/figure12-12.jpg)

1. 使用者A向聊天伺服器1發送了一條聊天訊息。
2. 聊天伺服器1從ID生成器獲得一個訊息ID。
3. 聊天伺服器1將訊息發送至訊息同步佇列。
4. 訊息被儲存在一個鍵值儲存中。
5. a. 如果使用者B在線，訊息被轉發到使用者B所連接的聊天伺服器2。
6. b. 如果使用者B處於離線狀態，則從推送通知（PN）伺服器發送推送通知。
7. 聊天伺服器2將訊息轉發給使用者B，使用者B和聊天伺服器2之間有一個持久的WebSocket連接。

**多個裝置間的訊息同步**

許多使用者有多個裝置。我們將解釋如何在多個裝置上同步訊息。圖12-13顯示了一個訊息同步的例子。

![](images/chapter12/figure12-13.jpg)

在圖12-13中，使用者A有兩台裝置：一台手機和一台筆記型電腦。當使用者A用手機登入聊天應用程式時，它與聊天伺服器1建立了一個WebSocket連接。同樣地，筆記型電腦和聊天伺服器1之間也有一個連接。

每個裝置都維護著一個叫做 `cur_max_message_id`的變數，它記錄著裝置上最新的訊息ID。滿足以下兩個條件的訊息被認為是新訊息。

* 收件人ID等於當前登入的使用者ID。
* 鍵值儲存中的訊息ID大於 `cur_max_message_id`

由於每個裝置上都有不同的 `cur_max_message_id`，訊息同步很容易，因為每個裝置都可以從鍵值儲存獲得新的訊息。

**群組聊天流程**

與一對一的聊天相比，群組聊天的邏輯更加複雜。圖12-14和12-15解釋了這個流程。

![](images/chapter12/figure12-14.jpg)

圖12-14解釋了使用者A在群聊中發送訊息時發生的情況。假設群裡有3個成員（使用者A、使用者B和使用者C）。首先，使用者A的訊息被複製到每個組員的訊息同步佇列中：一個給使用者B，另一個給使用者C。你可以把訊息同步佇列看成是一個收件人的收件箱。這種設計選擇很適合小群組聊天，因為。

* 它簡化了訊息同步流程，因為每個客戶只需要檢查自己的收件箱就可以獲得新的訊息。
* 當群組人數較少時，在每個收件人的收件箱中儲存一份副本並不太多昂貴。

微信使用類似的方法，它將一個群組限制在500個成員\[8]。然而，對於擁有大量使用者的群組來說，為每個成員儲存一份訊息副本是不可接受的。

在收件人方面，一個收件人可以接收來自多個使用者的訊息。每個收件人都有一個收件箱（訊息同步佇列），其中包含來自不同發送者的訊息。圖12-15說明了這種設計。

![](images/chapter12/figure12-15.jpg)

#### 在線狀態

在線狀態指示器是許多聊天應用程式的一個基本功能。通常情況下，你可以在使用者的個人照片或使用者名稱旁邊看到一個綠點。本節解釋幕後發生的事情。

在高層設計中，在線伺服器負責管理在線狀態，並通過WebSocket與用戶端進行通訊。有幾個流程會觸發在線狀態的變化。讓我們來看看它們中的每一個。

**使用者登入**

使用者登入的流程在「服務發現」一節中解釋。在用戶端和即時服務之間建立WebSocket連接後，使用者A的在線狀態和最後活動時間戳被儲存在KV儲存中。狀態指示器顯示使用者在登入後處於在線狀態。

![](images/chapter12/figure12-16.jpg)

**使用者登出**

當使用者登出登入時，會經歷如圖 12-17 所示的使用者登出流程。 KV store 中在線狀態變為離線狀態。 狀態指示器顯示使用者離線。

![](images/chapter12/figure12-17.jpg)

**使用者斷開連接**

我們都希望我們的網路連接是一致和可靠的。然而，情況並非總是如此；因此，我們必須在我們的設計中解決這個問題。當一個使用者從網際網路斷開連接時，用戶端和伺服器之間的持久連接就會丟失。處理使用者斷開連接的一個天真的方法是將使用者標記為離線，並在連接重新建立時將其狀態改為在線。然而，這種方法有一個重大缺陷。使用者在短時間內頻繁地斷開和重新連接到網路是很常見的。例如，當使用者通過隧道時，網路連接可能會打開和關閉。在每次斷開/重新連接時更新在線狀態會使存在指標變化得太頻繁，導致使用者體驗不佳。

我們引入一個心跳機制來解決這個問題。定期地，一個在線用戶端向狀態伺服器發送一個心跳事件。如果狀態伺服器在一定時間內收到心跳事件，比如说來自用戶端的X秒，那麼使用者被認為是在線的。否則，它就處於離線狀態。

在圖12-18中，用戶端每5秒向伺服器發送一個心跳事件。在發送了3個心跳事件後，用戶端被斷開連接，並且在x=30秒內沒有重新連接（這個數字是任意選擇的，以演示邏輯）。在線狀態被改變為離線。

![](images/chapter12/figure12-18.jpg)

**在線狀態輸出**

使用者 A 的好友如何知道狀態變化？ 圖 12-19 解釋了它是如何工作的。 狀態伺服器使用發布-訂閱模型，其中每個朋友對都維護一個頻道。 當使用者A的在線狀態發生變化時，將事件發布到三個頻道，頻道A-B，A-C，A-D。 這三個頻道分別由使用者 B、C 和 D 訂閱。 因此，朋友們很容易獲得在線狀態更新。 用戶端和伺服器之間的通訊是通過即時 WebSocket 進行的。

![](images/chapter12/figure12-19.jpg)

上述設計對小規模的使用者群是有效的。例如，微信使用類似的方法，因為它的使用者群上限為500人。對於較大的群組，通知所有成員的在線狀態是昂貴和耗時的。假設一個群組有100,000個成員。每一個狀態變化將產生100,000個事件。為了解決效能瓶頸，一個可能的解決方案是只在使用者進入群組或手動刷新好友列表時取得在線狀態。

### 第4步：總結

在本章中，我們介紹了一個聊天系統架構，它支援1對1的聊天和小群組聊天。WebSocket用於用戶端和伺服器之間的即時通訊。聊天系統包含以下元件：用於即時訊息傳遞的聊天伺服器、用於管理在線狀態的狀態伺服器、用於發送推送通知的推送通知伺服器、用於聊天歷史持久性的鍵值儲存以及用於其他功能的API伺服器。

如果你在面試結束時有多餘的時間，這裡有額外的談話要點：

* 擴展聊天應用程式以支援媒體檔案，如照片和影片。媒體檔案的大小明顯大於文字。壓縮、雲儲存和縮圖是值得討論的話題。
* 端到端加密。Whatsapp支援訊息的端到端加密。只有發件人和收件人可以閱讀訊息。有興趣的讀者請參考參考資料中的文章\[9]。
* 在用戶端快取訊息，可以有效地減少用戶端和伺服器之間的資料傳輸。
* 提高載入時間。Slack建立了一個地理分佈的網路來快取使用者的資料、頻道等，以獲得更好的載入時間\[10]。
* 故障處理。
  * 聊天伺服器錯誤。可能有數十萬，甚至更多的，堅持不懈的連接到一個聊天伺服器。如果一個聊天伺服器離線，服務發現（Zookeeper）會提供一個新的聊天伺服器，讓客戶建立新的連接。
  * 訊息重發機制。重試和排隊是重發訊息的常用技術。

恭喜你走到了這一步！現在給自己一個鼓勵，幹得漂亮！
