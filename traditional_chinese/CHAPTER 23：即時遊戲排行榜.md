# 第10章 即時遊戲排行榜

在本章中，我們將討論，如何為一款連網手機遊戲設計排行榜。

什麼是排行榜？排行榜在遊戲和其他領域很常見，用於顯示誰在特定的賽季或比賽對局中領先。使用者在完成任務或挑戰後會被分配分數，分數最高的使用者就會在排行榜上名列前茅。圖 10.1 範例，展示了一款手機遊戲的排行榜。排行榜顯示領先競爭對手的排名，同時也顯示使用者在排行榜上的位置。

> 譯者注：賽季在原文中是 particular tournament，通常翻譯為特定賽事或錦標賽，在遊戲場景中，翻譯為賽季更加貼近國內讀者的習慣。之後的 particular tournament 均譯為賽季。

## 第1步 - 理解問題並確定設計範圍

排行榜可以非常簡單，但有許多不同的事項會增加複雜性。我們應該明確要求。

**候選人**：排行榜的分數是如何計算的？

**面試官**：使用者贏得一場對局就能得到一分。我們可以採用一個簡單的積分系統，每個使用者都有一個與之相關的分數。使用者每贏得一場對局，我們就應該在他們的總分上加上一分。

**候選人**：排行榜是否包括所有選手？

**面試官**：是的。

**候選人**：排行榜有時間段嗎？

**面試官**：每個月都會有一次新的賽季拉開帷幕，並開始新的排行榜。

**候選人**：是否可以認為，我們只關心前 10 名使用者？

**面試官**：我們希望顯示排行榜上前10名使用者以及特定使用者的位置。如果時間允許，我們還將討論如何返回比特定使用者高 4 位和低 4 位的身邊的4位玩家。

**候選人**：一次賽季有多少使用者？

**面試官**：日活躍使用者 (DAU) 平均500萬，月活躍使用者 (MAU) 平均 2500 萬。

**候選人**：一次賽季平均要進行多少場對局？

**面試官**：每位玩家平均每天要打 10 次對局。

**候選人**：如果兩名選手得分相同，如何確定名次？

**面試官**：在這種情況下，他們的排名是一樣的。如果時間允許，我們可以討論一下如何打破平局。

**候選人**：排行榜必須是即時的嗎？

**面試官**：是的，我們希望呈現即時結果，或者盡可能接近即時結果。不能展示批量的歷史結果。

現在，我們已經收集了所有需求，讓我們列出功能需求。

### 功能需求

- 在排行榜上顯示前 10 名玩家。
- 顯示使用者的具體排名。
- 顯示比指定使用者排名高四位和低四位的玩家。

除了明確功能性需求外，瞭解非功能性需求也很重要。

### 非功能性需求

- 即時更新分數。
- 得分更新會即時反映在排行榜上。
- 常見的可擴展性、可用性和可靠性要求。

### 粗略估算

讓我們進行一些簡單的計算，以確定我們的解決方案需要應對的潛在規模和挑戰。

在 500 萬 DAU 的情況下，如果遊戲的玩家在 24 小時內分布均勻，那麼平均每秒將有 50 個使用者 $({5,000,000 DAU \over 10^6 seconds}\approx 50)$ 。但是，我們知道使用者數量很可能不是均勻分布的，在晚上可能會出現高峰，因為不同時區的許多人都有時間玩遊戲。為了考慮到這一點，我們可以假設峰值負載是平均值的 5 倍。因此，我們希望允許每秒 250 個使用者的峰值負載。

使用者獲得積分的 QPS：如果一個使用者平均每天玩 10 場遊戲對局，則使用者獲得積分的 QPS 為： $50\times10\approx 500$。峰值QP 是平均值的 5 倍： $500\times5=2500$。

獲取排行榜前 10 名的 QPS：假設使用者每天打開一次遊戲，而排行榜前 10 名僅在使用者首次打開遊戲時載入。其 QPS 約為 50。

### 資料模型

系統的關鍵元件之一是排行榜儲存。我們將討論三種可能的解決方案：關聯式資料庫、Redis 和 NoSQL（NoSQL解決方案將在本文的深入設計部分進行解釋）。

#### 關聯式資料庫解決方案

要獲取使用者排名，我們將通過分數對排行榜表排序：

|------|------|------|
| 1 | happy_tomato | 987 |
| 2 | mallow | 902 |
| 3 | smith | 870 |
| 4 | mary1934 | 850 |

表 10.4: 按分數排序的結果

當資料集較小時，這種解決方案還能奏效，但當資料行數達到數百萬行時，查詢速度就會變得非常慢。讓我們來看看原因何在。

要確定使用者的排名，我們需要將每個玩家，排序到排行榜上的正確位置，這樣我們才能確定正確的排名。請記住，也可能有重複的分數，因此排名不僅僅是使用者在列表中的位置。

當我們需要處理大量持續變化的資訊時，SQL 資料庫的效能並不理想。嘗試對數百萬行進行排序操作將需要 10 秒鐘的時間，這對於所需的即時方法來說是不可接受的。由於資料是不斷變化的，因此考慮使用快取也是不可行的。

關聯式資料庫在設計上，無法處理這種即時的高負載查詢。如果以批量操作的方式進行，則可以使用 RDS，但這不符合為使用者返回即時排名的需求。

我們可以做的一項優化是添加索引，並使用 LIMIT 語句限制要掃描的頁面數。如下：

```sql
SELECT (@rownum := @rownum + 1) AS rank, user_id, score
FROM leaderboard
ORDER BY score DESC
LIMIT 10
```

然而，這種方法並不能很好地擴展。首先，查找使用者排名的效能不高，因為它基本上需要通過表掃描來確定排名。其次，這種方法沒有提供直接的解決方案，來確定不在排行榜頂端的使用者的排名。

#### Redis 解決方案

我們希望找到一種解決方案，即使在數百萬使用者的情況下也能提供可預測的效能，並允許我們輕鬆存取常用的排行榜操作，而無需依賴複雜的資料庫查詢。

Redis 為我們的問題提供了一個潛在的解決方案。Redis 是一種支援鍵值對的記憶體資料儲存。由於它在記憶體中工作，因此可以實現快速讀寫。Redis 有一種稱為 **sorted sets** 的資料類型，非常適合解決排行榜系統設計問題。

**什麼是 sorted sets？**

sorted sets 是一種類似於集合的資料類型。它的每個成員都與一個分數相關聯。集合的成員必須是唯一的，但分數可以重複。分數用於按升序排列 sorted sets。

我們的排行榜用例完美地映射了 sorted sets。在底層，sorted sets 由兩種資料結構實現：hashtab 哈希表和 skip list 跳表 [2]。哈希表將使用者映射到分數，跳過列表將分數映射到使用者。在 sorted sets 中，使用者按分數排序。如圖 10.8 所示，理解 sorted sets 的一個好方法是把它想像成一個帶有分數和成員列的表格。該表按分數降序排序。

![Figure10.8.png](../images/v2/chapter10/Figure10.8.png)

圖 10.8: 二月份的排行榜通過 sorted sets 表示

在本章中，我們不會詳述 sorted sets 的全部實現細節，但會介紹一些高層設計思路。

skip list 跳表是一種可實現快速搜尋的列表結構。它由基本排序連結串列和多級索引組成。讓我們來看一個範例。在圖 10.9 中，列表是一個已排序的單向連結串列。插入、移除和查詢操作的時間複雜度為 $O(n)$。

怎樣才能讓這些操作變得更快？其中一個辦法就是像二分查找演算法那樣，快速找到中間節點。為了達到這個目的，我們增加了一個跳過其他節點的一級索引，然後又增加了一個跳過一級索引的其他節點的二級索引。我們會不斷引入更多級索引，每一級索引都會跳過上一級索引中的每一個節點。當節點間的距離為 $({n \over 2}-1)$（其中為 n 為節點總數）時，我們將停止這種添加。如圖 10.9 所示，當我們使用多級索引時，搜尋數字 45 的速度會快很多。

![Figure10.9.png](../images/v2/chapter10/Figure10.9.png)

圖 10.9: 跳表

當資料集較小時，使用跳表對速度的提升並不明顯。圖 10.10 顯示了一個有 5 層索引的表示例。在基本連結串列中，需要遍歷 62 個節點才能到達正確的節點。而在跳表中，只需遍歷 11 個節點 [3]。

![Figure10.10.png](../images/v2/chapter10/Figure10.10.png)

圖 10.10: 帶有 5 級索引的跳表

與關聯式資料庫相比，sorted sets 的效能更高，因為在插入或更新時，每個元素都會自動按照正確的順序定位，而且在 sorted sets 中進行添加或查找操作的複雜度是 $O\big(log(n)\big)$。

相比之下，要計算關聯式資料庫中特定使用者的排名，我們需要運行嵌套查詢：

```sql
SELECT *,(SELECT COUNT(*) FROM leaderboard lb2
WHERE lb2.score >= lb1.score) RANK
FROM leaderboard lb1
WHERE lb1.user_id = {:user_id};
```

**Redis sorted sets 實現**

既然我們已經知道 sorted sets 的速度很快，那加下來，我們來看看通過 Redis 建立排行榜的具體操作 [4] [5] [6] [7]：

- ZADD：如果使用者尚不存在，則將其插入資料集。否則，更新該使用者的分數。執行需要 $O\big(log(n)\big)$。

- ZINCRBY：按指定的增量，遞增使用者的分數。如果使用者不存在於集合中，則假定分數從 0 開始。執行需要 $O\big(log(n)\big)$ 的時間。

- ZRANGE/ZREVRANGE：獲取按分數排序的使用者範圍。我們可以指定順序（range 與 revrange）、條目數和起始位置。這需要 $O\big(log(n)+m\big)$ 來執行，其中 m 是要獲取的條目的數量（在我們的情況下通常較少），n 是 sorted sets 中條目的數量。

- ZRANK/ZREVRANK：以對數的時間複雜度按升序/降序獲取任何使用者的位置。

1. **使用者得分**

![Figure10.11.png](../images/v2/chapter10/Figure10.11.png)

圖 10.11: 使用者得分

我們每個月都會建立一個新的排行榜 sorted sets ，之前的 sorted sets 會被移至歷史資料儲存區。當使用者贏得一場比賽時，他們會得到 1 分；因此我們會調用 ZINCRBY 將該使用者在當月排行榜中的得分增加 1，或者將該使用者添加到排行榜中（如果他們還沒有在排行榜中的話）。ZINCRBY 的語法是：

```shell
ZINCRBY <key> <increment> <user>
```

下面的命令會在使用者 mary1934 贏得比賽後為其增加一個積分。

```shell
ZINCRBY leaderboard_feb_2021 1 'mary1934'
```

雖然這不是一個明確的要求，但我們可以利用 ZREVRANGE 和所需上下使用者的的人數，輕鬆獲取使用者的相對位置。例如，如果使用者 Mallow007 的排名是 361，而我們想獲取在其上下的 4 名使用者，我們可以運行以下命令。

```shell
ZREVRANGE leaderboard_feb_2821 357 365
```

**儲存要求**

我們至少需要儲存使用者 ID 和得分。最壞的情況是，所有 2500 萬月活躍使用者都至少贏過一場遊戲，而且他們都在當月的排行榜上有記錄。假設使用者 ID 是 24 個字元的字串，得分是 16 位整數（或 2 個位元組），則每個排行榜條目需要 26 個位元組的儲存空間。假設最壞的情況是每個 MAU 有一個排行榜條目，那麼我們將需要 26 個位元組 x 25 百萬位元組 = 650 百萬位元組或約 650 MB 用於 Redis 快取中的排行榜儲存。即使我們將記憶體使用量增加一倍，以考慮到跳表和 sorted sets 的開銷，一台現代 Redis 伺服器也足以儲存這些資料。

另一個需要考慮的相關因素是 CPU 和 I/O 使用率。我們通過回溯估算得出的峰值 QPS 為 2500 次更新。這完全在單個 Redis 伺服器的效能範圍之內。

Redis 快取的一個問題是持久性，因為 Redis 節點可能會出現故障。幸運的是，Redis 確實支援持久性，但從磁碟重啟大型 Redis 執行個體的速度很慢。通常情況下，Redis 會配置一個讀取副本，當主執行個體發生故障時，讀取副本會被升級，並附加一個新的讀取副本。

此外，我們還需要在 MySQL 等關聯式資料庫中建立兩個支援表（使用者表和點表）。使用者表將儲存使用者 ID 和使用者顯示名（在實際應用中，這將包含更多資料）。積分表將包含使用者 ID、得分和贏得比賽的時間戳。這可用於其他遊戲功能，如遊戲歷史，也可用於在基礎設施發生故障時重新建立 Redis 排行榜。

作為一個小的效能優化，為使用者詳細資訊建立一個額外的快取可能是有意義的，因為前 10 名玩家的檢索頻率最高。這並不會產生大量資料。

**服務管理**

在這種方法中，我們將每月建立一個排行榜 sorted sets ，以儲存該期間的排行榜資料。sorted sets 儲存使用者和分數資訊。使用者的其他詳細資訊（如姓名和個人資料圖片）則儲存在 MySQL 資料庫中。在獲取排行榜時，除了排行榜資料外，API 伺服器還會查詢資料庫，以獲取相應的使用者名稱和個人資料圖像，並顯示在排行榜上。如果長期這樣做效率太低，我們可以快取前 10 名玩家的使用者詳細資訊。設計如圖 10.15 所示。

#### Redis 擴展

在 DAU 為 500 萬的情況下，從儲存和 QPS 的角度來看，我們只需使用一個 Redis 快取即可。但是，假設我們有 5 億 DAU，這是我們原始規模的 100 倍。現在，在最壞的情況下，我們的排行榜規模會增加到 65 GB（650MB x 100），而 QPS 則會增加到每秒 250,00（2,500 x 100）次查詢。這就需要採用分片解決方案。

> 譯者注：這種計算方法，只是從儲存的角度，實際業務中，我們還需要考慮 Redis 的熱 KEY 和 大 KEY 問題喔。

**資料分片**

我們考慮用以下兩種方式之一進行分片：固定分片或哈希分片。

**固定分片**

了解固定分區的一種方法是查看排行榜上積分的總體範圍。假設一個月內贏得的分數從 1 到 1000 不等，我們按範圍對資料進行分割。例如，我們可以有 10 個分區，每個分區有 100 個分數範圍（例如，1 ~ 100，101 ~ 200，201 ~ 300，...），如圖 10.18 所示。

![Figure10.18.png](../images/v2/chapter10/Figure10.18.png)

圖 10.18: 固定分片

為此，我們要確保排行榜上的分數分布均勻。否則，我們就需要調整每個分區的分數範圍，以確保分數分布相對均勻。在這種方法中，我們自己在應用程式碼中對資料進行分片。

在插入或更新使用者分數時，我們需要知道使用者所在的分區。我們可以從 MySQL 資料庫中計算出使用者目前的得分。這種方法可行，但效能更好的方法是建立二級快取來儲存使用者 ID 到分數的映射。當使用者提高分數並在分片之間移動時，我們需要小心。在這種情況下，我們需要將使用者從目前分片移除，並將其轉移到新的分片。

要獲取排行榜中的前 10 名選手，我們需要從得分最高的分片（sorted sets）中獲取前 10 名選手。在圖 10.18 中，得分[901, 1000]的最後一個分片包含了前 10 名選手。

要獲取使用者的排名，我們需要計算使用者在目前分片內的排名（本地排名），以及所有分配內分數較高的玩家總數。請注意，可以通過運行 info keyspace 命令 $O(1)$ 的獲取分區內玩家的總數 [11]。

**哈希分片**

第二種方法是使用 Redis 叢集，如果分數非常集中或成塊，這種方法就比較理想。Redis 叢集提供了一種在多個 Redis 節點間自動分片的方法。它使用的不是一致性哈希，而是另一種形式的分片，即每個鍵都是哈希槽的一部分。哈希槽有 16384 個 [12]，我們可以通過 CRC16(key) %16384 [13] 來計算給定金鑰的哈希槽。這樣我們就可以用圖 10.19 的方法，我們有 3 個節點，其中：

- 第一個節點包含哈希槽 [0,5500]。
- 第二個節點包含哈希槽 [5501, 11000]。
- 第三個節點包含哈希槽 [11001, 16383]。

![Figure10.19.png](../images/v2/chapter10/Figure10.19.png)

圖 10.19: 哈希分片

更新只需改變使用者在相應分塊中的得分（由 CRC16(key) %16384 決定）。檢索排行榜上的前 10 名玩家則更為複雜。我們需要收集每個分區的前 10 名玩家，並讓應用程式對資料進行排序。具體範例如圖 10.20 所示。這些查詢可以並行處理，以減少延遲。

![Figure10.20.png](../images/v2/chapter10/Figure10.20.png)

圖 10.20: 分散收集

這種方法有一些局限性：

- 當我們需要在排行榜上返回頂部 k 個結果（其中 k 是一個非常大的數字）時，延遲會很高，因為每個分片都會返回大量條目並需要進行排序。
- 如果有很多分片，延遲就會很高，因為查詢必須等待最慢的分片。
- 這種方法的另一個問題是，它沒有為確定特定使用者的排名提供直接的解決方案。

因此，我們傾向於第一種方案：固定分片。

**確定 Redis 節點的大小**

在確定 Redis 節點的大小時，需要考慮多個方面 [14]。寫入量大的應用程式需要更多可用記憶體，因為我們需要能夠容納所有寫入，以便在發生故障時建立快照。為安全起見，應為寫入量大的應用程式分配兩倍的記憶體。

#### 替代方案：NoSQL

一種替代解決方案是考慮使用 NoSQL 資料庫。我們應該選擇什麼樣的 NoSQL 資料庫呢？理想情況下，我們希望選擇具備以下特性的 NoSQL 資料庫：

- 為寫入操作進行了優化。
- 能夠在分區內根據得分高效地排序項目。

NoSQL 資料庫，如 Amazon 的 DynamoDB [16]、Cassandra 或 MongoDB 都是不錯的選擇。在本章中，我們以 DynamoDB 為例。DynamoDB 是一種完全托管的 NoSQL 資料庫，提供可靠的效能和卓越的可擴展性。為了高效地存取除主鍵以外的其他屬性，我們可以在 DynamoDB 中使用全域二級索引 [17]。全域二級索引包含從父表選擇的屬性，但它們以不同的主鍵方式組織起來。我們來看一個範例。

此表結構可用，但無法很好地擴展。隨著添加更多行，我們必須掃描整個表以找到最高分項。

為了避免線性掃描，我們需要添加索引。我們的第一種嘗試是使用 year-month 作為分區鍵，並使用 score 作為排序鍵，如圖 10.23 所示。

![Figure10.23.png](../images/v2/chapter10/Figure10.23.png)

圖 10.23: 分區鍵和排序鍵

在高負載下，這種設計會遇到問題。DynamoDB 使用一致性哈希（consistent hashing）將資料分布到多個節點中。每個資料項基於其分區鍵被映射到對應的節點。

在上述表設計（圖 10.23）中，所有最近月份的資料都會集中在同一個分區，從而形成「熱點分區」（hot partition）。我們該如何解決這個問題？

一種方法是將資料拆分為多個分區，並在分區鍵後附加一個分區編號（例如 user_id % n，其中 n 為分區數）。這種模式被稱為寫分片（write sharding）。它會增加讀寫操作的複雜性，因此需要慎重考慮。

需要回答的關鍵問題是，我們應該使用多少個分區？這取決於寫入量或 DAU（每日活躍使用者數量）。重要的是要確保負載可以均勻分布在多個分區上，從而減少讀操作的複雜度。

由於同一月份的資料會分布在多個分區中，要讀取給定月份的資料，必須查詢所有分區的結果，這增加了讀操作的複雜度。

更新後的分區鍵設計像這樣：`game_name#{year-month}#p{partition_number}`。這代表了更新後的表結構。

![Figure10.24.png](../images/v2/chapter10/Figure10.24.png)

圖 10.24: 更新分區鍵

全域二級索引使用 `game_name#{year-month}#p{partition_number}` 作為分區鍵，使用 score 作為排序鍵。最終我們得到的是 n 個分區，每個分區內部都是排序好的（局部排序）。假設我們有 3 個分區，那麼為了獲取前 10 名排行榜，我們會使用之前提到的「scatter-gather」方法。我們會在每個分區中獲取前 10 個結果（這是「scatter」部分），然後讓應用程式對所有分區的結果進行排序（這是「gather」部分）。如圖 10.25 所示。

![Figure10.25.png](../images/v2/chapter10/Figure10.25.png)

圖 10.25: Scatter-gather

我們如何決定分區數量？這需要仔細的基準測試。更多的分區會減少每個分區的負載，但也增加了複雜性，因為我們需要跨更多分區進行 scatter 操作來建構最終的排行榜。通過基準測試，我們可以更清楚地看到這種權衡。

然而，類似於之前提到的 Redis 分區方案，這種方法並不能直接得到使用者的準確排名。但是，我們可以獲得使用者位置的百分位數，這在實際應用中可能已經足夠好了。在現實中，告訴玩家他們在前 10-20% 可能比顯示具體排名（如1,200,001）更好。因此，如果規模大到需要分片，我們可以假設所有分片的分數分布大致相同。如果這個假設成立，我們可以用一個定時任務來分析每個分片的分數分布，並快取結果。

結果會是這樣的：

第 10 百分位 = 分數 < 100
第 20 百分位 = 分數 < 500
...
第 90 百分位 = 分數 < 6500

然後我們可以快速返回使用者的相對排名（比如第 90 百分位）。

## 第4步 - 總結

在本章中，我們建立了一個支援百萬 DAU 的即時遊戲排行榜解決方案。我們探討了使用 MySQL 資料庫的直接方案，但因為它無法擴展到百萬使用者而放棄了這種方法。然後我們使用 Redis 有序集合設計了排行榜。我們還通過在不同 Redis 快取之間使用分片來將解決方案擴展到 5 億 DAU。我們還提出了一個替代的 NoSQL 解決方案。

如果在面試最後還有額外時間，你可以討論以下幾個話題：

**更快的檢索和打破相同排名**

Redis 哈希表提供了字串欄位和值之間的映射。我們可以利用哈希實現兩個用例：

1. 儲存使用者 ID 到使用者物件的映射，以便在排行榜上顯示。這比從資料庫獲取使用者物件更快。
2. 在兩個玩家分數相同的情況下，我們可以基於誰先獲得該分數來對使用者進行排名。當我們增加使用者分數時，我們還可以儲存使用者ID 到最近獲勝遊戲時間戳的映射。在平局情況下，較早的時間戳排名更高。

**系統故障恢復**

Redis 叢集可能會遇到大規模故障。基於上述設計，我們可以建立一個腳本，利用 MySQL 資料庫中記錄的每次使用者獲勝時的時間戳。我們可以遍歷每個使用者的所有記錄，並為每個使用者的每條記錄調用一次 ZINCRBY 命令。這樣，在發生大規模故障時，我們可以在離線狀態下重建排行榜。
