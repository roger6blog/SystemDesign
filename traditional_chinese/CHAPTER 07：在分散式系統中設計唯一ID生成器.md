# 第07章：在分散式系統中設計唯一 ID 生成器



在本章中，要求在分散式系統中設計一個唯一 ID 生成器。 你的第一個想法可能是在傳統資料庫中使用具有 auto\_increment 屬性的主鍵。 但是，auto\_increment 在分散式環境中不起作用，因為 單一資料庫伺服器不夠大，跨多個資料庫以最小延遲生成唯一 ID 具有挑戰性。 以下是唯一 ID 的一些範例：

![](../images/chapter7/figure7-1.jpg)

### 第1步：了解問題並確定設計範圍

了解清楚問題是解決任何系統設計面試問題的第一步。下面是一個候選人與面試官互動的例子：

候選人：唯一ID有什麼特點？

面試官：ID 必須是唯一的，並且是可排序的。

候選人：每增加一條記錄，ID 是否加 1？

面試官：ID是按時間遞增的，不一定只遞增1，當天晚上建立的ID比早上建立的ID大。

候選人：ID 是否只包含數值？

面試官：是的，這很正確。

候選人：ID長度要求是多少？

面試官：ID 應該適合 64 位。

候選人：系統的規模是多少？

面試官：系統應該可以每秒生成10000個ID。

以上是可以向面試官提出的一些範例問題。了解需求並釐清歧義很重要。

本次面試題，要求如下：

* ID 必須是唯一的。
* ID 只是數值。
* ID 適合 64 位。
* ID 按日期排序。
* 能夠每秒生成超過 10,000 個唯一 ID。

### 第2步：提出高層次的設計方案並獲得認同

可以使用多個選項在分散式系統中生成唯一ID。

我們考慮的選項有：

* 多主複製
* 通用唯一識別碼 (UUID)
* Ticket 伺服器
* 推特雪花演算法

讓我們看看它們是如何工作的，以及每個選項的優缺點。

#### 多主複製

如圖7-2所示，第一種方式是多主複製。

![](../images/chapter7/figure7-2.jpg)

這種方法使用資料庫的 auto\_increment 特性。我們不是將下一個 ID 增加 1，而是將其增加 k，其中 k 是正在使用的資料庫伺服器的數量。如圖 7-2 所示，下一個要生成的 ID 等於同一伺服器中的前一個 ID 加 2。這解決了一些可擴展性問題，因為 ID 可以隨著資料庫伺服器的數量而擴展。

然而，這種策略有一些主要的缺點：

* 難以通過多個資料中心進行擴展
* ID在多個伺服器上不隨時間而增長
* 在增加或刪除伺服器時，不能很好地擴展

#### UUID

UUID 是另一種取得唯一 ID 的簡單方法。 UUID 是一個 128 位元數字，用於識別計算機系統中的資訊。UUID獲得重複的概率非常低。引自維基百科，「在每秒產生10億個UUIDs，大約100年後，創造一個重複的概率會達到50%」 \[1] 。

以下是 UUID 的範例：09c93e62-50b4-468d-bf8a-c07e1040bfb2。 UUID 可以獨立生成，無需伺服器之間的協調。圖 7-3 展示了 UUID 的設計。

![](../images/chapter7/figure7-3.jpg)

在這個設計中，每個Web伺服器都包含一個ID生成器，並且Web伺服器負責獨立生成ID。

**優點：**

* 生成 UUID 很簡單。伺服器之間不需要協調，因此不會有任何同步問題。
* 該系統易於擴展，因為每個 Web 伺服器負責生成它們使用的 ID。 ID 生成器可以輕鬆地與 Web 伺服器一起擴展。

**缺點：**

* ID 是 128 位元長，但我們的要求是 64 位元。
* ID 不會隨時間上升
* ID 可以是非數字的。

#### Ticket 伺服器

票據伺服器是產生唯一ID的另一種有趣的方式。`Flicker`開發了票據伺服器來生成分散式主鍵\[2]。值得一提的是，該系統是如何工作的。

![](../images/chapter7/figure7-4.jpg)

這個想法是在一個單一的資料庫伺服器（Ticket Server）中使用一個集中化的自動遞增功能。要了解更多這方面的資訊，請參考flicker的工程部落格文章\[2] 。

**優點：**

* 數字 ID
* 易於實施，適用於中小型應用程式

**缺點：**

* 單點故障。單一票務伺服器意味著如果票務伺服器發生故障，所有依賴它的系統都將面臨問題。為了避免單點故障，我們可以設定多個票務伺服器。然而，這將引入新的挑戰，如資料同步。

#### 推特雪花演算法

上面提到的方法給了我們一些關於不同的ID生成系統如何工作的想法。然而，它們都不符合我們的具體要求；因此，我們需要另一種方法。Twitter 獨特的 ID 生成系統「snowflake」\[3] 很有啟發性，可以滿足我們的要求。

分而治之是我們的朋友。我們不是直接生成一個ID，而是將一個ID分成不同的部分。圖7-5顯示了一個64位元ID的佈局。

![](../images/chapter7/figure7-5.jpg)

下面對每個部分進行解釋：

* 符號位元：1 位元，它將始終為 0。這是為將來使用保留的。它可以潛在地用於區分有符號數和無符號數。
* 時間戳：41 位元。自紀元或自訂紀元以來的毫秒數。我們使用 Twitter 雪花預設紀元 1288834974657，相當於 2010 年 11 月 4 日 01:42:54 UTC。
* 資料中心 ID：5 位元，這給了我們 $$2 ^ 5 = 32$$ 個資料中心。
* 機器 ID：5 位元，每個資料中心有 $$2 ^ 5 = 32$$ 台機器
* 序列號：12 位元。對於在該機器/程序上生成的每個 ID，序列號都會遞增 1。該數字每毫秒重置為 0。

### 第3步：深入設計

在高層設計中，我們討論了在分散式系統中設計唯一ID生成器的各種方案。我們確定了基於Twitter雪花ID生成器的方法。讓我們深入了解一下這個設計。為了恢復我們的記憶，設計圖被重新列在下面。

![](../images/chapter7/figure7-6.jpg)

資料中心ID和機器ID是在啟動時選擇的，一般在系統運行後就固定下來。資料中心ID和機器ID的任何變化都需要仔細審查，因為這些數值的意外變化會導致ID衝突。時間戳和序列號是在ID生成器運行時生成的。

#### 時間戳

最重要的41位元組成了時間戳部分。由於時間戳隨時間增長，ID可按時間排序。

圖 7-7 顯示了二進位表示如何轉換為 UTC 的範例。您還可以使用類似的方法將 UTC 轉換回二進位表示。

![](../images/chapter7/figure7-7.jpg)

可以用 41 位元表示的最大時間戳是： $$2 ^ {41} - 1 = 2199023255551毫秒(ms)$$，這樣我們就可以得到。 $$\approx 69年=2199023255551毫秒/1000秒/365天/24小時/3600秒$$。

這意味著 ID 生成器將工作 69 年，並且自訂紀元時間接近今天的日期會延遲溢位時間。 69年後，我們將需要一個新的紀元時間或採用其他技術來遷移ID。

#### 序列號

序列號是 12 位元，這給了我們 $$\mathbf{2 ^ {12} = 4096}$$ 種組合。除非在同一台伺服器上在一毫秒內生成多個 ID，否則該欄位為 0。理論上，一台機器每毫秒最多可以支援4096個新ID。

### 第4步：總結

在本章中，我們討論了設計唯一 ID 生成器的不同方法：多主複製、UUID、票務伺服器和類似 Twitter 雪花的唯一 ID 生成器。我們選擇了雪花，因為它支援我們所有的用例，並且在分散式環境中是可擴展的。

如果面試結束時有額外時間，這裡有一些額外的談話要點：

* 時鐘同步。在我們的設計中，我們假設 ID 生成伺服器具有相同的時鐘。當伺服器在多個核心上運行時，此假設可能不成立。同樣的挑戰存在於多機場景中。時鐘同步的解決方案超出了本書的範圍；但是，了解問題的存在很重要。網路時間協議是這個問題最流行的解決方案。有興趣的讀者可以參考參考資料\[4]。
* 區段長度調整。例如，較少的序列號但較多的時間戳位元對低並發性和長期應用是有效的。
* 高可用性。由於 ID 生成器是關鍵任務系統，因此它必須具有高可用性

恭喜你走到了這一步！現在給自己一個鼓勵，幹得漂亮！
